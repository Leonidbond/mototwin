# MotoTwin — UI-спек окна «Добавить свою деталь»
## Functional UI Specification v1.0

---

# 1. Главный принцип

Не использовать wizard на 4 отдельных экрана.

Окно должно работать как одна последовательная форма с динамическим раскрытием блоков.

Пользователь проходит сценарий сверху вниз:

1. выбранный мотоцикл;
2. выбор узла;
3. данные детали;
4. статус детали;
5. fitment-блок, если деталь установлена;
6. фото и комментарий;
7. отправка.

Без переключения между отдельными шагами и экранами.

---

# 2. Формат окна

## Desktop

Формат:

- широкая модалка;
- либо отдельная страница в стиле Mototwin.

Рекомендуемые размеры:

```text
max-width: 820–900 px
height: 85–90vh
scroll внутри модалки
```

Структура:

```text
Header
↓
Vehicle context
↓
Node selector
↓
Part data
↓
Cart / install status
↓
Fitment block
↓
Evidence / comment
↓
Footer actions
```

---

## Mobile

Формат:

```text
full-screen modal
```

Параметры:

```text
width: 100%
height: 100%
vertical scroll
sticky bottom CTA
```

# 3. Header

## Заголовок

```text
Добавить свою деталь
```

## Подзаголовок

```text
Добавьте деталь в корзину замен. Если вы уже установили ее, MotoTwin поможет поделиться совместимостью с владельцами такой же модели.
```

# 4. Блок — Контекст мотоцикла

## Название блока

```text
Мотоцикл
```

## Содержимое

Компактная карточка:

```text
BMW R 1250 GS Adventure
2022 • K51 • EU • Стоковая конфигурация
```

Справа:

```text
Изменить
```

---

## Поведение

Если пользователь пришел со страницы конкретного мотоцикла:

- блок уже заполнен.

Если мотоцикл не выбран:

Показывать CTA:

```text
Выбрать мотоцикл
```

---

# 5. Блок — Узел

Используется существующая модалка выбора узла.

Новый UI выбора узла не создается.

---

## Название блока

```text
Узел
```

---

## Состояние до выбора

Карточка:

```text
Выберите узел
Например: тормозные колодки, масляный фильтр, цепь, шина
```

Кнопка:

```text
Выбрать узел
```

---

## Состояние после выбора

Карточка:

```text
Тормоза → Передние тормозные колодки
Совместимость будет проверяться для этого узла и выбранной модели.
```

Действие:

```text
Изменить
```

---

## Важно

НЕ показывать пользователю:

```text
ID узла (leaf)
```

Это техническое поле.

---

# 6. Блок — Деталь

## Название блока

```text
Деталь
```

Блок отображается всегда.

До выбора узла можно показывать disabled-state или hint.

---

## Поле — Бренд

Label:

```text
Бренд детали
```

Placeholder:

```text
Например: EBC, Brembo, HifloFiltro, DID
```

Поле обязательное.

---

## Поле — Артикул / SKU

Label:

```text
Артикул / SKU
```

Placeholder:

```text
Например: FA209HH
```

Поле обязательное.

При вводе SKU желательно запускать поиск существующих деталей.

---

## Поле — Название

Label:

```text
Название
```

Placeholder:

```text
Например: Передние тормозные колодки EBC FA209HH
```

Поле обязательное.

---

## Поле — Категория

Категория должна автоматически предлагаться от выбранного узла.

Пример:

```text
Категория
[Тормозные колодки передние]
```

Разрешается изменить через dropdown.

Не использовать свободный текстовый ввод.

---

# 7. Поиск существующих деталей

После ввода бренда и SKU показывать блок поиска совпадений.

---

## Если совпадений нет

```text
Похожих деталей не найдено. Будет создана новая карточка детали.
```

---

## Если совпадения найдены

```text
Возможно, эта деталь уже есть в базе
```

Карточки:

```text
EBC FA209HH
Передние тормозные колодки
Использовать эту деталь
```

---

## Поведение

Если пользователь выбирает существующую деталь:

- поля заполняются автоматически;
- новая PartMaster не создается;
- пользователь продолжает flow.

---

# 8. Блок — Статус детали

Этот блок разделяет:

- корзину замен;
- fitment-report.

---

## Название блока

```text
Статус детали
```

---

## Варианты

Использовать segmented control или selectable cards.

Варианты:

```text
Планирую купить
Купил, но не установил
Установил
Не подошла
```

---

## Логика

### Планирую купить

Создается:

```text
ReplacementCartItem.status = PLANNED
```

Fitment-report не создается.

---

### Купил, но не установил

Создается:

```text
ReplacementCartItem.status = PURCHASED
```

Fitment-report не создается.

---

### Установил

Создается:

```text
ReplacementCartItem.status = INSTALLED
```

После выбора раскрывается блок:

```text
Совместимость
```

---

### Не подошла

Создается:

```text
ReplacementCartItem.status = REJECTED
```

Также раскрывается fitment-блок.

---

# 9. Блок — Совместимость

Блок отображается только если пользователь выбрал:

```text
Установил
```

или:

```text
Не подошла
```

---

## Название блока

```text
Совместимость с вашей моделью
```

---

## Подсказка

```text
Отчет будет привязан к вашей модели мотоцикла и выбранному узлу.
```

---

## Варианты совместимости

Для установленной детали:

```text
Подошла без доработок
Подошла с доработкой
Подошла частично / не уверен
OEM-замена
```

Для неподошедшей детали:

```text
Не подошла
```

---

## Если выбрано «Подошла с доработкой»

Показывать дополнительное поле:

```text
Что пришлось доработать?
```

Placeholder:

```text
Например: потребовалась замена проставки или подгонка крепления.
```

---

# 10. Блок — Фото и подтверждение

Блок должен быть легким и не перегружать форму.

---

## Название блока

```text
Фото и подтверждение
```

---

## Подсказка

```text
Фото не обязательно, но повышает доверие к отчету.
```

---

## Кнопки загрузки

```text
Фото детали / упаковки
Фото установленной детали
Чек или заказ
```

---

## Safety-critical warning

Для тормозов, подвески, двигателя:

```text
Для ответственных узлов фото особенно важно. Такие отчеты могут проверяться вручную.
```

---

# 11. Блок — Комментарий

## Название

```text
Комментарий
```

---

## Placeholder

```text
Расскажите коротко: как встала деталь, были ли нюансы и сколько уже проехали после установки.
```


# 18. Acceptance Criteria

Форма считается готовой, если:

1. пользователь не видит технический nodeId;
2. узел выбирается через существующую модалку;
3. совместимость привязана к modelVariantId + nodeId;
4. можно добавить деталь в корзину без fitment-report;
5. fitment-report создается только при статусе «Установил» или «Не подошла»;
6. новая деталь проходит duplicate-check;
7. CTA меняется в зависимости от статуса;
8. safety-critical узлы показывают warning;
9. форма работает в desktop и mobile;
10. интерфейс не выглядит как админский CRUD-screen.
