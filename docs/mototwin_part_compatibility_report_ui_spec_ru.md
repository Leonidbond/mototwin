# MotoTwin — UI-спецификация страницы «Итоговый отчет по совместимости детали»
## Part Compatibility Report Page — v1.1

---

# 1. Назначение страницы

Страница «Итоговый отчет по совместимости детали» показывает агрегированную информацию о том, насколько конкретная деталь подходит конкретной модели мотоцикла и конкретному узлу.

Это не карточка товара интернет-магазина и не форумная ветка.

Это продуктовая страница MotoTwin, которая объединяет:

- structured fitment rules;
- community fitment reports;
- количество пользователей по каждому статусу совместимости;
- confidence status;
- evidence;
- фото установки;
- сервисные события;
- конфигурации мотоциклов;
- комментарии владельцев;
- рекомендации по дальнейшему действию.

Главный вопрос, на который должна отвечать страница:

```text
Подойдет ли эта деталь именно на мой мотоцикл, для этого узла, и насколько этим данным можно доверять?
```

---

# 2. Базовый принцип совместимости

Совместимость всегда отображается для связки:

```text
Part + ModelVariant + Node
```

Расширенный контекст:

```text
Part + ModelVariant + VehicleConfig + Node
```

Пример:

```text
EBC FA209HH
+
BMW R 1250 GS Adventure 2022 / K51 / EU
+
Тормоза → Передние тормозные колодки
```

Важно:

- одна и та же деталь может подходить одной модели и не подходить другой;
- одна и та же деталь может подходить одной конфигурации и не подходить другой;
- community reports не должны подменять deterministic fitment rules;
- structured fitment и community-fitment должны отображаться как разные источники данных.

---

# 3. Основные сценарии использования

## 3.1. Пользователь проверяет деталь перед покупкой

Пользователь открывает деталь из страницы «Подбор детали» и хочет понять:

- подходит ли она;
- сколько владельцев подтвердили совместимость;
- были ли доработки;
- были ли негативные отчеты;
- зависит ли совместимость от конфигурации;
- можно ли добавить деталь в корзину замен.

## 3.2. Пользователь уже установил деталь

Пользователь открывает отчет и хочет:

- добавить свой опыт;
- подтвердить чужой отчет;
- загрузить фото;
- уточнить статус совместимости.

## 3.3. Пользователь видит противоречивые данные

Страница должна объяснить:

- почему статус mixed;
- какие конфигурации дали разные результаты;
- какие доработки встречались;
- какие отчеты наиболее надежные.

---

# 4. Источники данных

Страница использует 2 слоя данных.

## 4.1. Structured Fitment Layer

Источник:

- `FitmentRule`;
- `PartMaster`;
- `ModelVariant`;
- `VehicleConfig`;
- `Node`.

Используется для:

- verified compatibility;
- OEM compatibility;
- базовых расходников;
- safety-critical категорий;
- core recommendations.

## 4.2. Community Fitment Layer

Источник:

- `FitmentReport`;
- `FitmentVote`;
- `FitmentEvidence`;
- `ServiceEvent`;
- `ReplacementCartItem`.

Используется для:

- aftermarket деталей;
- аналогов;
- нестандартных решений;
- fitment с доработками;
- long-tail моделей;
- подтверждения реального опыта владельцев.

---

# 5. Статусы совместимости

Страница должна поддерживать все итоговые статусы.

## 5.1. VERIFIED_BY_MOTOTWIN

UI-текст:

```text
Проверено MotoTwin
```

Смысл:

- совместимость подтверждена structured fitment rule;
- либо проверена модератором / администратором;
- имеет максимальный приоритет над community reports.

Визуальный стиль:

- зеленый или зеленый + синий акцент;
- бейдж высокого доверия.

---

## 5.2. COMMUNITY_CONFIRMED

UI-текст:

```text
Подтверждено владельцами
```

Смысл:

- есть достаточное количество положительных community reports;
- нет существенных конфликтов;
- отчеты привязаны к той же модели и узлу.

Визуальный стиль:

- синий акцент;
- подпись «на основе опыта владельцев».

---

## 5.3. FITS_WITH_MODIFICATION

UI-текст:

```text
Подходит с доработкой
```

Смысл:

- деталь может быть использована;
- значимая часть пользователей указывает на необходимость доработки;
- страница должна показывать, какие именно доработки требовались.

Визуальный стиль:

- желтый / оранжевый акцент;
- предупреждающий, но не негативный тон.

---

## 5.4. MIXED_REPORTS

UI-текст:

```text
Есть противоречивые отчеты
```

Смысл:

- часть пользователей подтверждает совместимость;
- часть пользователей сообщает о проблемах;
- совместимость может зависеть от конфигурации, года, рынка, ABS, колес, поколения или установленного тюнинга.

Визуальный стиль:

- желтый акцент;
- prominent warning.

---

## 5.5. LOW_CONFIDENCE

UI-текст:

```text
Недостаточно данных
```

Смысл:

- отчетов мало;
- нет фото;
- нет подтверждений;
- нет связи с сервисными событиями;
- недостаточно данных для уверенного вывода.

Визуальный стиль:

- нейтральный / серый / синий низкой насыщенности.

---

## 5.6. REJECTED_LIKELY_INCOMPATIBLE

UI-текст:

```text
Скорее всего не подходит
```

Смысл:

- много негативных отчетов;
- либо deterministic layer помечает деталь как несовместимую;
- либо есть критичные конфликты для выбранной модели и узла.

Визуальный стиль:

- красный акцент;
- четкое предупреждение.

---

UI-текст:


UI-текст:

```text

```

UI-текст:


# 9. Hero-блок

Hero должен сразу давать ответ пользователю.

## Состав hero

Слева:

- фото детали или placeholder;
- бренд;
- SKU;
- название;
- категория.

Центр:

- мотоцикл;
- модель / год / модификация / рынок;
- узел;
- текущая конфигурация, если известна.

Справа:

- итоговый статус;
- confidence level;
- количество отчетов;
- основной CTA.

---

```text
EBC FA209HH
Передние тормозные колодки

BMW R 1250 GS Adventure
2022 • K51 • EU • ABS

Узел:
Тормоза → Передние колодки

Итог:
Подтверждено владельцами

Уверенность:
Высокая

68 отчетов владельцев
```

# 10. Главный verdict-блок

## Название

```text
Итог по совместимости
```

## Формат

Большая карточка с:

- итоговым статусом;
- коротким human-readable выводом;
- confidence level;
- источником данных;
- предупреждением, если есть.

## Пример для COMMUNITY_CONFIRMED

```text
Подтверждено владельцами

Деталь успешно установлена владельцами этой модели.
Большинство отчетов указывает на установку без доработок.

Уверенность: высокая
Источник: community reports + совпадение модели и узла
```

## Пример для MIXED_REPORTS

```text
Есть противоречивые отчеты

Часть владельцев установила деталь без проблем, но есть отчеты о несовместимости.
Проверьте конфигурацию мотоцикла перед покупкой.

Уверенность: средняя
Возможная причина: различия по году, рынку или ABS
```

# 11. Compatibility Breakdown

Это ключевой блок страницы.

## Название

```text
Распределение отчетов владельцев
```

## Что показывать

Для каждого статуса:

- количество пользователей;
- процент от всех отчетов;
- короткую подпись.

## Пример

```text
Подошла без доработок      42 владельца   62%
Подошла с доработкой       11 владельцев  16%
Подошла частично           5 владельцев   7%
Не подошла                 4 владельца    6%
OEM-замена                 6 владельцев   9%
```

## Визуал

Использовать:

- stacked horizontal bar;
- отдельные status cards;
- абсолютные значения;
- проценты;
- краткие пояснения.

Важно:

- не показывать только проценты;
- абсолютные числа обязательны;
- маленькая выборка должна визуально отличаться от большой.

---

# 12. Маленькая выборка

Если отчетов меньше 3, показывать предупреждение:

```text
Данных пока мало

Отчетов недостаточно для уверенного вывода.
Добавьте свой опыт, если вы уже устанавливали эту деталь.
```

Не использовать сильные формулировки вроде:

```text
Деталь точно подходит
```

если выборка маленькая.

---

# 13. Confidence Level

## Не показывать технический score как главное значение

Не делать главным элементом:

```text
83/100
```

Вместо этого использовать:

```text
Высокая уверенность
Средняя уверенность
Низкая уверенность
```

Технический score можно оставить во внутренней логике.

## Tooltip

```text
Учитываются:
- количество отчетов;
- совпадение модели и узла;
- наличие фото;
- связь с сервисными событиями;
- подтверждения других владельцев;
- противоречивые отчеты;
- structured fitment rules.
```

# 14. Source Priority Block

Блок нужен, чтобы пользователь понимал, откуда взялись данные.

## Название

```text
Источник совместимости
```

## Варианты отображения

### Structured

```text
Проверено по правилам совместимости MotoTwin
```

### Community

```text
Подтверждено отчетами владельцев
```

### Hybrid

```text
Совпадает с правилами MotoTwin и подтверждено владельцами
```

### Conflict

```text

```

```text
Внимание: часть пользовательских отчетов противоречит правилам совместимости.
Для ответственных узлов приоритет имеют проверенные правила MotoTwin.
```

# 15. Configuration Compatibility

Очень важный блок для мотоциклов.

## Название

```text
Совместимость по конфигурациям
```

## Что показывать

- год;
- рынок;
- поколение;
- ABS / non-ABS;
- тип тормозной системы;
- размер колес;
- конфигурация цепи;
- stock / aftermarket элементы;
- engine / cooling тип, если важно.

## Формат

Таблица или cards.

Пример:

```text
2021 K51 EU ABS        Подтверждено     18 отчетов
2022 K51 EU ABS        Подтверждено     12 отчетов
2020 K51 US ABS        Мало данных       2 отчета
2018 K50 EU Non-ABS    Противоречиво     4 отчета
```

## Задача блока

Показать, что совместимость может зависеть не только от модели, но и от конфигурации.

---

# 16. Modification Insights

Показывается, если есть отчеты `FIT_WITH_MODIFICATION`.

## Название

```text
Какие доработки встречались
```

## Что показывать

- тип доработки;
- количество отчетов;
- примеры комментариев;
- предупреждение, если доработка критичная.

## Пример

```text
Другая проставка              6 отчетов
Замена крепежного болта       4 отчета
Небольшая подгонка крепления  3 отчета
```

## Если доработки единичные

```text
Есть отдельные отчеты о доработках, но они не являются массовыми.
```

---

# 17. Negative Reports Block

Если есть негативные отчеты, их нельзя прятать.

## Название

```text
Почему деталь не подошла
```

## Показывать

- основные причины несовместимости;
- количество отчетов по причине;
- конфигурации, где возникла проблема.

## Пример

```text
Не совпало крепление             3 отчета
Толщина не подходит под суппорт  2 отчета
Артикул похож, но другая версия  1 отчет
```

---

# 18. Safety Warning

Для ответственных узлов блок обязателен.

## Категории

- тормоза;
- подвеска;
- двигатель;
- электроника;
- критичный крепеж.

## Текст

```text
Ответственный узел

Перед установкой проверьте совместимость по сервисной документации.
Community reports помогают оценить опыт владельцев, но не заменяют техническую проверку.
```

## Для тормозов

```text
Тормозная система влияет на безопасность.
Установка несовместимых деталей может быть опасна.
```

# 19. Ride Profile Compatibility

Показывается только если данных достаточно.

## Название

```text
Лучше всего подходит для
```

## Категории

- город;
- туринг;
- активная езда;
- off-road;
- mixed use;
- агрессивная езда;
- езда с пассажиром / багажом.

## Важно

Это recommendation layer, а не deterministic fitment.

Не писать:

```text
Подходит для агрессивной езды
```

если нет достаточных данных.

Лучше:

```text
Чаще всего положительно оценивают владельцы с профилем: туринг / mixed use.
```

# 32. Что не делать

Не превращать страницу:

- в интернет-магазин;
- в SEO-каталог;
- в форум;
- в social feed;
- в страницу с рекламными описаниями детали.

Не делать главным:

- цену;
- кнопку купить;
- маркетинговое описание бренда;
- отзывы в стиле marketplace.

Главный акцент:

```text
Совместимость + confidence + опыт владельцев такой же модели
```

В первую версию страницы обязательно входит:

# 34. Acceptance Criteria

Страница считается готовой, если:

1. совместимость отображается только в контексте `partId + modelVariantId + nodeId`;
2. пользователь видит итоговый статус совместимости;
3. пользователь видит количество отчетов по каждому статусу;
4. пользователь видит проценты по каждому статусу;
5. пользователь видит абсолютное количество владельцев, а не только проценты;
6. отображается confidence level;
7. различаются structured compatibility и community compatibility;
8. видны отчеты владельцев;
9. видны причины доработок, если они есть;
10. видны причины несовместимости, если они есть;
11. для safety-critical узлов есть warning;
12. есть CTA «Добавить свой опыт»;
13. есть CTA «Добавить в корзину замен»;
14. mobile layout остается читаемым;
15. страница не выглядит как marketplace page;
16. при малом количестве данных система не делает категоричных выводов.
