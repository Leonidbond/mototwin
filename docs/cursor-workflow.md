# Cursor: рабочий процесс в MotoTwin

## Промпты на новые фичи

Каждый запрос на **новую пользовательскую фичу** или **изменение контракта данных/API** должен явно содержать:

1. **Web impact** — какие страницы/поведение в `src/`; либо «нет», либо «отложено» с причиной.  
2. **Expo impact** — какие экраны/поведение в `apps/app/`; аналогично.  
3. **Shared impact** — нужны ли правки в `packages/` (типы, domain, api-client, токены, контракты); либо «нет».  
4. **Parity expectation** — полный паритет по смыслу данных и исходам, намеренные платформенные отличия (перечислить), или документированный разрыв / отложенный клиент.

Без этих четырёх пунктов задачу легко принять так, что один клиент или shared-слой останутся неучтёнными.

Подробный чеклист и **Definition of Done**: [web-mobile-parity-workflow.md](./web-mobile-parity-workflow.md).  
Готовый шаблон промпта: [parity-task-template.md](./parity-task-template.md).  
Общие правила кода и паритета: [coding-rules.md](./coding-rules.md) (раздел 25).

## Прочее

- Соблюдайте ограничения из самого промпта (**Files allowed / must not be changed**).  
- После изменений API сначала выравнивайте **типы и api-client**, затем web и Expo.
