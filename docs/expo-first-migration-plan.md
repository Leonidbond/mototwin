# MotoTwin Expo-First Migration Plan

## Зачем Expo становится основной frontend-платформой

MotoTwin уже имеет рабочий web MVP на Next.js + Prisma + PostgreSQL с ключевыми модулями: гараж, профиль мотоцикла, top nodes, сервисный журнал, обновление состояния, расходы и fitment.  
На следующем этапе основной пользовательский сценарий смещается в мобильный формат: быстрый доступ к данным мотоцикла в дороге, ввод сервисных событий "на месте", регулярная проверка статусов обслуживания и расходов.

Почему Expo-first:
- мобильный UX становится первичным продуктовым каналом;
- одна React-экосистема и единый стек разработки для iOS/Android (+ web через Expo web при необходимости);
- проще переиспользовать доменную логику и типы между платформами;
- быстрее развивать продукт при ограниченных ресурсах MVP-команды.

## Почему миграция должна быть поэтапной (а не rewrite)

Полный одномоментный rewrite создаст высокий риск регрессий по уже работающим функциям MotoTwin MVP (особенно сервисному журналу, расчетам статусов и сценариям гаража).  
Безопаснее идти инкрементально: сначала выделять shared-слой, затем добавлять мобильные экраны, затем доводить web parity и только после этого принимать решение по судьбе Next.js фронтенда.

Преимущества gradual migration:
- сохраняем работающий web как fallback во время перехода;
- переносим и проверяем функциональность по модулям;
- уменьшаем риск блокировки команды на "большом переписывании";
- получаем раннюю ценность на мобильных платформах до полной миграции.

## Целевая структура репозитория

Целевая структура (минимально необходимая для Expo-first):

```text
apps/
  app/                  # Expo app (iOS/Android, при необходимости web)

packages/
  types/                # Общие TS-типы API и доменные типы
  domain/               # Бизнес-логика/форматтеры/правила без UI
  api-client/           # Клиент API (fetch wrappers, auth-aware calls)
  design-tokens/        # (опционально) цвета, отступы, типографика
```

Примечания:
- `apps/app` становится primary frontend entrypoint.
- Next.js frontend остается временно до достижения согласованного уровня паритета.
- Backend (Next.js API routes + Prisma + PostgreSQL) на миграционном этапе не меняется.

## Фазы миграции

## 1) Architecture & planning

Цель: зафиксировать границы shared и platform-specific слоев.

Шаги:
- инвентаризация текущих MVP-модулей в Next.js (гараж, vehicle detail, service log, state update, expenses, fitment);
- фиксация контрактов API (что уже есть и используется фронтендом);
- определение "первой мобильной вертикали" (например: гараж -> карточка мотоцикла -> сервисный журнал);
- определение критериев готовности каждой фазы (definition of done).

Результат фазы:
- документированные границы модулей;
- последовательность переноса экранов без изменения backend-контрактов.

## 2) Monorepo setup

Цель: подготовить структуру для совместного использования кода.

Шаги:
- создать `apps/app` и `packages/*` в текущем репозитории;
- настроить workspace-структуру (без изменений прикладной логики);
- подключить TypeScript path aliases для shared пакетов;
- вынести базовые типы в `packages/types`.

Результат фазы:
- репозиторий готов к инкрементальному переносу кода в shared-пакеты.

## 3) Expo app bootstrap

Цель: получить рабочий скелет мобильного приложения.

Шаги:
- инициализировать Expo app в `apps/app`;
- настроить базовую навигацию экранов и app shell;
- подключить API base URL и базовую авторизацию на клиенте;
- добавить базовые states: loading/error/empty для ключевых экранов.

Результат фазы:
- приложение запускается на iOS/Android и может обращаться к текущему backend.

## 4) Shared package extraction

Цель: уменьшить дублирование и подготовить единый frontend foundation.

Шаги:
- выделить DTO и типы ответов API в `packages/types`;
- выделить `api-client` вызовы (гараж, vehicle detail, service events, profile updates);
- вынести чистые форматтеры/утилиты/доменные функции в `packages/domain` (например форматирование статусов, дат, вспомогательные вычисления для сервисного журнала);
- оставить UI-компоненты платформо-специфичными.

Результат фазы:
- один источник правды для типов и бизнес-логики, который используется и в Expo, и в web.

## 5) Mobile MVP screens

Цель: реализовать мобильный MVP-контур как основной пользовательский путь.

Приоритет экранов:
1. Аутентификация (базовый вход)
2. Гараж
3. Детали мотоцикла
4. Service log (timeline + фильтры MVP-уровня)
5. Добавление сервисного события
6. Обновление состояния (пробег/моточасы)

Принцип:
- сначала функциональный паритет core-flow, затем polish.

Результат фазы:
- мобильное приложение покрывает основной ownership workflow.

## 6) Web parity

Цель: выровнять оставшиеся critical сценарии между web и Expo-first фронтендом.

Шаги:
- добрать экраны/флоу, которые пока есть только в Next.js UI;
- унифицировать формат отображения данных и поведения форм;
- сверить UX и бизнес-правила с существующим MVP.

Результат фазы:
- web и mobile используют один shared foundation и сопоставимые сценарии.

## 7) Next.js decommission decision

Цель: принять управляемое решение, оставлять ли Next.js frontend.

Критерии решения:
- мобильный клиент покрывает agreed MVP/PMF-сценарии;
- нет блокирующих platform-specific gap;
- эксплуатационные метрики и качество стабильны.

Возможные варианты:
- оставить Next.js как light web клиент;
- перевести web на Expo web;
- постепенно decommission Next.js UI при сохранении backend API.

## Что становится shared

К shared-слою переводим:
- API-контракты и DTO типы (`packages/types`);
- доменные типы MotoTwin (vehicle, node status, service event, expense и т.д.);
- API client функции (`packages/api-client`);
- чистую бизнес-логику и форматтеры (`packages/domain`);
- (опционально) design tokens (`packages/design-tokens`).

## Что остается platform-specific

Platform-specific остается:
- навигация экранов и layout (Expo navigation vs web router);
- platform UI-компоненты и интеракции (touch patterns, модалки, жесты);
- локальные UX-паттерны (например mobile-first формы и плотность контента);
- web-only особенности (если web останется отдельным клиентом).

## Риски миграции и как их снижать

## 1) Риск: регрессии в уже работающем MVP
- Снижение: перенос по вертикальным слайсам, smoke-checklist для каждого модуля, сохранение web fallback до стабилизации mobile.

## 2) Риск: дублирование логики между web и mobile
- Снижение: ранний вынос `types/domain/api-client` и правило "новая бизнес-логика только в shared пакете".

## 3) Риск: расползание scope и затяжная миграция
- Снижение: фазовые критерии done, фокус на core ownership workflow, жесткий MVP backlog.

## 4) Риск: несовместимость UX ожиданий между платформами
- Снижение: единые продуктовые сценарии и acceptance criteria, platform-specific только на уровне представления.

## 5) Риск: сложность решения по валютам/форматам/локализации
- Снижение: хранить источник правды в backend и shared типах, форматирование централизовать в `packages/domain`.

## Практический порядок запуска работ (первые 4-6 недель)

1. Зафиксировать target architecture и карту модулей.  
2. Поднять monorepo-структуру и shared пакеты (`types`, `api-client`).  
3. Запустить Expo app shell с авторизацией и базовым API ping.  
4. Перенести core-flow: гараж -> карточка мотоцикла -> сервисный журнал.  
5. Вынести повторяющиеся форматтеры и доменную логику в `packages/domain`.  
6. Провести review готовности к следующей фазе (web parity / расширение mobile функционала).

---

Этот план описывает целевое направление и этапы миграции. Он не означает, что миграция уже выполнена: текущий рабочий frontend MotoTwin остается на Next.js до достижения согласованного уровня готовности Expo-first приложения.
