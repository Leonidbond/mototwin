# MotoTwin — полный дизайн-концепт для Cursor

> [!IMPORTANT]
> Актуальная версия дизайн-правил перенесена в единый документ: `docs/mototwin_unified_design_concept.md`.
> Этот файл сохранен как исторический источник и базовый reference.

## Назначение документа

Этот документ описывает полный дизайн-концепт MotoTwin для реализации через Cursor.

Фокус документа:

1. страница “Мой гараж” со списком мотоциклов;
2. карточка мотоцикла;
3. страница конкретного мотоцикла;
4. мобильная версия;
5. светлая и темная темы;
6. визуальный стиль силуэта мотоцикла;
7. UX-состояния;
8. требования к реализации в MVP.

Цель: сделать MotoTwin визуально сильным продуктом для мотолюбителей, но не перегрузить MVP сложной графикой, 3D, анимациями и модельными изображениями.

---

# 1. Общая идея дизайна

MotoTwin должен выглядеть как **цифровой гараж**, а не как таблица учета ремонтов.

Главное ощущение:

> “Это мой мотоцикл, его состояние, история обслуживания, расходы и ближайшие задачи в одном интерфейсе.”

Интерфейс должен объединять:

- эмоцию владения мотоциклом;
- техническую аккуратность;
- понятный контроль состояния;
- быстрые действия;
- премиальный, но сдержанный визуальный стиль.

MotoTwin не должен выглядеть как:

- обычная CRM;
- таблица ремонтов;
- маркетплейс;
- дешевая игровая панель;
- перегруженный киберпанк-dashboard.

- технологичный;

- взрослый;
- премиальный;
- понятный;
- mobile-first;

Интерфейс должен ассоциироваться с:

- гаражом;

- технической картой;
- цифровым двойником;

- контролем готовности к поездке.

# 3. Основные дизайн-принципы

## 3.1. Мотоцикл в центре

Главный объект интерфейса — не таблица, а мотоцикл пользователя.

Даже на странице списка мотоциклов каждая карточка должна ощущаться как мини-дашборд конкретного мотоцикла.

## 3.2. Силуэт вместо фотографии

На этапе MVP не использовать реалистичные фотографии конкретных моделей.

Причина: владельцы разных марок и годов могут быть недовольны, если визуал не совпадает с их реальным мотоциклом.

Правильная формула:

```text
Схематичный силуэт класса + точные данные конкретного мотоцикла
```

## 3.3. Состояние должно быть видно сразу

Пользователь должен за 3–5 секунд понять:

- все ли в порядке;
- есть ли просрочка;
- что скоро обслуживать;
- что сделать дальше.

## 3.4. Минимум визуального шума

Не использовать слишком много декоративных элементов.

Красота должна строиться на:

- хорошем layout;
- контрасте;
- аккуратных карточках;
- типографике;
- статусах;
- технических силуэтах.

## 3.5. MVP-safe

Дизайн должен быть реализуемым в MVP:

- без 3D;
- без тяжелых анимаций;
- без сложных canvas-сцен;
- без модельных изображений для каждой версии;
- без лишних библиотек.

# 4. Темная тема

Темная тема является основной визуальной темой MotoTwin.

## 4.1. Цветовая палитра

```css
--background-main: #0B0F14;
--background-soft: #0F141B;
--card-main: #111827;
--card-soft: #161C24;
--border-muted: #263241;

--text-primary: #F9FAFB;
--text-secondary: #9CA3AF;
--text-muted: #6B7280;

--accent-orange: #F97316;
--accent-orange-hover: #FB923C;
--accent-orange-soft: rgba(249, 115, 22, 0.12);

--status-ok: #22C55E;
--status-soon: #FACC15;
--status-overdue: #EF4444;
--status-recently: #38BDF8;
```

## 4.2. Использование цветов

Оранжевый использовать для:

- главной CTA-кнопки;
- активного пункта навигации;
- логотипа;
- маленьких акцентов;
- выделения важных действий.

Статусные цвета использовать только для статусов:

- OK;
- Soon;
- Overdue;
- Recently.

Не использовать статусные цвета как декоративные.

# 5. Светлая тема

Светлая тема должна быть не “белой копией темной”, а чистой технической версией.

## 5.1. Цветовая палитра

```css
--background-main: #F8FAFC;
--background-soft: #F1F5F9;
--card-main: #FFFFFF;
--card-soft: #F8FAFC;
--border-muted: #E5E7EB;

--text-primary: #111827;
--text-secondary: #4B5563;
--text-muted: #9CA3AF;

--accent-orange: #F97316;
--accent-orange-hover: #EA580C;
--accent-orange-soft: rgba(249, 115, 22, 0.10);

--status-ok: #16A34A;
--status-soon: #EAB308;
--status-overdue: #EF4444;
--status-recently: #0284C7;
```

## 5.2. Визуальный характер светлой темы

Светлая тема должна быть:

- чистой;
- воздушной;
- менее агрессивной;
- хорошо читаемой;
- похожей на современное сервисное приложение;
- без ощущения “медицинской формы” или бухгалтерии.

## 5.3. Карточки в светлой теме

Карточки:

- белые;
- с легкой тенью;
- с тонким серым бордером;
- с радиусом 16–20 px;
- с аккуратным разделением блоков.

Пример:

```css
background: #FFFFFF;
border: 1px solid #E5E7EB;
box-shadow: 0 8px 24px rgba(15, 23, 42, 0.06);
border-radius: 20px;
```

# 6. Типографика

Рекомендуемые шрифты:

- Inter;
- Manrope;
- Geist;
- IBM Plex Sans.

## 6.1. Заголовки

Стиль:

- плотные;
- уверенные;
- без чрезмерной декоративности;
- хорошо читаемые.

Пример:

```text
Мой гараж
BMW F 850 GS Adventure
```

## 6.2. Цифры

Цифры должны быть выразительными:

```text
82 / 100
42 180 км
38 700 ₽
1 800 км
```

Можно использовать тот же шрифт, но с большим размером и весом.

## 6.3. Текст интерфейса

UI-тексты на русском языке.

Код, переменные, компоненты, имена файлов — на английском.

---

# 7. Иконки

Иконки должны быть:

- outline;
- технические;
- без мультяшности;
- 1.5–2 px stroke;
- понятные;
- в одном стиле.

Нужные иконки:

- мотоцикл;
- двигатель;
- тормозной диск;
- шина;
- цепь;
- аккумулятор;
- подвеска;
- масляная канистра;
- фильтр;
- гаечный ключ;
- календарь;
- кошелек / расходы;
- журнал;
- профиль.

Для MVP можно использовать иконки из уже подключенной библиотеки, если она есть. Не подключать новую библиотеку без необходимости.

# 8. Силуэты мотоциклов

## 8.1. Главный принцип

Силуэт мотоцикла — это не точная модель, а **схематичный вид класса**.

Подпись:

```text
Схематичный вид · Adventure class
```

или:

```text
Схематичный вид · Cruiser class
```

## 8.2. Почему не использовать точные модели

Не использовать фотореалистичные изображения, потому что:

- не тот год;
- не тот цвет;
- не та комплектация;
- не тот обвес;
- не тот руль;
- не та фара;
- не тот выхлоп;
- пользователь может почувствовать, что продукт “не про его мотоцикл”.

## 8.3. Классы силуэтов для MVP

Рекомендуемые классы:

1. Adventure / Touring
2. Enduro / Dual-sport
3. Naked / Roadster
4. Sport / Supersport
5. Cruiser
6. Classic / Retro
7. Scooter / Maxi-scooter

Для первого MVP достаточно:

1. Adventure
2. Enduro
3. Naked
4. Sport Touring
5. Cruiser

## 8.4. Стиль силуэта

Силуэт должен быть:

- монохромным;
- в оттенках серого;
- в стиле technical line art;
- без логотипов;
- без узнаваемых брендовых деталей;
- без цветного пластика;
- без фотореализма.

## 8.5. Размер силуэта в мобильном приложении

На мобильной странице списка мотоциклов силуэт не должен занимать слишком много места.

Рекомендуемый размер:

- примерно 35–45% ширины карточки;
- в 2 раза меньше, чем большой hero-силуэт на странице конкретного мотоцикла;
- рядом или напротив Garage Score.

Принцип:

```text
Силуэт помогает распознать класс, но не доминирует над данными.
```

# 9. Страница “Мой гараж” — desktop

## 9.1. Назначение

Страница “Мой гараж” — главный экран после авторизации.

Она должна отвечать на вопросы:

1. какие мотоциклы у меня есть;
2. в каком они состоянии;
3. что требует внимания;
4. какие ближайшие задачи;
5. сколько расходов за сезон;
6. что можно быстро сделать.

## 9.2. Общая структура

```text
Sidebar

- Мой гараж
- подзаголовок
- уведомления

- 3 ближайших задачи

- BMW F 850 GS Adventure
- KTM 790 Adventure
- Add motorcycle card

Upcoming tasks:
- проверить колодки
- замена масла
- задняя шина просрочена
```

```text
MotoTwin

Узлы
Журнал
```



- крупное число;

## 9.6. Vehicle cards grid

На desktop:

- 1 колонка на узком экране;
- 2 колонки на среднем;
- 2–3 колонки на широком, если позволяет layout.

Карточки должны быть широкими, не слишком мелкими.

# 10. Карточка мотоцикла

## 10.1. Содержание карточки

Каждая карточка должна содержать:

1. название мотоцикла;
2. год;
3. пробег;
4. ride profile;
5. меню “...”;
6. монохромный силуэт класса;
7. подпись класса;
8. Garage Score;
9. статусы OK / Soon / Overdue / Recently;
10. блок “Требует внимания” или “Все в порядке”;
11. быстрые действия.

## 10.2. Пример карточки

```text
BMW F 850 GS Adventure
2021 · 42 180 км · Mixed Touring

[малый силуэт Adventure]        Garage Score
                                82 / 100

7 OK · 2 Soon · 1 Overdue · 0 Recently

Требует внимания:
- Задняя шина · Просрочено
- Тормозные колодки · Скоро

[Открыть] [Добавить ТО] [Расход]
```

## 10.3. Состояние “все хорошо”

```text
KTM 790 Adventure
2020 · 31 400 км · Off-road

Garage Score
91 / 100

9 OK · 1 Soon · 0 Overdue

Все в порядке
Следующее ТО через 1 200 км
```

## 10.4. Кнопки карточки

Основная кнопка:

```text
Открыть
```

Вторичные:

```text
Добавить ТО
Расход
```

На мобильном можно сокращать:

```text
ТО
Расход
```

# 11. Empty state гаража

Если у пользователя нет мотоциклов, показать отдельное состояние.

## 11.1. Текст

```text
У вас пока нет мотоциклов

Добавьте первый мотоцикл, чтобы вести историю обслуживания,
получать напоминания и подбирать совместимые детали.

[Добавить мотоцикл]
```

## 11.2. Визуал

Можно использовать:

- схематичный гараж;
- серый силуэт мотоцикла;
- легкую техническую иллюстрацию;
- без фотореализма.

## 11.3. Функциональные блоки в empty state

Показать 3 преимущества:

```text
Контроль обслуживания и регламентов
Напоминания о важных заменах
Подбор совместимых запчастей и расходников
```

# 13. Мобильная карточка мотоцикла

## 13.1. Структура

```text
BMW F 850 GS Adventure       [...]
2021 · 42 180 км · Mixed Touring

[малый силуэт]     82 / 100
                   Garage Score

7 OK · 2 Soon · 1 Overdue · 0 Recently

Требует внимания
Задняя шина · Просрочено
Тормозные колодки · Скоро

[Открыть] [ТО] [Расход]
```

## 13.2. Размер силуэта

Силуэт должен быть маленьким.

Рекомендуемый принцип:

- не больше половины ширины внутреннего ряда;
- не больше 90–120 px по высоте;
- не занимать весь hero-блок;
- основное внимание на статус и actions.

## 13.3. Поведение карточки

Клик по карточке или кнопке “Открыть” ведет на страницу конкретного мотоцикла.

Кнопка “ТО” открывает добавление сервисного события.

Кнопка “Расход” открывает добавление расхода.

# 15. Карта мотоцикла на странице конкретного мотоцикла

На странице конкретного мотоцикла силуэт может быть больше, чем в списке гаража.

Но он все равно должен быть:

- схематичным;
- монохромным;
- без точной модели;
- с узлами вокруг.

## 15.1. Узлы вокруг силуэта

Показывать точки:

```text
Двигатель
Тормоза
Шины
Цепь / звезды
Аккумулятор
Подвеска
```

Каждая точка должна иметь статус.

## 15.2. Подпись

```text
Схематичный вид · Adventure class
```

или:

```text
Схематичный вид · Cruiser class
```

# 16. Блок “Требует внимания”

## 16.1. Назначение

Этот блок должен быть одним из самых важных.

Он отвечает на вопрос:

> “Что мне нужно сделать сейчас?”

## 16.2. Пример

```text
Требует внимания

Задняя шина
Просрочено
Рекомендуется замена
[Подобрать шину]

Тормозные колодки
Скоро
Проверить через 450 км
[Открыть узел]

Масло двигателя
Скоро
Следующая замена через 1 800 км
[Добавить ТО]
```

## 16.3. Визуальные требования

Карточки:

- компактные;
- с иконкой узла;
- с цветным статусом;
- с коротким пояснением;
- с chevron или кнопкой действия.

# 23. Garage Score

## 23.1. Назначение

Garage Score — агрегированный индикатор состояния мотоцикла.

Он помогает быстро понять состояние техники.

## 23.2. Формат

```text
Garage Score
82 / 100
```

## 23.3. Визуал

Можно использовать:

- круговой индикатор;
- дугу;
- крупную цифру;
- цвет в зависимости от состояния.

## 23.4. Цветовая логика

```text
90–100: зеленый
70–89: оранжевый / желтый
0–69: красный
```

Для MVP можно не реализовывать сложный расчет, но UI должен быть готов.

# 27. Layout guidelines

## 27.1. Радиусы

```css
border-radius: 16px;
border-radius: 20px;
```

Для крупных карточек:

```css
border-radius: 24px;
```

## 27.2. Отступы

Использовать щедрые, но не чрезмерные отступы:

```css
padding: 16px;
padding: 20px;
padding: 24px;
```

На mobile:

```css
padding: 16px;
gap: 12px;
```

## 27.3. Карточки

Карточки должны быть визуально отделены от фона, но не слишком контрастны.

Темная тема:

```css
background: #111827;
border: 1px solid #263241;
```

Светлая тема:

```css
background: #FFFFFF;
border: 1px solid #E5E7EB;
box-shadow: 0 8px 24px rgba(15, 23, 42, 0.06);
```

# 28. Что не делать в MVP

Не делать:

- 3D-модели мотоциклов;
- фотореалистичные изображения;
- точные силуэты каждой модели;
- marketplace;
- community;
- telemetry;
- hardware/CAN-интеграции;
- сложные графики;
- сложные фильтры;
- тяжелые анимации;
- перегруженный sidebar;
- случайные цвета;
- большие декоративные иллюстрации, мешающие данным.

# 29. Acceptance Criteria

Результат считается готовым, если:

1. Страница “Мой гараж” выглядит как цифровой гараж.
2. Пользователь видит список мотоциклов в виде карточек.
3. Карточка каждого мотоцикла показывает состояние.
4. Силуэт маленький, монохромный и не похож на конкретную модель.
5. Есть Garage Score.
6. Есть статусы OK / Soon / Overdue / Recently.
7. Есть блок “Требует внимания”.
8. Есть быстрые действия.
9. Есть empty state для пустого гаража.
10. Есть mobile layout.
11. Есть dark theme.
12. Есть light theme.
13. UI не перегружен.
14. Тексты на русском.
15. Код и названия компонентов на английском.
16. Не добавлены лишние библиотеки.
17. Интерфейс соответствует MVP, а не future platform.

# 33. Итоговая формула дизайна

Главная формула MotoTwin UI:

```text
Точный профиль мотоцикла
+ схематичная карта класса
+ понятные статусы
+ быстрые действия
+ история обслуживания
+ расходы
= цифровой гараж
```

MotoTwin должен понравиться мотолюбителю не потому, что показывает красивую картинку, а потому что дает ощущение:

> “Я контролирую свой мотоцикл. Я знаю, что с ним происходит. Я не ошибусь с обслуживанием и деталями.”

Это и есть главный UX-принцип продукта.
