Главная задача страницы:

**показать владельцу, сколько реально стоит владение мотоциклом, куда уходят деньги и какие узлы самые дорогие.**

# 1. Верхняя сводка

Вверху страницы 4–6 ключевых карточек:

### Основные показатели

1. **Всего расходов**

   * за выбранный период;
   * например: `128 450 ₽`.

2. **Расходы за сезон**

   * текущий мотосезон;
   * например: `Апрель–Октябрь 2026`.

3. **Средний расход в месяц**

   * полезно для понимания стоимости владения.

4. **Средний расход на 1 000 км**

   * очень сильная метрика для мотоциклистов.

5. **Количество событий с расходами**

   * например: `17 событий`.

6. **Самый дорогой узел**

   * например: `Шины — 42 000 ₽`.

# 3. Разрез «по узлам»

Это один из главных экранов.

Показываем, сколько денег ушло на каждый узел.

Пример:

| Узел               |    Сумма | Событий | Доля |
| ------------------ | -------: | ------: | ---: |
| Шины               | 42 000 ₽ |       2 |  33% |
| Двигатель / смазка | 18 500 ₽ |       5 |  14% |
| Тормоза            | 16 800 ₽ |       4 |  13% |
| Цепь / звезды      | 14 200 ₽ |       2 |  11% |
| Подвеска           |  9 500 ₽ |       1 |   7% |

По клику на узел открывается детализация:

* какие события были;
* какие детали установлены;
* какие работы оплачивались;
* сколько стоили запчасти;
* сколько стоила работа;
* дата;
* пробег;
* комментарий.

Это хорошо связывается с уже существующей логикой узлов и service log.

# 4. Разрез «по времени»

Нужен график расходов по месяцам.

### Варианты отображения:

1. **Месячная динамика**

   * январь, февраль, март и т. д.

2. **Накопительный итог**

   * сколько всего вложено в мотоцикл с начала года / сезона / владения.

3. **Сравнение сезонов**

   * сезон 2025 vs сезон 2026.

4. **Пики расходов**

   * система подсвечивает месяцы, где были крупные траты.

Пример инсайта:

> В мае 2026 расходы выше обычного из-за замены шин и обслуживания тормозов.

# 6. Категории расходов

Я бы разделил расходы не только по узлам, но и по экономическому смыслу.

### Категории:

1. **Запчасти**

   * колодки, фильтры, цепь, звезды.

2. **Расходники**

   * масло, жидкости, смазки.

3. **Работы сервиса**

   * стоимость механика / СТО.

4. **Шины**

   * отдельно, потому что это крупная статья.

5. **Внеплановый ремонт**

   * поломки, аварийные замены.

6. **Тюнинг / апгрейды**

   * не обязательное обслуживание.

7. **Диагностика**

   * осмотры, проверки, компьютерная диагностика.

8. **Прочее**

   * парковка, мойка, доставка деталей и т. п.

Я бы не смешивал **обязательное обслуживание** и **тюнинг**, иначе стоимость владения будет искажена.

# 14. Важные вопросы

Нужно уточнить 5 вещей перед ТЗ для Cursor:

1. **Что считаем расходами?**
   Только обслуживание / запчасти / ремонт или также бензин, страховка, штрафы, парковка, мойка, экипировка?

2. **Как определяем сезон?**
   Фиксированно апрель–октябрь или пользователь сам задает сезон для каждого мотоцикла?

3. **Нужна ли мультивалютность?**
   Сейчас делать только ₽ или сразу заложить `currency`?

4. **Расход может быть без сервисного события?**
   Например, купил масло заранее, но еще не установил.

5. **Нужен ли учет “куплено / установлено”?**
   Это важно для связки со списком покупок: пользователь мог купить деталь сейчас, а поставить через месяц.

Моя рекомендация: **сразу заложить расходы как отдельные ExpenseItem, которые могут быть привязаны к ServiceEvent, но не обязаны.** Тогда система поддержит и “купил заранее”, и “установил в сервисе”, и “просто внес расход”.


 Фиксируем логику блока расходов так:

## Принятые решения

**Расходами считаем только:**

* обслуживание;
* запчасти;
* ремонт.

Не включаем в этот блок:

* бензин;
* страховку;
* штрафы;
* парковку;
* мойку;
* экипировку.

Это правильно: страница будет считать именно **стоимость технического владения мотоциклом**, а не все бытовые расходы владельца.

---

## Сезон

Раз сезон = **календарный год**, то в интерфейсе лучше не называть это «мотосезон апрель–октябрь», а использовать формулировку:

**Сезон 2026**

По сути:

`01.01.2026 — 31.12.2026`

Это проще для MVP и хорошо подходит для аналитики:

* расходы за год;
* сравнение сезонов 2025 / 2026;
* стоимость владения за календарный период;
* динамика расходов по месяцам внутри сезона.

# 5. Статус расхода: куплено / установлено

Это важная часть.

## ExpenseLifecycleStatus

```ts
PURCHASED
INSTALLED
PURCHASED_NOT_INSTALLED
CANCELLED
```

Но лучше сделать проще и надежнее:

```ts
purchaseStatus: PURCHASED | PLANNED
installationStatus: NOT_INSTALLED | INSTALLED
```

Тогда можно описать разные ситуации:

| Сценарий                            | purchaseStatus | installationStatus |
| ----------------------------------- | -------------- | ------------------ |
| Купил деталь, еще не поставил       | `PURCHASED`    | `NOT_INSTALLED`    |
| Купил и сразу поставил              | `PURCHASED`    | `INSTALLED`        |
| Запланировал покупку                | `PLANNED`      | `NOT_INSTALLED`    |
| Работа сервиса без отдельной детали | `PURCHASED`    | `INSTALLED`        |

Для MVP можно оставить один enum:

```ts
ExpenseStatus {
  PURCHASED
  INSTALLED
  PURCHASED_NOT_INSTALLED
}
```

Но с точки зрения будущей логики лучше разделить покупку и установку.

# 6. Связь со списком покупок

Правильная модель:

**ShoppingListItem → ExpenseItem → ServiceEvent**

Пользователь может:

1. добавить позицию в список покупок;
2. отметить ее как купленную;
3. после покупки создать расход;
4. позже установить деталь;
5. при установке связать расход с сервисным событием.

Пример:

### Купил заранее

`Масляный фильтр HF155 — 900 ₽`

* есть `ExpenseItem`;
* есть `ShoppingListItem`;
* нет `ServiceEvent`;
* статус: `куплено, не установлено`.

### Потом установил

`Замена масла и фильтра`

* появляется `ServiceEvent`;
* расход связывается с событием;
* статус меняется на `установлено`.

```ts
currency: string
```

Примеры:

```ts
amount: 4200
currency: "RUB"
```

```ts
amount: 85
currency: "EUR"
```

Для MVP не обязательно сразу делать конвертацию валют. Можно просто хранить валюту и показывать суммы отдельно.

На следующем этапе можно добавить:

```ts
exchangeRateToBase?: Decimal
baseCurrencyAmount?: Decimal
baseCurrency?: string
```

Например, пользователь купил деталь за евро, а аналитика считается в рублях.

# 11. Реализованная интеграция с деревом узлов

Расходы не добавляются как отдельный dashboard в дерево. Дерево остаётся деревом узлов, а расходы показываются как контекст к узлу.

## API

```http
GET /api/expenses/node-summary?vehicleId=<vehicleId>&year=<year>
```

Ответ возвращает агрегаты по `nodeId`:

```ts
{
  nodeId: string
  totalByCurrency: { currency: string; amount: number }[]
  expenseCount: number
  purchasedNotInstalledCount: number
  latestExpenses: {
    id: string
    date: string
    title: string
    amount: number
    currency: string
    category: ExpenseCategory
    installationStatus: ExpenseInstallationStatus
  }[]
}
```

## Правила дерева

* leaf node показывает только расходы своего `nodeId`;
* parent node показывает расходы своего `nodeId` и всех дочерних узлов любого уровня;
* сезон = календарный год;
* разные валюты показываются отдельно и не конвертируются;
* узлы без расходов не получают пустых строк;
* “куплено, не установлено” считается только для:
  * `purchaseStatus = PURCHASED`;
  * `installationStatus = NOT_INSTALLED`;
  * `serviceEventId = null`.

## UI

В свернутом состоянии узла показывается компактно:

```md
Расходы за сезон: 8 500 ₽
Куплено, не установлено: 1
```

В раскрытом узле показывается блок:

```md
Расходы по узлу
8 500 ₽ за сезон 2026
3 расхода
Куплено, не установлено: 1

Последние расходы:
- 14.04.2026 Передние колодки — 6 800 ₽
- 14.04.2026 Работа сервиса — 1 700 ₽

Все расходы по узлу →
```

Ссылка:

```md
/expenses?vehicleId=<vehicleId>&nodeId=<nodeId>&year=<year>
```

Страница `/expenses` трактует такой `nodeId` как выбранный узел плюс всё поддерево, чтобы переход из parent node не давал пустой список.

# 12. Правильная формулировка для Cursor

```md
Нужно реализовать полнофункциональную страницу расходов MotoTwin.

Расходами считаются только технические расходы: обслуживание, запчасти и ремонт. Бензин, страховка, штрафы, парковка, мойка и экипировка не входят в expense analytics.

Сезон определяется как календарный год. Например, сезон 2026 = 01.01.2026–31.12.2026.

Расход должен поддерживать currency с самого начала.

Расход может существовать без ServiceEvent. Это нужно для сценария, когда пользователь купил деталь заранее, но еще не установил ее.

Нужно поддержать статус куплено / установлено:
- куплено, но не установлено;
- установлено;
- не применимо для расходов без установки, например диагностика или работа сервиса.

Страница /expenses должна показывать:
- общую сумму расходов;
- расходы за выбранный год / сезон;
- расходы по узлам;
- расходы по категориям;
- расходы по месяцам;
- список всех расходов;
- отдельный показатель “куплено, но не установлено”.

ExpenseItem должен иметь optional связь с ServiceEvent и optional связь с ShoppingListItem.
```

Моя рекомендация: **сначала реализовать ExpenseItem как самостоятельную сущность**, а уже потом аккуратно связать ее с журналом обслуживания и списком покупок. Так мы не сломаем существующий service log и сможем быстро получить рабочую страницу расходов.
