# «Требует внимания» (MVP)

**Дата:** 2026-04-18  
**Статус:** MVP на существующем дереве узлов и `statusExplanation` из API. Список гаража дополняется лёгким **`attentionSummary`** на тот же расчёт, что и `buildAttentionSummaryFromNodeTree` (без отдельных запросов `node-tree` с клиента на каждую карточку).

## Смысл

Кнопка **«Требует внимания»** рядом с названием мотоцикла открывает отдельный экран (Expo) или модалку (web) со списком узлов, по которым уже рассчитаны статусы **«Просрочено»** или **«Скоро»**. Данные — те же, что в `GET …/node-tree`: `effectiveStatus`, `statusExplanation`, иерархия узлов.

## Какие статусы попадают в список

- **Включены:** `OVERDUE`, `SOON` (`effectiveStatus`).
- **Исключены:** `OK`, `RECENTLY_REPLACED`, `null` / отсутствующий статус.

## Отбор и сортировка

- **Приоритет:** сначала все **Просрочено**, затем **Скоро**.
- **Дедупликация:** предпочитаются **наиболее конкретные** узлы (обычно листья). Если у родителя и у потомка в поддереве есть `OVERDUE`/`SOON`, в списке остаются потомки (родитель с тем же «проблемным» поддеревом не дублируется). Если проблемный только внутренний узел без проблемных потомков — показывается он.
- Порядок внутри одной группы приоритета — **обход дерева** (корни по порядку API, затем глубина).

## Счётчик на кнопке

Число на кнопке — **количество элементов списка** после отбора (сумма просроченных и «скоро»), одинаково на web и Expo.

## Цвет кнопки «Требует внимания»

Семантика окраски общая для web и Expo, через **`buildAttentionActionViewModel`** / **`getAttentionActionSeverity`** (`packages/domain/src/attention.ts`) и **`statusSemanticTokens`** (`@mototwin/design-tokens`):

- есть хотя бы один **OVERDUE** в списке → стиль **критический** (`OVERDUE`);
- иначе есть хотя бы один **SOON** → стиль **предупреждение** (`SOON`);
- иначе (список пуст) → **нейтральный** вид (`semanticKey` **`UNKNOWN`**, без «тревожного» красного/янтаря).

Счётчик остаётся читаемым во всех состояниях (фон бейджа через `accent` токена или нейтральный `divider` при пустом списке).

## Гараж: индикатор внимания у названия

- **Данные:** `GET /api/garage` добавляет в каждый элемент поле **`attentionSummary`**: `{ totalCount, overdueCount, soonCount }` — те же числа, что даёт `buildAttentionSummaryFromNodeTree` для полного дерева (сервер один раз грузит общий каталог узлов и батчит `nodeState` / `serviceEvent` по всем мотоциклам в ответе).
- **UI:** web (`src/app/garage/page.tsx`) и Expo (`apps/app/app/index.tsx`) показывают **в одной строке с заголовком** компактный **inline-чип** (пилюля уровня метаданных карточки: `text-xs` / ~12pt, тонкая рамка `1px`, без «кнопочной» тени и без чрезмерного padding), только если `totalCount > 0`. Цвет — **`statusSemanticTokens`**, та же семантика, что у «Требует внимания» на ТС (**`buildGarageAttentionIndicatorViewModel`** / **`buildAttentionActionViewModel`**): OVERDUE → красный набор, иначе SOON → янтарный; при `totalCount === 0` индикатор **скрыт** (не вводим в заблуждение).
- **Вёрстка:** длинное название **обрезается** (`truncate` / `numberOfLines`), чип **`shrink-0`** остаётся видимым справа.
- **Навигация:** чип — отдельная цель: **клик/тап по индикатору** ведёт на ту же страницу ТС, что и ссылка/кнопка с названием (`/vehicles/[id]`). Отдельной модалки внимания с гаража нет.
- **Клиент без поля:** если `attentionSummary` отсутствует (старый ответ), **`buildGarageAttentionIndicatorViewModel`** считает индикатор невидимым.

## Детальное пояснение расчёта

**Отдельного** разворачиваемого блока «подробный расчёт» только для экрана внимания **нет**.

- Если у узла в API есть **`statusExplanation`**, в карточке внимания показывается та же **кликабельная краткая строка**, что и у узла в дереве (`shortExplanation` / `shortExplanationLabel`), и по нажатию открывается **та же модалка полного пояснения**, что и при объяснении из дерева узлов (web: одна модалка на странице ТС; Expo: `StatusExplanationModal` на экране ТС и на экране внимания — один компонент, одинаковая структура полей).
- Если **`statusExplanation` нет** — полная модалка недоступна; краткая строка ведёт себя как в дереве (только текст без открытия модалки).
- Контент модалки: те же подписи и блоки, что в пояснении дерева (`reasonShort`, `reasonDetailed`, текущее состояние, последний сервис, правило обслуживания, использование / остаток, `triggeredBy` при наличии) — без сырого JSON.

## Скоро

Поведение как у дерева: при наличии **`statusExplanation`** доступно открытие **той же** модалки пояснения; без неё остаётся только некликабельная краткая строка, если она есть.

## Действия с карточки

- **Журнал по узлу** — открывает журнал обслуживания с **фильтром по узлу/поддереву** (`createServiceLogNodeFilter`), как при тапе по статусу в дереве.
- **Добавить сервис** — только для **листьев** (`canAddServiceEvent`); на Expo с предзаполнением `nodeId` в форме нового события.

## Web

- Кнопка у заголовка ТС на `src/app/vehicles/[id]/page.tsx`.
- Модалка **«Требует внимания»** (`z-index` ниже модалки пояснения статуса узла), без отдельного постоянного блока в теле страницы.
- Журнал открывается в **существующей** модалке журнала; при переходе из внимания сбрасывается режим «только расходы» (`paidOnly`), чтобы не мешать фильтру по узлу.

## Expo

- Кнопка-«пилюля» в карточке идентичности рядом с названием на `apps/app/app/vehicles/[id]/index.tsx`.
- Экран `vehicles/[id]/attention.tsx`: заголовок стека **«Требует внимания»**, при фокусе подгружается `getNodeTree` (как журнал подгружает свои данные).

## Shared

- Типы: `packages/types/src/attention.ts` (`AttentionItemViewModel`, `AttentionSummaryViewModel`, `AttentionActionViewModel`, `canOpenStatusExplanation`, …).
- Логика: `packages/domain/src/attention.ts` — `buildAttentionSummaryFromNodeTree`, `getAttentionItemsFromNodeTree`, `groupAttentionItemsByStatus`, `sortAttentionItemsByPriority`, `getWorstAttentionStatus`, `getAttentionActionSeverity`, `buildAttentionActionViewModel`; гараж: `packages/domain/src/garage-attention.ts` — `buildGarageAttentionIndicatorViewModel`, `getAttentionSeverityFromStatuses`.

## QA (кратко)

1. Есть OVERDUE и SOON — счётчик, порядок секций, подписи статусов.  
2. Кнопка «Требует внимания»: красная при OVERDUE, янтарная при только SOON, нейтральная при пустом списке.  
3. Пояснение из карточки внимания открывает **ту же** модалку, что из дерева; содержимое совпадает.  
4. Пустое дерево проблем — **«Нет узлов, требующих внимания»**.  
5. **Журнал по узлу** из внимания совпадает по смыслу с кликом по статусу в дереве.  
6. Дерево, расходы, профиль, состояние, добавление сервиса — без регрессий.  
7. **Гараж:** у мотоцикла с вниманием — компактный чип у названия (число + цвет OVERDUE/SOON), по высоте в ритме заголовка; без проблемных узлов чипа нет; переход на ТС и **с названия**, и **с чипа**; длинное имя не ломает строку; список гаража не делает N+1 `node-tree` с клиента.
