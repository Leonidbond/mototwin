# MotoTwin — единый дизайн-концепт

## Назначение документа

Этот документ является единым актуальным источником дизайн-правил для MotoTwin.

Цель: дать один актуальный источник правил для web и mobile, с учетом уже реализованных фич (полное дерево узлов, полный журнал обслуживания, top-node overview).

## Анализ референсов (images/examples)

### 1) Общий визуальный паттерн

По всем референсам читается единый стиль **technical premium dashboard**:

- темная тема как базовая (`navy/graphite`);
- оранжевый как основной action-accent;
- статусные цвета используются как семантика состояния, а не как декор;
- карточная сетка с четкими границами, умеренными радиусами и высокой контрастностью текста;
- визуальная иерархия строится вокруг состояния техники и следующего действия.

Вывод для рефакторинга: сохранять сдержанный high-contrast UI, избегать декоративной перегрузки, акцентировать только CTA и статусы.

### 2) Структура desktop (garage + vehicle)

Из `garage web.png`, `motocycle web.png`, `image.png`:

- стабильный левый sidebar с основными разделами;
- верхняя зона страницы: заголовок + краткое описание + primary CTA;
- блок summary KPI в отдельной строке над основным контентом;
- центральный контент разбит на независимые карточки-модули;
- внизу/ниже fold — краткий operational блок (ближайшие задачи/итоги).

Вывод для рефакторинга: desktop должен оставаться модульным (widget-based layout), где ключевые блоки можно развивать независимо без ломки общей композиции.

### 3) Структура mobile (garage + vehicle + detail)

Из `garage mobile.png`, `motocycle mobile.png`, `motocycle mobile 1.png`:

- mobile-first вертикальный сценарий;
- после header сразу primary CTA;
- summary в компактном формате (плитки/строка);
- карточки мотоциклов и статусы идут до тяжелых вторичных блоков;
- фиксированная нижняя навигация с 5 разделами;
- detail-screen узла и расходов строится как продолжение общей карточной системы.

Вывод для рефакторинга: мобильная версия должна быть основной рабочей траекторией, а не упрощенной копией desktop.

### 4) Паттерн карточки мотоцикла

На референсах карточка содержит устойчивый набор:

1. идентификация байка (название/год/пробег/профиль);
2. визуал мотоцикла (силуэт/рендер класса);
3. Garage Score + счетчики статусов;
4. "Требует внимания" или "Все в порядке";
5. быстрые действия (`Открыть`, `ТО`, `Расход`).

Вывод для рефакторинга: карточка должна быть action-first и status-first; визуал мотоцикла поддерживает контекст, но не замещает данные.

### 5) Паттерн страницы конкретного мотоцикла

На web/mobile референсах повторяется одна и та же логика:

- hero с мотоциклом и агрегированным состоянием;
- отдельный приоритетный блок "Требует внимания";
- отдельный блок "Состояние узлов";
- preview "Последние события";
- preview "Расходы";
- preview "Подбор деталей";
- явные ссылки в полные разделы.

Вывод для рефакторинга: не смешивать эти блоки в один длинный "универсальный" список; структура должна оставаться секционной и предсказуемой.

### 6) Узлы и иконографика

По референсам узлы представлены как компактные сущности:

- иконка группы + бейджи конкретных узлов;
- в mobile допустим формат сетки;
- в detail-screen узла есть фокус на конкретной задаче, деталях и истории работ.

Вывод для рефакторинга: top-node карточки должны быть короткими и не быть кликабельной поверхностью целиком. Интерактивны только явные элементы: иконка группы ведет в дерево с подсветкой `Скоро` / `Просрочено`, а бейдж конкретного узла ведет к этому узлу.

### 7) Наблюдения по расходам и аналитике

На референсах расходы показаны в формате:

- конкретный период (`Апрель 2026`);
- общая сумма;
- количество событий;
- разбивка по категориям;
- CTA для добавления расхода и перехода в детализацию.

Вывод для рефакторинга: блок расходов должен давать быстрый финансовый срез, не пытаясь заменить полный аналитический экран.

### 8) Состояния и приоритеты интерфейса

В референсах явно видны:

- healthy-state ("Все в порядке");
- attention-state (Overdue/Soon карточки);
- empty-state гаража с ценностным объяснением и CTA.

Вывод для рефакторинга: эти 3 состояния должны быть первоклассными сценариями UI, с понятным визуальным различием и отдельными текстами.

### 9) Что берем из референсов в MVP, а что нет

Берем:

- карточную модульную структуру;
- четкую иерархию "состояние -> действие -> детализация";
- dark-first + полноценный light;
- компактные preview-блоки;
- bottom nav на mobile и sidebar на desktop.

Не берем напрямую (адаптируем под текущие ограничения проекта):

- точные фотореалистичные изображения конкретных мотоциклов;
- избыточные визуальные эффекты, если они мешают читаемости;
- элементы, требующие изменения существующей бизнес-логики tree/journal.

### 10) Практические правила для следующего этапа UI-рефакторинга

1. Сначала фиксировать layout и семантику блоков, затем визуальный polish.
2. Сохранять единый порядок секций на web/mobile (различается только плотность и композиция).
3. Держать top-node overview компактным: агрегированный статус окрашивает саму иконку группы, бейджи внутри карточки показывают конкретные узлы и их статус.
4. Для каждой секции на overview оставлять короткий CTA в full-flow.
5. Проверять каждый экран на три сценария: attention / healthy / empty.

## 1. Продуктовая идея

MotoTwin должен ощущаться как **цифровой гараж владельца мотоцикла**, а не как таблица учета.

Ключевая формула:

```text
Эмоция владения + техническая аккуратность + быстрые понятные действия
```

Что пользователь должен понимать за 3-5 секунд:

1. что это за мотоцикл;
2. в каком состоянии ключевые узлы;
3. что требует внимания сейчас;
4. какое следующее действие.

---

## 2. Основные принципы

1. **Мотоцикл в центре.** Интерфейс строится вокруг конкретного байка, а не абстрактных списков.
2. **Силуэт вместо фото.** MVP использует схематичный силуэт класса, без "фото точной модели".
3. **Минимум визуального шума.** Приоритет: читаемость, статусы, иерархия, аккуратные карточки.
4. **MVP-safe.** Без 3D, тяжелых анимаций, сложной графики и лишних зависимостей.
5. **Parity web/mobile.** Смысл, порядок информации и действия должны совпадать на платформах.

---

## 3. Информационная архитектура (3 уровня)

### Уровень 1 — Гараж

Список мотоциклов пользователя с краткой диагностикой.

Показывает:

- карточки мотоциклов;
- агрегированное состояние;
- Garage Score;
- быстрые действия;
- краткое "Требует внимания".

### Уровень 2 — Страница мотоцикла (overview)

Компактный dashboard конкретного мотоцикла:

- профиль и текущее состояние;
- top-node overview;
- "Требует внимания";
- превью последних событий;
- превью расходов;
- превью подбора деталей;
- явные переходы в полные разделы.

### Уровень 3 — Детальные экраны

Отдельные экраны для глубокой работы:

- полное дерево узлов;
- страница узла;
- полный журнал обслуживания;
- расширенные расходы;
- подбор деталей.

---

## 4. Страница "Мой гараж"

Карточка мотоцикла должна включать:

1. бренд/модель/год;
2. пробег и ключевые данные;
3. компактный силуэт класса;
4. Garage Score;
5. краткие статусы по top-nodes;
6. краткий блок "Требует внимания";
7. быстрые действия (`Открыть`, `ТО`, `Расход`).

Важно:

- не показывать полное дерево на карточке;
- не превращать карточку в "мини-CRM".

---

## 5. Страница мотоцикла (единая модель)

Рекомендуемый порядок блоков:

1. Header (название, версия/год, VIN, служебные действия);
2. Hero (схематичный силуэт + контекст состояния);
3. быстрые действия;
4. блок top-node overview;
5. блок "Требует внимания";
6. блок "Последние события" (preview);
7. блок "Расходы" (preview);
8. блок "Подбор деталей" (preview);
9. ссылки/кнопки в полные разделы.

### Быстрые действия в header/hero

Desktop:

```text
[Добавить ТО] [Добавить расход] [Подобрать деталь]
```

Mobile:

```text
[ТО] [Расход] [Деталь]
```

Принцип:

- главная CTA визуально сильнее остальных;
- действия не должны ломать текущие маршруты и бизнес-логику.

---

## 6. Карта мотоцикла и top-nodes

### Что показывается на overview

На overview-странице выводятся **только top-level агрегаты**, без вложенного дерева.

Актуальный MVP-формат overview:

```text
6 агрегированных карточек:
1) Смазка                  (ENGINE.LUBE.OIL, ENGINE.LUBE.FILTER)
2) Двигатель / охлаждение  (INTAKE.FILTER, ELECTRICS.IGNITION.SPARK, COOLING.LIQUID.COOLANT)
3) Тормоза                 (BRAKES.FRONT.PADS, BRAKES.REAR.PADS, BRAKES.FLUID)
4) Шины                    (TIRES.FRONT, TIRES.REAR)
5) Цепь / звезды           (DRIVETRAIN.CHAIN, DRIVETRAIN.FRONT_SPROCKET, DRIVETRAIN.REAR_SPROCKET)
6) Подвеска                (SUSPENSION.FRONT.SEALS, SUSPENSION.FRONT.OIL)
```

Источник данных:

- агрегирование из TOP-набора сервисных узлов (TOP-15);
- пользователь видит 6 карточек, а не техническую глубину дерева.
- маппинг «TOP-узел → карточка» живёт в `packages/domain/src/top-node-overview.ts`; это UX-группировка, а не отражение `serviceGroup` в БД.

Правила содержимого:

- «Смазка» объединяет всё про моторное масло (масло + масляный фильтр) — для владельца это единый сервисный акт.
- «Двигатель / охлаждение» объединяет tune-up (воздушный фильтр + свеча) и базовый контур охлаждения (coolant).
- «Подвеска» выделена отдельной карточкой для регулярных операций по передней вилке (сальники + масло).

### Что НЕ показывать на overview

- полный список child-узлов;
- длинные иерархии;
- технические регламенты в полном объеме.

---

## 7. Полное дерево узлов (отдельный flow)

Принципы:

1. Полное дерево остается отдельным экраном/режимом.
2. На overview — только ссылка/кнопка перехода (`Все узлы →`).
3. Существующая бизнес-логика дерева не переписывается.

Требования к экрану дерева:

- видны статусы и на группах, и на дочерних узлах;
- структура согласована с темой (dark/light);
- допускается дальнейшее расширение (поиск/фильтры).

---

## 8. Журнал обслуживания (отдельный flow + preview)

### На странице мотоцикла

Показывать только preview последних событий (обычно 3-5 строк) и переход:

```text
Последние события → Все события
```

### На полной странице журнала

Используется существующий функционал:

- список событий;
- фильтры по узлу/периоду;
- фильтр "только с расходами";
- создание нового события.

Важно:

- не переносить "весь журнал" на overview;
- сохранять текущую бизнес-логику.

---

## 9. Расходы

Принцип периода: **конкретный месяц/год**, а не абстрактное "Месяц/Год".

Пример корректного периода:

```text
Апрель 2026
```

Overview показывает:

- сумму за выбранный период;
- количество затратных событий;
- краткую разбивку по категориям.

Детализация и расширенные фильтры остаются на отдельном экране.

---

## 10. Блок "Требует внимания"

Роль блока: ответ на вопрос "что сделать сейчас".

Карточка элемента:

- узел/подсистема;
- статус (`Soon`/`Overdue` и т.д.);
- краткое пояснение;
- явное действие (`Добавить ТО`, `Открыть узел`, `Подобрать деталь`).

Требования:

- блок визуально приоритетный;
- компактный формат;
- без перегруза длинными описаниями.

---

## 11. Статусы

Единый набор статусов:

```text
OK
Soon
Overdue
Recently
```

Русские подписи:

- OK = В норме
- Soon = Скоро
- Overdue = Просрочено
- Recently = Недавно обслужено

Цветовая семантика:

- OK — зеленый
- Soon — желтый
- Overdue — красный
- Recently — голубой

Правило доступности:

- нельзя полагаться только на цвет;
- текстовый статус обязателен.

---

## 12. Иконки (MotoTwin line-art)

Стиль:

- line icons;
- stroke ~1.75-2 px;
- rounded caps;
- без мультяшности и лишней детализации;
- читаемость в диапазоне 20-32 px.

Рекомендуемая логика окраски:

- базовая иконка нейтральная;
- статус — отдельным индикатором/бейджем.

Иконки overview-групп (6 шт.):

1. Смазка
2. Двигатель / охлаждение
3. Тормоза
4. Шины
5. Цепь / звезды
6. Подвеска
5. Цепь / звезды

---

## 13. Темы (dark/light)

Dark — базовая визуальная тема, light — полноценная альтернативная, не "упрощенная копия".

Общие требования:

- одинаковая смысловая семантика цветов;
- достаточный контраст текста и границ;
- одинаковая статусная логика и иконографика.

---

## 14. Navigation

Desktop: sidebar-ориентированная навигация.

Mobile: bottom navigation.

Обе платформы должны обеспечивать быстрый доступ к:

- Гаражу
- Узлам
- Журналу
- Расходам
- Профилю

---

## 15. Что нельзя делать

1. Перегружать overview полной технической иерархией.
2. Ломать уже работающие маршруты дерева и журнала.
3. Дублировать один и тот же контент в preview и full-screen без необходимости.
4. Заменять четкие данные "декоративным" UI.
5. Строить UX, который зависит только от цветовой дифференциации.

---

## 16. Единые acceptance criteria

Концепт считается соблюденным, если:

1. На странице мотоцикла есть компактный обзор и быстрые действия.
2. Top-node overview отображается агрегировано (6 карточек: Смазка, Двигатель/охлаждение, Тормоза, Шины, Цепь/звезды, Подвеска), без полного дерева на первом экране.
3. Полное дерево и полный журнал доступны отдельными явными переходами.
4. Превью "Последние события" ограничено, а детализация вынесена в full-screen.
5. Статусы и иконки единообразны на web и mobile.
6. Темы dark/light поддерживают одинаковую смысловую модель.
