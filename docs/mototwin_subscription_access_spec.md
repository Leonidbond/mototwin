# MotoTwin - разграничение функционала по подпискам Free / Rider / Pro

## 1. Назначение документа

Документ описывает продуктовую и техническую логику разграничения функционала MotoTwin по трем подпискам:

- `FREE`
- `RIDER`
- `PRO`

Документ предназначен для дальнейшей реализации в Cursor. Его нужно использовать как основу для:

- настройки paywall-логики;
- обновления Prisma-схемы;
- ограничения API;
- ограничения UI;
- изменения экранов Garage, Vehicle Detail, Nodes, Service Log и Add Service Event;
- добавления страницы подписок;
- подготовки тестовых сценариев.

Главный принцип: ограничения подписки должны быть реализованы не только в интерфейсе, но и на уровне серверной бизнес-логики. UI может скрывать или блокировать действия, но источник истины должен быть на сервере.

---

## 2. Контекст MotoTwin

MotoTwin - это цифровой гараж для владельца мотоцикла. Основные продуктовые блоки:

- гараж пользователя;
- профиль мотоцикла;
- дерево узлов;
- журнал обслуживания;
- напоминания;
- подбор совместимых деталей;
- учет расходов;
- future fitment/community logic.

Подписки должны усиливать эту модель, а не превращать продукт в набор случайных платных кнопок. Ограничения должны быть понятными пользователю и логически связанными с ценностью продукта:

- больше мотоциклов;
- больше глубина узлов;
- больше история обслуживания;
- более детальная запись ТО.

## 3. Тарифы и базовые ограничения

| Возможность | Free | Rider | Pro |
| --- | --- | --- | --- |
| Количество мотоциклов | 1 | до 3 | без ограничения |
| Доступ к узлам | только топ-узлы | только топ-узлы | все узлы |
| Выбор топ-узла | нельзя | можно | можно |
| Выбор дочерних узлов | нельзя | нельзя | можно |
| Сервисные события | последние 10 активных событий | без ограничения | без ограничения |
| Режим добавления ТО | только быстрый | быстрый и подробный | быстрый и подробный |
| Глубина сервисной истории | ограниченная | полная | полная |
| Детализация ТО по подузлам | нет | нет | да |
| Подходит для | попробовать сервис | обычный владелец 1-3 мотоциклов | power user, коллекционер, сервис, продвинутый владелец |

## 4. Продуктовая логика тарифов

### 4.1. Free

Free должен дать пользователю почувствовать базовую ценность MotoTwin, но не открывать полную эксплуатационную глубину.

Free нужен для:

- быстрого входа в продукт;
- добавления одного мотоцикла;
- понимания концепции цифрового гаража;
- базовой фиксации обслуживания;
- демонстрации статусов топ-узлов;
- формирования желания перейти на Rider или Pro.

Ограничения Free:

1. Пользователь может добавить только 1 мотоцикл.
2. Пользователь видит только топ-узлы.
3. Топ-узлы отображаются как read-only карточки статуса.
4. Пользователь не может вручную выбирать топ-узел при добавлении ТО.
5. Новое ТО добавляется только в быстром режиме.
6. В журнале обслуживания отображаются только последние 10 активных сервисных событий.
7. При добавлении 11-го события срабатывает ротация: новое событие становится активным, самое старое активное событие уходит в скрытую историю Free.
8. Старые события не удаляются из базы, но не показываются пользователю Free в основном журнале.
9. При попытке открыть скрытую историю, выбрать узел или включить подробный режим показывается paywall.

Важно: Free не должен удалять пользовательские данные. Даже если в интерфейсе показываются только последние 10 событий, старые события нужно сохранять в базе. Это снижает риск потери доверия к продукту.

### 4.2. Rider

Rider - основной платный тариф для обычного владельца мотоцикла.

Rider нужен для:

- ведения полноценной истории обслуживания;
- использования нескольких мотоциклов;
- выбора топ-узлов;
- нормального сценария обслуживания без глубокой детализации по всем подузлам.

Возможности Rider:

1. Пользователь может добавить до 3 мотоциклов.
2. Пользователь видит только топ-узлы.
3. Пользователь может выбирать топ-узел в сервисном событии.
4. Пользователь может открывать карточку топ-узла.
5. Пользователь может вести неограниченный журнал обслуживания.
6. Пользователь может использовать быстрый и подробный режим добавления ТО.
7. Подробный режим в Rider работает на уровне топ-узлов, без выбора дочерних узлов.
8. Дочерние узлы в дереве не показываются или показываются как заблокированные Pro-функции.

Rider должен восприниматься как нормальный тариф для владельца 1-3 мотоциклов, которому не нужна профессиональная детализация по всем компонентам.

### 4.3. Pro

Pro - максимальный тариф для продвинутых владельцев, нескольких мотоциклов, коллекционеров, механиков и power users.

Возможности Pro:

1. Пользователь может добавить неограниченное количество мотоциклов.
2. Пользователь видит все дерево узлов.
3. Пользователь может выбирать топ-узлы и дочерние узлы.
4. Пользователь может вести неограниченный журнал обслуживания.
5. Пользователь может использовать быстрый и подробный режим добавления ТО.
6. Подробный режим поддерживает точную привязку к конкретному узлу или подузлу.
7. Pro открывает максимальную детализацию сервисной истории и подготовки будущей fitment/community-логики.

Pro должен восприниматься как тариф для тех, кто хочет вести мотоцикл максимально точно.

## 5. Термины и определения для реализации

### 5.1. Top node

Top node - верхнеуровневый узел дерева мотоцикла.

Примеры:

- Engine / Cooling
- Brakes
- Tires
- Chain / Sprockets
- Suspension
- Electrics

Для реализации у узла должно быть поле, позволяющее отличить уровень:

```ts
node.level === 1
```

или

```ts
node.parentId === null
```

### 5.2. Child node

Child node - вложенный узел ниже топ-узла.

Примеры:

- ENGINE.LUBE.OIL
- ENGINE.LUBE.FILTER
- BRAKES.FRONT.PADS
- BRAKES.REAR.PADS
- DRIVETRAIN.CHAIN
- SUSPENSION.FRONT.SEALS

Для Pro доступны все уровни дерева.

### 5.3. Быстрый режим ТО

Быстрый режим - минимальная форма добавления обслуживания.

Назначение: быстро зафиксировать факт обслуживания без сложной детализации.

Базовые поля:

- дата;
- пробег;
- моточасы, если применимо;
- тип обслуживания или preset;
- стоимость, опционально;
- комментарий, опционально.

В Free быстрый режим не должен требовать выбора узла. Система может автоматически сопоставить preset с топ-узлом, если это возможно.

Пример:

- пользователь выбирает `Замена масла`;
- система автоматически связывает событие с `Engine / Cooling` или `Engine`;
- пользователь не видит ручной выбор узла.

### 5.4. Подробный режим ТО

Подробный режим - расширенная форма добавления обслуживания.

Назначение: точно описать выполненные работы, детали, расходы и привязку к узлам.

Поля:

- дата;
- пробег;
- моточасы;
- выбранный узел;
- тип работ;
- установленные детали;
- стоимость деталей;
- стоимость работ;
- исполнитель;
- комментарий;
- вложения, если уже реализованы;
- связь с расходами;
- связь с fitment-report в будущем.

В Rider подробный режим доступен только с выбором топ-узлов.

В Pro подробный режим доступен с выбором любого узла дерева.

### 5.5. Ротация сервисных событий в Free

Ротация означает, что в Free активными и видимыми остаются только последние 10 сервисных событий по мотоциклу.

Правило:

- при создании 1-10 событий все они видимы;
- при создании 11-го события новое событие сохраняется и показывается;
- самое старое активное событие становится скрытым для Free;
- скрытое событие не удаляется из базы;
- при переходе на Rider или Pro вся история снова становится доступна.

Для пользователя Free нужно показывать сообщение:

> В Free отображаются последние 10 сервисных событий. Старые записи сохранены и будут доступны после перехода на Rider или Pro.

## 6. Матрица ограничений по модулям

### 6.1. Garage

#### Free

- Можно создать только 1 мотоцикл.
- Кнопка `Добавить мотоцикл` после создания первого мотоцикла остается видимой, но ведет на paywall.
- На карточке мотоцикла показывается бейдж `Free`.
- Если пользователь пытается добавить второй мотоцикл, показывается экран апгрейда до Rider или Pro.

#### Rider

- Можно создать до 3 мотоциклов.
- При попытке добавить 4-й мотоцикл показывается предложение перейти на Pro.
- На карточках мотоциклов показывается обычная сводка без ограничений истории.

#### Pro

- Количество мотоциклов не ограничено.
- Ограничение может быть только техническим через abuse protection, но не продуктовым.

### 6.2. Vehicle Detail

#### Free

- Показывает базовый профиль мотоцикла.
- Показывает топ-узлы в read-only режиме.
- Не дает открыть глубокую карточку узла.
- Не дает выбрать узел для ручной детализации.

#### Rider

- Показывает базовый профиль мотоцикла.
- Показывает топ-узлы как активные элементы.
- Дает открыть топ-узел.
- Не показывает дочерние узлы или показывает их как Pro-locked.

#### Pro

- Показывает полный профиль мотоцикла.
- Показывает полное дерево узлов.
- Дает открыть любой узел.

### 6.3. Nodes Overview

#### Free

- Видны только топ-узлы.
- Карточки топ-узлов не являются полноценными селекторами.
- Клик по узлу показывает paywall или короткую read-only подсказку.
- Статусы могут отображаться, но без детальной истории узла.

#### Rider

- Видны только топ-узлы.
- Топ-узлы кликабельны.
- Можно фильтровать сервисную историю по топ-узлу.
- Дочернее дерево недоступно.

#### Pro

- Видно полное дерево.
- Можно раскрывать уровни.
- Можно фильтровать историю по любому узлу.
- Можно добавлять событие на конкретный дочерний узел.

### 6.4. Service Log

#### Free

- Показывает последние 10 активных событий.
- Старые события скрыты.
- Вверху журнала показывается лимит:

> Free: последние 10 сервисных событий

- Если есть скрытые события, показывается уведомление:

> Еще N событий сохранены в истории, но недоступны в Free.

- Добавление события доступно только через быстрый режим.

#### Rider

- Показывает все события.
- Доступны фильтры по топ-узлам.
- Доступен быстрый и подробный режим.
- В подробном режиме можно выбирать только топ-узлы.

#### Pro

- Показывает все события.
- Доступны фильтры по всем узлам.
- Доступен быстрый и подробный режим.
- В подробном режиме можно выбирать любой узел.

### 6.5. Add Service Event Modal

#### Free

- Отображается только вкладка `Быстро`.
- Вкладка `Подробно` скрыта или отображается заблокированной.
- Поле выбора узла не показывается.
- Пользователь выбирает preset типа работ.
- Система автоматически определяет топ-узел, если preset связан с узлом.
- Если preset не связан с узлом, событие сохраняется без ручного выбора узла или с nodeId = null.

#### Rider

- Доступны вкладки `Быстро` и `Подробно`.
- В быстром режиме можно выбрать top node, если это нужно.
- В подробном режиме выбор узла ограничен топ-узлами.
- Дочерние узлы не отображаются.

#### Pro

- Доступны вкладки `Быстро` и `Подробно`.
- В быстром режиме можно выбрать top node или preset.
- В подробном режиме доступен весь node tree.
- Можно точно привязать обслуживание к дочернему узлу.

### 6.6. Expenses

В текущей версии расходы не указаны как отдельный ограничиваемый блок, поэтому их лучше не ограничивать жестко на первом этапе.

Рекомендация:

- Free видит расходы, связанные с видимыми сервисными событиями.
- Rider видит все расходы по всем своим мотоциклам.
- Pro видит все расходы и в будущем может получить расширенную аналитику по дочерним узлам.

Не нужно сейчас вводить отдельный лимит расходов, чтобы не усложнять подписку.

### 6.7. Fitment / Parts

Для текущего этапа подписки можно не вводить жесткое ограничение fitment, если оно еще не реализовано полностью.

Рекомендация на будущее:

- Free: базовый просмотр рекомендованных деталей без расширенных фильтров.
- Rider: рекомендации по топ-узлам и базовый fitment.
- Pro: полный fitment по дочерним узлам, расширенные фильтры, confidence details.

На текущем этапе основными ограничителями остаются:

- количество мотоциклов;
- доступ к узлам;
- глубина сервисной истории;
- режим добавления ТО.

## 7. Paywall-логика

### 7.1. Что такое paywall в MotoTwin

Paywall - это не просто запрет. Это объяснение, почему действие недоступно и какой тариф открывает возможность.

Каждый paywall должен отвечать на 3 вопроса:

1. Что пользователь хотел сделать?
2. Почему это недоступно на текущем тарифе?
3. Какой тариф открывает эту возможность?

### 7.2. Paywall-сценарии

| Сценарий | Текущий тариф | Что показать |
| --- | --- | --- |
| Добавление 2-го мотоцикла | Free | Перейдите на Rider для 3 мотоциклов или Pro для неограниченного гаража |
| Добавление 4-го мотоцикла | Rider | Перейдите на Pro для неограниченного гаража |
| Выбор топ-узла | Free | Выбор узлов доступен в Rider и Pro |
| Открытие топ-узла | Free | Детальная работа с узлами доступна в Rider и Pro |
| Открытие дочернего узла | Free/Rider | Полное дерево узлов доступно в Pro |
| Переключение на подробное ТО | Free | Подробный режим доступен в Rider и Pro |
| Просмотр старых событий после 10 | Free | Полная история обслуживания доступна в Rider и Pro |
| Фильтр по дочернему узлу | Rider | Детализация по всем узлам доступна в Pro |

### 7.3. Тексты paywall

#### Для Free при добавлении второго мотоцикла

Заголовок:

> Добавление нескольких мотоциклов доступно в Rider

Текст:

> В Free можно вести один мотоцикл. Rider открывает до 3 мотоциклов, а Pro - неограниченный гараж.

CTA:

- `Перейти на Rider`
- `Сравнить тарифы`
- `Остаться на Free`

#### Для Rider при добавлении четвертого мотоцикла

Заголовок:

> Неограниченный гараж доступен в Pro

Текст:

> Rider подходит для владельца до 3 мотоциклов. Pro открывает неограниченное количество мотоциклов и полное дерево узлов.

CTA:

- `Перейти на Pro`
- `Сравнить тарифы`

#### Для Free при попытке выбрать узел

Заголовок:

> Выбор узлов доступен в Rider

Текст:

> В Free топ-узлы показываются только как обзор состояния. В Rider можно выбирать топ-узлы и вести историю обслуживания по ним.

CTA:

- `Открыть Rider`
- `Сравнить тарифы`

#### Для Rider при попытке открыть дочерний узел

Заголовок:

> Полное дерево узлов доступно в Pro

Текст:

> Rider работает с топ-узлами. Pro позволяет вести обслуживание точно по деталям: колодки, фильтры, цепь, звезды, сальники вилки и другие подузлы.

CTA:

- `Перейти на Pro`
- `Сравнить тарифы`

#### Для Free при попытке открыть подробный режим ТО

Заголовок:

> Подробное ТО доступно в Rider

Текст:

> В Free доступна быстрая запись обслуживания. Rider и Pro открывают подробное ТО с выбором узла, деталями работ и расходами.

CTA:

- `Перейти на Rider`
- `Сравнить тарифы`

## 8. Правила серверной проверки

Все ограничения должны проверяться на сервере.

Нельзя полагаться только на:

- скрытые кнопки;
- disabled-состояния;
- клиентские проверки;
- фронтенд-конфиг.

### 8.1. Центральная функция capabilities

Нужно создать единый источник логики тарифов.

Рекомендуемый файл:

```text
src/lib/subscription/capabilities.ts
```

Пример структуры:

```ts
export type SubscriptionPlan = "FREE" | "RIDER" | "PRO";

export type NodeAccessLevel = "TOP_READ_ONLY" | "TOP_SELECTABLE" | "FULL_TREE";

export type ServiceEventMode = "QUICK" | "DETAILED";

export type SubscriptionCapabilities = {
  maxVehicles: number | null;
  nodeAccessLevel: NodeAccessLevel;
  canSelectTopNode: boolean;
  canSelectChildNode: boolean;
  maxActiveServiceEventsPerVehicle: number | null;
  allowedServiceEventModes: ServiceEventMode[];
};

export const SUBSCRIPTION_CAPABILITIES: Record<SubscriptionPlan, SubscriptionCapabilities> = {
  FREE: {
    maxVehicles: 1,
    nodeAccessLevel: "TOP_READ_ONLY",
    canSelectTopNode: false,
    canSelectChildNode: false,
    maxActiveServiceEventsPerVehicle: 10,
    allowedServiceEventModes: ["QUICK"],
  },
  RIDER: {
    maxVehicles: 3,
    nodeAccessLevel: "TOP_SELECTABLE",
    canSelectTopNode: true,
    canSelectChildNode: false,
    maxActiveServiceEventsPerVehicle: null,
    allowedServiceEventModes: ["QUICK", "DETAILED"],
  },
  PRO: {
    maxVehicles: null,
    nodeAccessLevel: "FULL_TREE",
    canSelectTopNode: true,
    canSelectChildNode: true,
    maxActiveServiceEventsPerVehicle: null,
    allowedServiceEventModes: ["QUICK", "DETAILED"],
  },
};
```

### 8.2. Проверка количества мотоциклов

При создании мотоцикла API должен:

1. определить пользователя;
2. получить текущий тариф;
3. получить количество существующих мотоциклов;
4. сравнить с `maxVehicles`;
5. если лимит достигнут, вернуть `403` и код ошибки.

Пример ошибки:

```json
{
  "error": "VEHICLE_LIMIT_REACHED",
  "requiredPlan": "RIDER",
  "message": "Your current plan allows only 1 motorcycle. Upgrade to Rider to add up to 3 motorcycles."
}
```

### 8.3. Проверка режима сервисного события

При создании сервисного события API должен проверять поле:

```ts
mode: "QUICK" | "DETAILED"
```

Если пользователь Free отправляет `DETAILED`, сервер должен вернуть `403`.

### 8.4. Проверка выбора узла

Если пользователь Free передает `nodeId`, сервер должен проверить:

- является ли это ручным выбором;
- разрешен ли выбор узла на тарифе;
- какой уровень узла выбран.

Правила:

- Free: ручной `nodeId` запрещен.
- Rider: разрешен только `nodeId` топ-узла.
- Pro: разрешен любой `nodeId`.

### 8.5. Ротация сервисных событий

При создании сервисного события для Free:

1. событие сохраняется;
2. система считает активные события по vehicleId;
3. если активных больше 10, самое старое активное событие помечается как rotated;
4. rotated-событие не показывается в Free-журнале;
5. при апгрейде до Rider или Pro rotated-события снова показываются.

Рекомендуемые поля:

```ts
rotatedOutAt?: Date | null;
rotatedOutReason?: "FREE_LIMIT" | null;
```

Альтернатива: не сохранять `rotatedOutAt`, а вычислять видимость через запрос последних 10 событий. Но для прозрачности и будущей диагностики лучше хранить признак ротации.

## 9. Рекомендуемые изменения в Prisma

Не нужно делать сложную billing-систему в первом шаге. Для MVP достаточно хранить текущий тариф пользователя.

### 9.1. Enums

```prisma
enum SubscriptionPlan {
  FREE
  RIDER
  PRO
}

enum SubscriptionStatus {
  ACTIVE
  TRIALING
  PAST_DUE
  CANCELED
}

enum ServiceEventMode {
  QUICK
  DETAILED
}

enum ServiceEventRotationReason {
  FREE_LIMIT
}
```

### 9.2. UserSubscription

```prisma
model UserSubscription {
  id                 String             @id @default(cuid())
  userId             String             @unique
  plan               SubscriptionPlan   @default(FREE)
  status             SubscriptionStatus @default(ACTIVE)
  currentPeriodStart DateTime?
  currentPeriodEnd   DateTime?
  createdAt          DateTime           @default(now())
  updatedAt          DateTime           @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

### 9.3. ServiceEvent

Добавить поля:

```prisma
mode                  ServiceEventMode             @default(QUICK)
createdUnderPlan      SubscriptionPlan             @default(FREE)
rotatedOutAt          DateTime?
rotatedOutReason      ServiceEventRotationReason?
```

Также важно, чтобы `nodeId` мог быть nullable, потому что Free быстрый режим может создавать событие без ручного выбора узла.

```prisma
nodeId String?
```

### 9.4. Node

Если в текущей схеме еще нет уровня дерева, добавить одно из двух:

Вариант 1:

```prisma
level Int @default(1)
```

Вариант 2:

```prisma
parentId String?
```

Лучше использовать `parentId`, если дерево узлов уже проектируется как иерархия. Уровень можно вычислять или хранить дополнительно для быстрых запросов.

## 10. API-требования

### 10.1. GET /api/subscription/current

Возвращает текущую подписку и capabilities.

Ответ:

```json
{
  "plan": "FREE",
  "status": "ACTIVE",
  "capabilities": {
    "maxVehicles": 1,
    "nodeAccessLevel": "TOP_READ_ONLY",
    "canSelectTopNode": false,
    "canSelectChildNode": false,
    "maxActiveServiceEventsPerVehicle": 10,
    "allowedServiceEventModes": ["QUICK"]
  }
}
```

### 10.2. POST /api/vehicles

Должен проверять лимит мотоциклов.

Ошибки:

- `VEHICLE_LIMIT_REACHED`
- `SUBSCRIPTION_REQUIRED`

### 10.3. GET /api/vehicles/[vehicleId]/nodes

Должен учитывать тариф:

- Free: вернуть топ-узлы с `selectable: false`;
- Rider: вернуть топ-узлы с `selectable: true`;
- Pro: вернуть все дерево с `selectable: true`.

Пример элемента:

```json
{
  "id": "node_brakes",
  "name": "Тормоза",
  "level": 1,
  "parentId": null,
  "status": "SOON",
  "selectable": true,
  "locked": false
}
```

Для locked-узлов:

```json
{
  "id": "node_brakes_front_pads",
  "name": "Передние колодки",
  "level": 2,
  "parentId": "node_brakes",
  "selectable": false,
  "locked": true,
  "requiredPlan": "PRO"
}
```

### 10.4. POST /api/service-events

Должен проверять:

- режим события;
- доступность выбора узла;
- уровень выбранного узла;
- лимит и ротацию для Free.

Ошибки:

- `SERVICE_EVENT_MODE_NOT_ALLOWED`
- `NODE_SELECTION_NOT_ALLOWED`
- `CHILD_NODE_REQUIRES_PRO`

### 10.5. GET /api/vehicles/[vehicleId]/service-events

Должен учитывать тариф:

- Free: вернуть только последние 10 активных событий;
- Rider: вернуть все события;
- Pro: вернуть все события.

Для Free ответ должен содержать summary по скрытым событиям:

```json
{
  "items": [],
  "meta": {
    "visibleLimit": 10,
    "hiddenCount": 4,
    "plan": "FREE"
  }
}
```

## 11. UI-требования

### 11.1. Общие правила UI

1. Нельзя просто скрывать платные функции полностью.
2. Пользователь должен понимать, что возможность существует.
3. Заблокированные возможности должны быть объяснены.
4. Paywall должен быть аккуратным и не раздражающим.
5. Важно показывать, какой тариф нужен.

### 11.2. Бейдж тарифа

В интерфейсе пользователя нужно показывать текущий тариф:

- в профиле;
- на странице подписки;
- возможно в sidebar/header.

Примеры:

- `Free`
- `Rider`
- `Pro`

### 11.3. Pricing / Subscription page

Нужно создать страницу сравнения тарифов.

Путь:

```text
/subscription
```

или

```text
/settings/subscription
```

Содержание:

- текущий тариф;
- таблица Free / Rider / Pro;
- ключевые отличия;
- CTA для апгрейда;
- пояснение, что старые сервисные события в Free не удаляются.

### 11.4. Locked state

Для заблокированных действий использовать единый компонент.

Рекомендуемый компонент:

```text
SubscriptionLock
```

Поля:

```ts
type SubscriptionLockProps = {
  title: string;
  description: string;
  requiredPlan: "RIDER" | "PRO";
  actionLabel?: string;
};
```

### 11.5. Feature guard

На клиенте можно использовать компонент:

```text
FeatureGate
```

Но он не заменяет серверную проверку.

Пример:

```tsx
<FeatureGate requiredPlan="RIDER" feature="DETAILED_SERVICE_EVENT">
  <DetailedServiceEventTab />
</FeatureGate>
```

## 13. Тестовые сценарии

### 13.1. Free

1. Новый пользователь получает тариф Free.
2. Пользователь может создать первый мотоцикл.
3. Пользователь не может создать второй мотоцикл.
4. Пользователь видит топ-узлы.
5. Пользователь не может выбрать топ-узел вручную.
6. Пользователь не видит дочерние узлы.
7. Пользователь может добавить сервисное событие в быстром режиме.
8. Пользователь не может открыть подробный режим.
9. После 10 событий пользователь может добавить 11-е.
10. После добавления 11-го события в журнале видны только 10 последних.
11. Старое событие не удалено из базы.

### 13.2. Rider

1. Пользователь Rider может создать 3 мотоцикла.
2. Пользователь Rider не может создать 4-й мотоцикл.
3. Пользователь видит топ-узлы.
4. Пользователь может выбрать топ-узел.
5. Пользователь не может выбрать дочерний узел.
6. Пользователь может использовать быстрый режим.
7. Пользователь может использовать подробный режим.
8. В подробном режиме доступны только топ-узлы.
9. Журнал обслуживания не ограничен 10 событиями.

### 13.3. Pro

1. Пользователь Pro может создавать больше 3 мотоциклов.
2. Пользователь видит все дерево узлов.
3. Пользователь может выбрать любой узел.
4. Пользователь может использовать быстрый режим.
5. Пользователь может использовать подробный режим.
6. Журнал обслуживания не ограничен.

## 14. Acceptance Criteria

Функционал разграничения подписок считается реализованным, если:

1. У пользователя есть текущий тариф.
2. Новый пользователь по умолчанию получает Free.
3. Все лимиты описаны в едином capabilities-конфиге.
4. UI показывает текущий тариф.
5. UI объясняет заблокированные действия через paywall.
6. API не позволяет обойти лимиты через прямой запрос.
7. Free ограничен 1 мотоциклом.
8. Rider ограничен 3 мотоциклами.
9. Pro не имеет лимита мотоциклов.
10. Free видит только топ-узлы read-only.
11. Rider может выбирать топ-узлы.
12. Pro может выбирать все узлы.
13. Free может создавать сервисные события только в quick mode.
14. Rider и Pro могут создавать quick и detailed события.
15. Free видит только последние 10 активных сервисных событий.
16. Старые Free-события сохраняются в базе и доступны после апгрейда.
17. Страница подписок показывает понятное сравнение Free / Rider / Pro.
18. Реализация не добавляет лишнюю billing-архитектуру до подключения реальной оплаты.

## 15. Что не делать на первом этапе

Не нужно в первом этапе:

- подключать реальный платежный шлюз;
- строить сложную биллинговую систему;
- делать купоны, промокоды, инвойсы;
- делать годовые и месячные тарифы, если это не требуется прямо сейчас;
- ограничивать каждую мелкую функцию;
- усложнять fitment-ограничения до готовности fitment-модуля;
- удалять старые сервисные события Free-пользователя;
- делать отдельную схему узлов под каждый тариф.

На первом этапе нужно реализовать чистую продуктовую основу:

- тариф пользователя;
- capabilities;
- ограничения гаража;
- ограничения узлов;
- ограничения service log;
- ограничения Add Service Event;
- понятный paywall.
