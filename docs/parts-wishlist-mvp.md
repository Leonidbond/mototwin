# Parts Wishlist MVP («Что нужно купить»)

## Scope

Практичный **операционный** список запчастей и расходников для конкретного мотоцикла: что нужно докупить, что заказано и что куплено. Статус **«Установлено»** в продукте ведёт к фиксации факта в **журнале обслуживания** (после сохранения сервисного события), а не к отдельному архиву внутри wishlist. Это **не** каталог, **не** маркетплейс и **не** подбор по SKU.

Один мотоцикл — много позиций. Позиция может быть **привязана к узлу** дерева (необязательно).

## Data model (Prisma)

- **Enum `PartWishlistItemStatus`:** `NEEDED`, `ORDERED`, `BOUGHT`, `INSTALLED`.
- **Model `PartWishlistItem`:**
  - `id`, `vehicleId`, опционально `nodeId`
  - `title`, `quantity` (по умолчанию 1), `status`, опционально `comment`
  - опционально **`costAmount`** (число, как у `ServiceEvent.costAmount`) и **`currency`** (строка, как у `ServiceEvent.currency`); оба поля необязательны, для существующих строк после миграции остаются `NULL`
  - `createdAt`, `updatedAt`
- Связи: позиция принадлежит `Vehicle`, опционально — `Node`.
- Индексы: `vehicleId`, `nodeId`, `status`.

## API (JSON, без авторизации)

Базовый префикс: `/api/vehicles/[id]/wishlist`.

| Метод | Путь | Назначение |
|--------|------|------------|
| `GET` | `.../wishlist` | Список позиций для мотоцикла |
| `POST` | `.../wishlist` | Создание |
| `PATCH` | `.../wishlist/[itemId]` | Частичное обновление |
| `DELETE` | `.../wishlist/[itemId]` | Удаление |

Валидация (сервер): обязательный `title`; `quantity` — целое ≥ 1, если передано; `status` — допустимый enum; `nodeId` — если передан, узел должен существовать; мотоцикл должен существовать. **`costAmount`** (если передан) — число **≥ 0**; **`currency`** — опционально; если задана стоимость без валюты, на сервере подставляется **`RUB`**. Если стоимости нет, валюта в БД не хранится (`NULL`). Ответы с явным полем `error` при 4xx.

**Wishlist в UI — только активный список:** в блоке «Что нужно купить» на web и в списке wishlist в Expo показываются только позиции со статусами **`NEEDED`**, **`ORDERED`**, **`BOUGHT`**. Позиции **`INSTALLED`** **не удаляются** из БД и по-прежнему приходят в **`GET .../wishlist`**, но **скрыты** из активного списка. **Отдельной вкладки, экрана или архива «Установленные» внутри wishlist нет** — история установок смотрится в **журнале обслуживания** после того, как пользователь сохранил соответствующее сервисное событие. Пустое состояние считается по активным позициям: если все строки только `INSTALLED`, показывается сообщение вроде «Список покупок пуст» / «Все позиции установлены» и краткая подсказка про журнал.

### Статус INSTALLED и журнал обслуживания

**Сервер не создаёт `ServiceEvent` при смене статуса на `INSTALLED`.** `PATCH .../wishlist/[itemId]` только обновляет строку списка покупок; ответ **`{ item }`** — обновлённая позиция, без полей про автосоздание события.

После **успешного** перевода в **`INSTALLED`** клиент (web / Expo) ведёт пользователя к **уже существующему** потоку «Добавить сервисное событие», с предзаполнением из позиции и текущего состояния мотоцикла:

- **Есть `nodeId`:** открывается форма добавления события; узел предвыбран; дата события — сегодня; пробег и моточасы — текущие значения ТС; тип работ — **«Установка запчасти»**; комментарий собирается из названия позиции, количества (если есть) и комментария из wishlist (если есть); при поддержке поля установленных запчастей — JSON из названия и количества; **если в позиции указаны стоимость и валюта — они переносятся в поля «Стоимость» / «Валюта» формы события** (по умолчанию валюта новых строк wishlist в клиентах — **`RUB`**). Пользователь может всё отредактировать и **явно сохранить** — только тогда создаётся запись в журнале (`POST .../service-events`). **Расходы и сводка по тратам по-прежнему считаются только по сохранённым `ServiceEvent`**, не по строкам wishlist напрямую.
- **Нет `nodeId`:** форма не открывается автоматически; показывается краткая ненавязчивая подсказка: позиция не привязана к узлу, сервисное событие не открыто автоматически. Статус **`INSTALLED`** всё равно сохраняется.

## Shared packages

- **`@mototwin/types`:** `PartWishlistItem` (в т.ч. `costAmount`, `currency`), `PartWishlistItemViewModel` (опционально `costLabelRu` для списка), `PartWishlistItemStatus`, `PartWishlistFormValues`, константа **`PART_WISHLIST_DEFAULT_CURRENCY`** (`RUB`), DTO создания/обновления, тип ответа **`UpdateWishlistItemResponse`** — **`{ item }`**.
- **`@mototwin/domain`:** подписи статусов на русском, начальные значения формы, валидация, нормализация payload (пустая стоимость → `null`, согласование валюты с бэкендом через `normalizePartWishlistCostMutationArgs`), `buildPartWishlistItemViewModel`, `groupPartWishlistItemsByStatus`, фильтр активного списка (`isActiveWishlistItem`, `filterActiveWishlistItems`), порядок статусов, `isWishlistTransitionToInstalled`, константа типа работ **«Установка запчасти»** (`WISHLIST_INSTALL_SERVICE_TYPE_RU`), префикс первой строки комментария для распознавания в журнале (`WISHLIST_INSTALL_SERVICE_COMMENT_PREFIX_RU`), эвристика **`isLikelyWishlistInstallServiceEvent`** (тип работ + префикс комментария, без нового `eventKind`), хелпер предзаполнения формы события после установки (`createInitialAddServiceEventFromWishlistItem`, в т.ч. стоимость/валюта из wishlist), текст подсказки без узла (`WISHLIST_INSTALLED_NO_NODE_SERVICE_HINT`). В **view model** строки журнала поле **`wishlistOriginLabelRu`** дублирует подсказку «Из списка покупок» для таких событий.
- **`@mototwin/api-client`:** `getVehicleWishlist`, `createWishlistItem`, `updateWishlistItem`, `deleteWishlistItem`.

## Web (`src/app/vehicles/[id]/page.tsx`)

- Секция «Что нужно купить»: группировка по статусу только для активных позиций (`INSTALLED` скрыты); поля карточки (название, узел, количество, **стоимость при наличии**, статус, комментарий); краткий текст, что установленные позиции попадают в **журнал обслуживания** после сохранения события (без вкладки «Установленные»).
- Создание/редактирование в модальном окне (в т.ч. **стоимость и валюта**, по умолчанию валюта **`RUB`**); смена статуса и удаление.
- После перевода в **«Установлено»**: при **`nodeId`** открывается модальное окно добавления сервисного события с предзаполнением; без узла — подсказка, без автосоздания события; обновляется список покупок; журнал и дерево обновляются после **сохранения** события пользователем (и при других действиях на странице, как и раньше).
- Из дерева узлов и блока внимания — быстрое добавление с предвыбранным `nodeId` (где уже сделано в продукте).

## Expo (`apps/app/app/vehicles/[id]/wishlist/**`)

- Экран списка: `/vehicles/[id]/wishlist` — группы по статусу только для активных позиций (`INSTALLED` скрыты), **строка стоимости при наличии**, подсказка про журнал обслуживания как место истории установок (без отдельного списка установленных), переход к редактированию по нажатию на карточку, смена статуса через системный `Alert`, удаление с подтверждением, кнопка «Добавить позицию».
- После установки статуса **Установлено** с **`nodeId`** — переход на экран добавления сервисного события с предзаполнением; без узла — `Alert` с подсказкой; список перезагружается; после сохранения события — возврат к списку покупок; при фокусе карточки мотоцикла дерево и журнал подтягиваются как раньше.
- Создание: `/vehicles/[id]/wishlist/new` (query `nodeId` — предзаполнение узла).
- Редактирование: `/vehicles/[id]/wishlist/[itemId]`.
- Форма создания/редактирования wishlist в Expo keyboard-aware: контент прокручивается при открытой клавиатуре (`KeyboardAvoidingView` + `ScrollView`), кнопка сохранения остаётся достижимой на iOS/Android.
- С карточки мотоцикла — блок-ссылка «Что нужно купить»; в дереве узлов — «В список»; на экране «Требует внимания» — «В список покупок» с передачей `nodeId`.

## Parity

Один и тот же backend и те же клиентские контракты. На обеих платформах wishlist остаётся **только активным списком**; **история установок** — через **журнал обслуживания**. Доступны: просмотр, создание, обновление статуса, удаление, опциональная привязка к узлу, **опциональная стоимость и валюта (по умолчанию `RUB`)**, перенос стоимости в форму сервисного события при **`INSTALLED`** с **`nodeId`**, и быстрый вход с предвыбранным узлом.

## Отложено (намеренно)

- Полноценный каталог запчастей и нормализация номенклатуры.
- Сопоставление SKU и производителей.
- Ссылки на маркетплейсы и цены.
- Правила совместимости (модель/год/рынок).
