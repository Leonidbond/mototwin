# Parts Wishlist MVP (подбор деталей / список покупок)

## Scope

Практичный **операционный** список запчастей и расходников для конкретного мотоцикла: что нужно докупить, что заказано и что куплено. Статус **«Установлено»** в продукте ведёт к фиксации факта в **журнале обслуживания** (после сохранения сервисного события), а не к отдельному архиву внутри wishlist. Это **не** маркетплейс и **не** полный fitment-движок; опционально позиция может быть связана с **каталожным SKU** через ручной поиск (`getPartSkus`) или через **консервативные node-based рекомендации** (`getRecommendedSkusForNode`).

Один мотоцикл — много позиций. Каждая позиция **обязательно** привязана к **конечному (leaf) узлу** дерева мотоцикла.

Также в MVP поддерживаются **комплекты обслуживания** (service kits): один action создаёт сразу несколько wishlist-строк. Это только batch-добавление в wishlist.

**Установка из списка покупок в одно сервисное событие (multi-install, волна 3.2):** в `installedPartsJson` сохраняется массив ссылок `{ source: "wishlist", wishlistItemId, ... }` (старый формат с одним объектом по-прежнему читается). После сохранения все отмеченные строки переводятся в `INSTALLED`; в журнале для нескольких id показываются отдельные ссылки в parts.

**Блок «Готово к установке» (web + Expo):** вместо двух отдельных блоков («Список покупок» и «Купленные, но не установленные детали») используется один список, данные для которого приходят с **`GET /api/vehicles/:vehicleId/installable`**: активный wishlist (`NEEDED` / `ORDERED` / `BOUGHT`) + uninstalled `ExpenseItem`, с дедупликацией по `expense.shoppingListItemId == wishlist.id` (`source: "wishlist+expense"`). Чипы фильтра: **Все / Куплено / В списке покупок**. Переключение чекбокса строки обновляет форму: wishlist — через `mergeWishlistItemIntoAddFormValues` / `removeWishlistItemFromAddFormValues` (для пары wishlist+expense **`skipPartsCostBump: true`**, чтобы не удваивать сумму в `partsCost`); чистый expense — через `installedExpenseItemIds` и прежнюю логику подстановки в строку bundle. Клиент: **`getInstallableForServiceEvent`** в `@mototwin/api-client`.

Дополнительно (паритет web/Expo, 2026-05):

- **Снятие отметки** wishlist/expense снимает соответствующие строки bundle / id из формы симметрично merge (в т.ч. суммы и фрагменты комментария из шаблонов merge).
- **Суммы** из wishlist/expense в форму попадают через тот же парсер денег, что и поля события (**`parseExpenseAmountInputToNumberOrNull`**, учёт `ru-RU` и NBSP).
- **Несколько чистых расходов** в одном событии: целевая строка bundle для каждого следующего расхода выбирается **`resolveInstallableExpenseTargetRow`**, чтобы не перезаписывать строку 0.
- **Сервер:** при сохранении события линковка выбранных расходов (**`linkInstalledExpenseItemsToServiceEvent`**) допускает id, уже синхронизированные с этим `serviceEventId` в той же транзакции, и не ломает сценарий wishlist+expense после `syncExpenseItemForServiceEvent`.

## Data model (Prisma)

- **Enum `PartWishlistItemStatus`:** `NEEDED`, `ORDERED`, `BOUGHT`, `INSTALLED`.
- **Model `PartWishlistItem`:**
  - `id`, `vehicleId`, `nodeId` (обязателен на уровне API/форм, transitional nullable в Prisma), опционально **`skuId`** (каталог `PartSku`)
  - `title`, `quantity` (по умолчанию 1), `status`, опционально `comment`
  - опционально **`costAmount`** (число, как у `ServiceEvent.costAmount`) и **`currency`** (строка, как у `ServiceEvent.currency`); оба поля необязательны, для существующих строк после миграции остаются `NULL`
  - `createdAt`, `updatedAt`
- Связи: позиция принадлежит `Vehicle`, опционально — `Node`, опционально — **`PartSku`**.
- Индексы: `vehicleId`, `nodeId`, `skuId`, `status`.

## API (JSON, без авторизации)

Базовый префикс: `/api/vehicles/[id]/wishlist`.

| Метод | Путь | Назначение |
|--------|------|------------|
| `GET` | `.../wishlist` | Список позиций для мотоцикла |
| `POST` | `.../wishlist` | Создание |
| `PATCH` | `.../wishlist/[itemId]` | Частичное обновление |
| `DELETE` | `.../wishlist/[itemId]` | Удаление |
| `POST` | `.../wishlist/kits` | Добавление комплекта (несколько независимых wishlist-строк) |

Валидация (сервер): **`title` обязателен, если не передан `skuId`** (при `skuId` название может быть пустым в теле запроса — сервер подставит `canonicalName` из SKU); `quantity` — целое ≥ 1, если передано; `status` — допустимый enum; **`skuId`** — если передан, SKU должен существовать и быть **активным** (`isActive`); **`nodeId` обязателен** и должен указывать на существующий **leaf-узел** (для не-leaf возвращается 400: «Выберите конечный узел для позиции списка покупок»). Если в create-запросе нет `nodeId`, сервер может взять `primaryNodeId` выбранного SKU только если это валидный leaf, иначе вернёт 400 и попросит явный выбор узла. PATCH не позволяет очистить `nodeId`. Мотоцикл должен существовать. **`costAmount`** (если передан) — число **≥ 0**; **`currency`** — опционально; если задана стоимость без валюты, на сервере подставляется **`RUB`**. Если стоимости **не задано** (нет ни суммы, ни валюты) и передан **`skuId`**, сервер может скопировать **`priceAmount` / `currency`** из SKU. Если стоимости нет, валюта в БД не хранится (`NULL`). Ответы с явным полем `error` при 4xx (тексты ошибок — по возможности на русском).

**Ответ `GET .../wishlist`:** каждая позиция содержит **`skuId`** (или `null`) и компактный блок **`sku`** (или `null`) — см. `@mototwin/types` (`WishlistItemSkuInfo`: в т.ч. **`primaryPartNumber`** при наличии первого артикула в каталоге).

`POST .../wishlist/kits` принимает `kitCode` (+ опционально `contextNodeId`) и возвращает `createdItems`, `skippedItems`, `warnings`. Дубликаты активных позиций не создаются (пропускаются в `skippedItems`).

**Wishlist в UI:** блок превью на обзоре мотоцикла (**«Подбор деталей»**) остаётся **активным** списком: учитываются только позиции со статусами **`NEEDED`**, **`ORDERED`**, **`BOUGHT`** (для сводки и фильтров). Полная web-страница **«Корзина замен и расходников»** (`/vehicles/[id]/parts`) и полный список в Expo (`/vehicles/[id]/wishlist`) показывают **все** позиции, включая **`INSTALLED`**, потому что они используются как рабочий экран подбора и как обратная ссылка из журнала. Позиции **`INSTALLED`** **не удаляются** из БД и по-прежнему приходят в **`GET .../wishlist`**; история установки смотрится в **журнале обслуживания** после сохранения соответствующего сервисного события. Пустое состояние превью считается по активным позициям: если в работе никого нет, но есть только `INSTALLED`, показывается подсказка про корзину и фильтр по статусу; если списка нет совсем — текст про добавление из дерева или вручную.

**Expense analytics:** строка со статусом **`BOUGHT`**, положительной стоимостью и валютой создаёт/обновляет связанный `ExpenseItem` (`shoppingListItemId = PartWishlistItem.id`, `category = PART`, `installStatus = BOUGHT_NOT_INSTALLED`). Это даёт показатель **«куплено, но не установлено»** на `/expenses` ещё до создания сервисного события. После установки через ServiceEvent расход синхронизируется как установленный (`INSTALLED`) через связанный сервисный расход.

### Статус INSTALLED и журнал обслуживания

**Сервер не создаёт `ServiceEvent` при смене статуса на `INSTALLED`.** `PATCH .../wishlist/[itemId]` только обновляет строку списка покупок; ответ **`{ item }`** — обновлённая позиция, без полей про автосоздание события.

После выбора **`INSTALLED`** клиент ведёт пользователя к **уже существующему** потоку «Добавить сервисное событие», с предзаполнением из позиции и текущего состояния мотоцикла (история затем доступна на dedicated Service Log page/screen):

- **Есть `nodeId`:** открывается форма добавления события; узел предвыбран; дата события — сегодня; пробег и моточасы — текущие значения ТС; тип работ — **«Установка запчасти»**; комментарий собирается из названия позиции, количества (если есть), **при наличии `sku` — строк с каталогом (бренд/название, артикул при наличии, тип SKU)** и комментария из wishlist (если есть); при поддержке поля установленных запчастей — JSON из названия и количества (и **`skuId` / `skuLabel`**, если позиция была связана с каталогом); **если в позиции указаны стоимость и валюта — они переносятся в поля «Стоимость» / «Валюта» формы события** (по умолчанию валюта новых строк wishlist в клиентах — **`RUB`**). На web и Expo статус позиции **не меняется сразу**: он переводится в **`INSTALLED`** только после успешного сохранения сервисного события. Если пользователь просто закрывает модалку/экран, строка остаётся в прежнем статусе. Пользователь может всё отредактировать и **явно сохранить** — только тогда создаётся запись в журнале (`POST .../service-events`) и применяются связанные изменения. **Расходы и сводка по тратам считаются по `ExpenseItem`: wishlist создаёт расход заранее, а сохранённый `ServiceEvent` пересинхронизирует его как установленный.**
- **Нет `nodeId`:** форма сервисного события не открывается автоматически. Пользователь переводится к редактированию позиции/получает подсказку выбрать leaf-узел; статус не должен применяться, пока невозможно создать корректное сервисное событие.

## Shared packages

- **`@mototwin/types`:** `PartWishlistItem` (в т.ч. `skuId`, `sku`, `costAmount`, `currency`), `PartWishlistItemViewModel` (опционально `costLabelRu` для списка), `PartWishlistItemStatus`, `PartWishlistFormValues` (в т.ч. **`skuId`** и обязательный узел на уровне валидации), константа **`PART_WISHLIST_DEFAULT_CURRENCY`** (`RUB`), DTO создания/обновления (`nodeId` обязателен, `skuId` остаётся опциональным), тип ответа **`UpdateWishlistItemResponse`** — **`{ item }`**.
- **`@mototwin/domain`:** подписи статусов на русском, начальные значения формы, валидация (**название или `skuId`** + обязательный узел: «Выберите узел мотоцикла»), нормализация payload (пустая стоимость → `null`, согласование валюты с бэкендом через `normalizePartWishlistCostMutationArgs`, `nodeId` не нормализуется в `null`), **`applyPartSkuViewModelToPartWishlistFormValues`** / **`clearPartWishlistFormSkuSelection`** (клиентский выбор SKU в форме; если узел пуст и у SKU есть `primaryNodeId`, узел предзаполняется), **`applySkuDefaultsToWishlistDraft`** (серверное автозаполнение из SKU при создании), `buildWishlistItemSkuInfo`, подписи для списка и результатов поиска (**`getWishlistItemSkuDisplayLines`**, **`getPartSkuViewModelDisplayLines`**, **`formatPartSkuSearchResultMetaLineRu`**, **`formatWishlistItemSkuSecondaryLineRu`**), `buildPartWishlistItemViewModel`, **`buildPartsCartSummary`** (сводка «все / по статусам» для карточек корзины и превью на обзоре мотоцикла), `groupPartWishlistItemsByStatus`, фильтр активного списка (`isActiveWishlistItem`, `filterActiveWishlistItems`), порядок статусов, `isWishlistTransitionToInstalled`, константа типа работ **«Установка запчасти»** (`WISHLIST_INSTALL_SERVICE_TYPE_RU`), префикс первой строки комментария для распознавания в журнале (`WISHLIST_INSTALL_SERVICE_COMMENT_PREFIX_RU`), эвристика **`isLikelyWishlistInstallServiceEvent`** (тип работ + префикс комментария, без нового `eventKind`), хелпер предзаполнения формы события после установки (`createInitialAddServiceEventFromWishlistItem`, в т.ч. стоимость/валюта из wishlist и **блок каталога в комментарии при `sku`**), текст подсказки для legacy-строк без узла (`WISHLIST_INSTALLED_NO_NODE_SERVICE_HINT`). В **view model** строки журнала поле **`wishlistOriginLabelRu`** дублирует подсказку «Из списка покупок» для таких событий.
- **`@mototwin/api-client`:** `getVehicleWishlist`, `getInstallableForServiceEvent`, `createWishlistItem`, `updateWishlistItem`, `deleteWishlistItem`, а также каталог: **`getPartSkus`**, **`getPartSku`** (см. [parts-catalog-mvp.md](./parts-catalog-mvp.md)).

## Web (`src/app/vehicles/[id]/page.tsx`)

- Обзор мотоцикла (`VehicleDashboard`): секция **«Подбор деталей»** — как компактное превью в мобильном приложении: три колонки **«Нужно купить» / «Заказано» / «Куплено»** с числом активных позиций по каждому статусу (тёмные карточки, цветная полоска сверху, те же акцентные цвета, что на корзине); клик по колонке открывает `/vehicles/[id]/parts?partsStatus=…`; действие **«Список»** в шапке секции — полная корзина без фильтра; **«Добавить деталь»** — модалка создания позиции (как раньше «подобрать» с гаража). Данные сводки: `buildPartsCartSummary` по **активным** позициям; отдельно передаётся число позиций в статусе **`INSTALLED`** для текста «только установленные».
- Раскрываемый блок «Расширенные данные…» в `vehicle-detail-client` по-прежнему может содержать полный инлайн-список wishlist с правками; основной UX превью перенесён в `VehicleDashboard`.
- Полная страница **«Корзина замен и расходников»** (`/vehicles/[id]/parts`): показывает все статусы, включая `INSTALLED`; группы идут в порядке `NEEDED` → `ORDERED` → `BOUGHT` → `INSTALLED`, внутри групп порядок приходит из API (`updatedAt desc`). Для больших списков есть MVP-управление: **ряд компактных карточек сводки** над списком — первая **«Все»** (количество позиций и сумма по `costAmount` по всей корзине), далее по статусам **«Нужно купить»**, **«Заказано»**, **«Куплено»**, **«Установлено»** с теми же метриками; карточка кликабельна и **единственный** переключатель фильтра по статусу (отдельной строки вкладок под списком нет). Поиск по названию/SKU/узлу/стоимости/комментарию и панель **«Фильтры»**; сворачивание и постраничное раскрытие групп работают **только при фильтре «Все»** и без активного поиска (по умолчанию свернута группа `Установлено`, в свернутом состоянии видно до 5 строк). При выборе статусной карточки или при поиске группа сразу показывает все найденные позиции без сворачивания.
- Правая web-панель деталей на `/vehicles/[id]/parts`: выбор строки меняет локальное состояние без `router.replace`, чтобы не мигал интерфейс; deep link `wishlistItemId` используется только при входе/переходе из журнала. Панель содержит compact product block, данные позиции, kit source, раскрываемую прокручиваемую историю и действия. История сортируется от нового события к старому; для смен статуса показывает переход вида **`Статус: Нужно купить → Заказано`**. Точные переходы фиксируются локально после действий пользователя в текущей сессии; для старых строк без сохраненного журнала используется fallback от `Нужно купить` к текущему статусу.
- **Создание новой позиции**: происходит через отдельную страницу **«Подбор детали»** (`/vehicles/[id]/parts/picker`, см. `docs/mototwin-parts-cart-and-picker-ui-spec.md` Часть B v2.0). На этой странице — рекомендации SKU, комплекты обслуживания и поиск каталога одновременно, с правой панелью `Корзина (draft)` и кнопкой `«Перейти к оформлению»`, которая batch-запросом создаёт позиции в wishlist. Точки входа: `«+ Добавить позицию»` и `«+ Добавить комплект»` на cart-странице; быстрые действия из дерева узлов / «Требует внимания» (с `?nodeId=...`); секция «Подбор деталей» на дашборде мотоцикла.
- **Редактирование существующей позиции**: открывается компактная модалка `WishlistItemEditModal` со стоимостью, валютой, статусом, комментарием и опциональным блоком **«SKU из каталога»** (поиск через `getPartSkus`, привязка/очистка SKU). Секций рекомендаций / комплектов в модалке редактирования нет — они только на странице `«Подбор детали»`. По умолчанию валюта **`RUB`**; смена статуса и удаление доступны.
- После выбора **«Установлено»**: при **`nodeId`** открывается модальное окно добавления сервисного события с предзаполнением, но статус применяется только после **успешного сохранения** события; закрытие модалки без сохранения не меняет статус. Без узла — подсказка и переход в редактирование позиции для выбора leaf-узла. Журнал и дерево обновляются после сохранения события пользователем.
- Для установленной позиции на полной странице показывается кнопка **«В журнал»**, если найдено сервисное событие с wishlist в `installedPartsJson` (`source = "wishlist"`, один или несколько `wishlistItemId` в объекте или массиве). Переход ведёт в `/vehicles/[id]/service-log?serviceEventId=...`; журнал скроллит к событию и подсвечивает его. В обратную сторону **«Из списка покупок»** в строке журнала: при одном id — одна ссылка на `/vehicles/[id]/parts?wishlistItemId=...`; при нескольких — подпись с номерами **1, 2, …**, каждый ведёт к своей позиции.
- Из дерева узлов и блока внимания — быстрое добавление: переход на `/vehicles/[id]/parts/picker?nodeId=...`, где блок рекомендаций SKU и комплекты сразу подгружены под выбранный узел.
- На странице **«Подбор детали»** (один маршрут) доступны как одиночные SKU (рекомендации + поиск), так и **комплекты обслуживания**: при клике `«Добавить комплект»` кит попадает в локальный draft cart как одна агрегированная строка; на сабмит backend (`POST .../wishlist/kits`) создаёт несколько независимых wishlist-строк.
- Перед сабмитом draft cart показывается preview-модалка с разбивкой `«Будет добавлено»` / `«Уже в активном списке»` / `«Невозможно добавить»`. Если доступных элементов нет, кнопка `«Перейти к оформлению»` disabled.
- Для строк, созданных из комплекта, в списке показывается компактный badge `Из комплекта: <название>` (UI-only маркировка из `comment`, без новых полей в БД/контракте). Основной комментарий позиции остаётся видимым без дублирования строки префикса.

## Expo (`apps/app/app/vehicles/[id]/wishlist/**`)

- Экран списка: `/vehicles/[id]/wishlist` — полный список всех статусов, включая `INSTALLED`; порядок групп такой же, как web (`NEEDED` → `ORDERED` → `BOUGHT` → `INSTALLED`), внутри групп порядок API (`updatedAt desc`). Сводные карточки по статусам и поиск/фильтры в духе web-корзины. Карточка строки показывает **компактный блок каталога при `sku`**, **строку стоимости при наличии**, комплектный badge, комментарий, **интерактивный бейдж статуса** (тап открывает **`Modal`** с выбором нового статуса; подтверждения удаления и ошибки по-прежнему через **`Alert.alert`**), удаление со свайпа влево с подтверждением. **Тап по строке** открывает **нижнюю модалку** с содержимым и оформлением как у **правой панели деталей** на web (блок товара, поля, история, действия, переход в журнал при связанном событии); **свайп вправо** — переход на экран редактирования `/wishlist/[itemId]`.
- **Фокус после смены статуса** (из bottom-sheet меню, если новый статус **не** «Установлено»): после успешного `PATCH` и `load()` вызывается **`applyWishlistRowFocus`** (`wishlist/index.tsx`) — фильтр сводки на статус позиции, сброс поиска, раскрытие группы, увеличение «Показать ещё» при необходимости, прокрутка к строке (~120 ms), открытие модалки деталей (~220 ms). Навигация **без** `router.replace?wishlistItemId=…` для этого сценария: иначе при смене URL снова срабатывает фокус экрана → `useFocusEffect` вызывает `load()` с `setIsLoading(true)` и полноэкранным «Загрузка…», что сбрасывает модалку и даёт мигание. **`load()`** при успехе возвращает **`PartWishlistItemViewModel[] | null`**, чтобы передать актуальный снимок списка в `applyWishlistRowFocus`. **Диплинк** `?wishlistItemId=` из журнала обрабатывается отдельным `useEffect`: фильтр/раскрытие/скролл, затем автооткрытие модалки и **`replace` URL без query** (чтобы не открывать повторно при следующем фокусе).
- После выбора статуса **Установлено** с **`nodeId`** — переход на экран добавления сервисного события с предзаполнением; статус применяется только после сохранения события. Без узла — подсказка выбрать узел; после сохранения события — возврат к списку покупок с подсветкой установленной позиции; при фокусе карточки мотоцикла дерево и журнал подтягиваются как раньше.
- Создание: переход на экран `/vehicles/[id]/wishlist/picker` (опциональный `?nodeId=`, `?focus=kits`); это single-page picker (см. spec Часть B v2.0). Старый маршрут `/vehicles/[id]/wishlist/new` остаётся как redirect на picker с пробросом query.
- Редактирование: `/vehicles/[id]/wishlist/[itemId]` (упрощённая форма edit-only, без секций рекомендаций / комплектов).
- Picker экран в Expo keyboard-aware: контент прокручивается при открытой клавиатуре (`KeyboardAvoidingView` + `ScrollView`), **`keyboardShouldPersistTaps="handled"`** для выбора SKU из списка при открытой клавиатуре; те же endpoint-ы, что и на web (`getPartSkus`, `getRecommendedSkusForNode`, `getServiceKits`).
- Комплекты обслуживания добавляются на picker как одна агрегированная строка draft; на сабмит вызывается `addServiceKitToWishlist`, который создаёт несколько строк (с тем же preview / дедупликацией, что и на web).
- На карточках wishlist в Expo используется та же UI-only маркировка происхождения из комплекта (`Из комплекта: ...`) с тем же смыслом, что на web.
- Для установленной позиции в Expo показывается **«В журнал»**, если найдено связанное сервисное событие. Журнал Expo принимает `serviceEventId`, скроллит/подсвечивает событие и делает **«Из списка покупок»** кликабельным переходом обратно к конкретной позиции (`wishlistItemId`).
- С экрана мотоцикла (`/vehicles/[id]`): секция **«Подбор деталей»** — три компактные колонки по статусам `NEEDED` / `ORDERED` / `BOUGHT` (клик ведёт в wishlist с `?partsStatus=…`), шапка **«Список»** — полный список, кнопка **«Добавить деталь»** — `/wishlist/new`. После перехода в wishlist/new и **возврата назад** список дашборда **прокручивается** к блоку «Подбор деталей» (флаг перед навигацией + скролл после завершения загрузки данных). В дереве узлов и в **«Требует внимания»** — быстрые действия с переходом в wishlist/new с `nodeId` (как в MVP ранее).

## Parity

Один и тот же backend и те же клиентские контракты. Dashboard/preview wishlist остаётся **активным списком**; full parts/wishlist screens на web и Expo показывают все статусы для операционной работы и обратных ссылок из журнала. **История установок** — через **журнал обслуживания**. Доступны: просмотр, создание, обновление статуса, удаление, обязательная привязка к leaf-узлу, **опциональный выбор SKU из каталога (поиск + очистка)**, **консервативные рекомендации SKU по выбранному узлу с одинаковыми метками**, **опциональная стоимость и валюта (по умолчанию `RUB`)**, перенос стоимости и **данных каталога в комментарий** сервисного события при **`INSTALLED`** с **`nodeId`**, и быстрый вход с предвыбранным узлом.
