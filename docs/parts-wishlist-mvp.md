# Parts Wishlist MVP («Что нужно купить»)

## Scope

Практичный **операционный** список запчастей и расходников для конкретного мотоцикла: что нужно докупить, что заказано и что куплено. Статус **«Установлено»** в продукте ведёт к фиксации факта в **журнале обслуживания** (после сохранения сервисного события), а не к отдельному архиву внутри wishlist. Это **не** маркетплейс и **не** полный fitment-движок; опционально позиция может быть связана с **каталожным SKU** через ручной поиск (`getPartSkus`) или через **консервативные node-based рекомендации** (`getRecommendedSkusForNode`).

Один мотоцикл — много позиций. Каждая позиция **обязательно** привязана к **конечному (leaf) узлу** дерева мотоцикла.

Также в MVP поддерживаются **комплекты обслуживания** (service kits): один action создаёт сразу несколько wishlist-строк. Это только batch-добавление в wishlist; установка и сервисные события по-прежнему выполняются **по одной позиции**.

## Data model (Prisma)

- **Enum `PartWishlistItemStatus`:** `NEEDED`, `ORDERED`, `BOUGHT`, `INSTALLED`.
- **Model `PartWishlistItem`:**
  - `id`, `vehicleId`, `nodeId` (обязателен на уровне API/форм, transitional nullable в Prisma), опционально **`skuId`** (каталог `PartSku`)
  - `title`, `quantity` (по умолчанию 1), `status`, опционально `comment`
  - опционально **`costAmount`** (число, как у `ServiceEvent.costAmount`) и **`currency`** (строка, как у `ServiceEvent.currency`); оба поля необязательны, для существующих строк после миграции остаются `NULL`
  - `createdAt`, `updatedAt`
- Связи: позиция принадлежит `Vehicle`, опционально — `Node`, опционально — **`PartSku`**.
- Индексы: `vehicleId`, `nodeId`, `skuId`, `status`.

## API (JSON, без авторизации)

Базовый префикс: `/api/vehicles/[id]/wishlist`.

| Метод | Путь | Назначение |
|--------|------|------------|
| `GET` | `.../wishlist` | Список позиций для мотоцикла |
| `POST` | `.../wishlist` | Создание |
| `PATCH` | `.../wishlist/[itemId]` | Частичное обновление |
| `DELETE` | `.../wishlist/[itemId]` | Удаление |
| `POST` | `.../wishlist/kits` | Добавление комплекта (несколько независимых wishlist-строк) |

Валидация (сервер): **`title` обязателен, если не передан `skuId`** (при `skuId` название может быть пустым в теле запроса — сервер подставит `canonicalName` из SKU); `quantity` — целое ≥ 1, если передано; `status` — допустимый enum; **`skuId`** — если передан, SKU должен существовать и быть **активным** (`isActive`); **`nodeId` обязателен** и должен указывать на существующий **leaf-узел** (для не-leaf возвращается 400: «Выберите конечный узел для позиции списка покупок»). Если в create-запросе нет `nodeId`, сервер может взять `primaryNodeId` выбранного SKU только если это валидный leaf, иначе вернёт 400 и попросит явный выбор узла. PATCH не позволяет очистить `nodeId`. Мотоцикл должен существовать. **`costAmount`** (если передан) — число **≥ 0**; **`currency`** — опционально; если задана стоимость без валюты, на сервере подставляется **`RUB`**. Если стоимости **не задано** (нет ни суммы, ни валюты) и передан **`skuId`**, сервер может скопировать **`priceAmount` / `currency`** из SKU. Если стоимости нет, валюта в БД не хранится (`NULL`). Ответы с явным полем `error` при 4xx (тексты ошибок — по возможности на русском).

**Ответ `GET .../wishlist`:** каждая позиция содержит **`skuId`** (или `null`) и компактный блок **`sku`** (или `null`) — см. `@mototwin/types` (`WishlistItemSkuInfo`: в т.ч. **`primaryPartNumber`** при наличии первого артикула в каталоге).

`POST .../wishlist/kits` принимает `kitCode` (+ опционально `contextNodeId`) и возвращает `createdItems`, `skippedItems`, `warnings`. Дубликаты активных позиций не создаются (пропускаются в `skippedItems`).

**Wishlist в UI:** dashboard-блок «Что нужно купить» остаётся активным preview и показывает только позиции со статусами **`NEEDED`**, **`ORDERED`**, **`BOUGHT`**. Полная web-страница **«Корзина замен и расходников»** (`/vehicles/[id]/parts`) показывает **все** позиции, включая **`INSTALLED`**, потому что она используется как рабочий экран подбора и как обратная ссылка из журнала. Позиции **`INSTALLED`** **не удаляются** из БД и по-прежнему приходят в **`GET .../wishlist`**; история установки смотрится в **журнале обслуживания** после сохранения соответствующего сервисного события. Пустое состояние dashboard считается по активным позициям: если все строки только `INSTALLED`, показывается сообщение вроде «Список покупок пуст» / «Все позиции установлены» и краткая подсказка про журнал.

**Expense analytics:** строка со статусом **`BOUGHT`**, положительной стоимостью и валютой создаёт/обновляет связанный `ExpenseItem` (`shoppingListItemId = PartWishlistItem.id`, `category = PART`, `installStatus = BOUGHT_NOT_INSTALLED`). Это даёт показатель **«куплено, но не установлено»** на `/expenses` ещё до создания сервисного события. После установки через ServiceEvent расход синхронизируется как установленный (`INSTALLED`) через связанный сервисный расход.

### Статус INSTALLED и журнал обслуживания

**Сервер не создаёт `ServiceEvent` при смене статуса на `INSTALLED`.** `PATCH .../wishlist/[itemId]` только обновляет строку списка покупок; ответ **`{ item }`** — обновлённая позиция, без полей про автосоздание события.

После выбора **`INSTALLED`** клиент ведёт пользователя к **уже существующему** потоку «Добавить сервисное событие», с предзаполнением из позиции и текущего состояния мотоцикла (история затем доступна на dedicated Service Log page/screen):

- **Есть `nodeId`:** открывается форма добавления события; узел предвыбран; дата события — сегодня; пробег и моточасы — текущие значения ТС; тип работ — **«Установка запчасти»**; комментарий собирается из названия позиции, количества (если есть), **при наличии `sku` — строк с каталогом (бренд/название, артикул при наличии, тип SKU)** и комментария из wishlist (если есть); при поддержке поля установленных запчастей — JSON из названия и количества (и **`skuId` / `skuLabel`**, если позиция была связана с каталогом); **если в позиции указаны стоимость и валюта — они переносятся в поля «Стоимость» / «Валюта» формы события** (по умолчанию валюта новых строк wishlist в клиентах — **`RUB`**). На web и Expo статус позиции **не меняется сразу**: он переводится в **`INSTALLED`** только после успешного сохранения сервисного события. Если пользователь просто закрывает модалку/экран, строка остаётся в прежнем статусе. Пользователь может всё отредактировать и **явно сохранить** — только тогда создаётся запись в журнале (`POST .../service-events`) и применяются связанные изменения. **Расходы и сводка по тратам считаются по `ExpenseItem`: wishlist создаёт расход заранее, а сохранённый `ServiceEvent` пересинхронизирует его как установленный.**
- **Нет `nodeId`:** форма сервисного события не открывается автоматически. Пользователь переводится к редактированию позиции/получает подсказку выбрать leaf-узел; статус не должен применяться, пока невозможно создать корректное сервисное событие.

## Shared packages

- **`@mototwin/types`:** `PartWishlistItem` (в т.ч. `skuId`, `sku`, `costAmount`, `currency`), `PartWishlistItemViewModel` (опционально `costLabelRu` для списка), `PartWishlistItemStatus`, `PartWishlistFormValues` (в т.ч. **`skuId`** и обязательный узел на уровне валидации), константа **`PART_WISHLIST_DEFAULT_CURRENCY`** (`RUB`), DTO создания/обновления (`nodeId` обязателен, `skuId` остаётся опциональным), тип ответа **`UpdateWishlistItemResponse`** — **`{ item }`**.
- **`@mototwin/domain`:** подписи статусов на русском, начальные значения формы, валидация (**название или `skuId`** + обязательный узел: «Выберите узел мотоцикла»), нормализация payload (пустая стоимость → `null`, согласование валюты с бэкендом через `normalizePartWishlistCostMutationArgs`, `nodeId` не нормализуется в `null`), **`applyPartSkuViewModelToPartWishlistFormValues`** / **`clearPartWishlistFormSkuSelection`** (клиентский выбор SKU в форме; если узел пуст и у SKU есть `primaryNodeId`, узел предзаполняется), **`applySkuDefaultsToWishlistDraft`** (серверное автозаполнение из SKU при создании), `buildWishlistItemSkuInfo`, подписи для списка и результатов поиска (**`getWishlistItemSkuDisplayLines`**, **`getPartSkuViewModelDisplayLines`**, **`formatPartSkuSearchResultMetaLineRu`**, **`formatWishlistItemSkuSecondaryLineRu`**), `buildPartWishlistItemViewModel`, `groupPartWishlistItemsByStatus`, фильтр активного списка (`isActiveWishlistItem`, `filterActiveWishlistItems`), порядок статусов, `isWishlistTransitionToInstalled`, константа типа работ **«Установка запчасти»** (`WISHLIST_INSTALL_SERVICE_TYPE_RU`), префикс первой строки комментария для распознавания в журнале (`WISHLIST_INSTALL_SERVICE_COMMENT_PREFIX_RU`), эвристика **`isLikelyWishlistInstallServiceEvent`** (тип работ + префикс комментария, без нового `eventKind`), хелпер предзаполнения формы события после установки (`createInitialAddServiceEventFromWishlistItem`, в т.ч. стоимость/валюта из wishlist и **блок каталога в комментарии при `sku`**), текст подсказки для legacy-строк без узла (`WISHLIST_INSTALLED_NO_NODE_SERVICE_HINT`). В **view model** строки журнала поле **`wishlistOriginLabelRu`** дублирует подсказку «Из списка покупок» для таких событий.
- **`@mototwin/api-client`:** `getVehicleWishlist`, `createWishlistItem`, `updateWishlistItem`, `deleteWishlistItem`, а также каталог: **`getPartSkus`**, **`getPartSku`** (см. [parts-catalog-mvp.md](./parts-catalog-mvp.md)).

## Web (`src/app/vehicles/[id]/page.tsx`)

- Dashboard-секция «Что нужно купить»: группировка по статусу только для активных позиций (`INSTALLED` скрыты); поля карточки (название, **компактный блок каталога при `sku`**, узел, количество, **стоимость при наличии**, статус, комментарий); краткий текст, что установленные позиции попадают в **журнал обслуживания** после сохранения события.
- Полная страница **«Корзина замен и расходников»** (`/vehicles/[id]/parts`): показывает все статусы, включая `INSTALLED`; группы идут в порядке `NEEDED` → `ORDERED` → `BOUGHT` → `INSTALLED`, внутри групп порядок приходит из API (`updatedAt desc`). Для больших списков есть MVP-управление: status chips с количеством (`Все`, `Нужно купить`, `Заказано`, `Куплено`, `Установлено`), поиск по названию/SKU/узлу/стоимости/комментарию, сворачиваемые группы (по умолчанию свернута `Установлено`) и постраничное раскрытие внутри группы через **«Показать ещё 10»**.
- Создание/редактирование в модальном окне: опциональный блок **«SKU из каталога»** — поиск через **`getPartSkus`** (от 2 символов; если в форме выбран **узел**, допускается запрос по узлу без текста); выбор позиции из списка, **«Очистить SKU»**; при выборе подставляются (если поля пустые) название, стоимость/валюта, узел из SKU; всё можно править вручную. Ручное создание **без SKU** сохраняется. В модалке также **стоимость и валюта**, по умолчанию валюта **`RUB`**; смена статуса и удаление.
- После выбора **«Установлено»**: при **`nodeId`** открывается модальное окно добавления сервисного события с предзаполнением, но статус применяется только после **успешного сохранения** события; закрытие модалки без сохранения не меняет статус. Без узла — подсказка и переход в редактирование позиции для выбора leaf-узла. Журнал и дерево обновляются после сохранения события пользователем.
- Для установленной позиции на полной странице показывается кнопка **«В журнал»**, если найдено сервисное событие с `installedPartsJson.source = "wishlist"` и `wishlistItemId`. Переход ведёт в `/vehicles/[id]/service-log?serviceEventId=...`; журнал скроллит к событию и подсвечивает его. В обратную сторону текст **«Из списка покупок»** в строке журнала кликабелен, если из `installedPartsJson` можно прочитать `wishlistItemId`; он открывает `/vehicles/[id]/parts?wishlistItemId=...`, раскрывает/фильтрует нужную группу, подсвечивает карточку и скроллит к ней.
- Из дерева узлов и блока внимания — быстрое добавление с предвыбранным `nodeId` (где уже сделано в продукте). При выбранном узле в форме показывается блок рекомендаций SKU с честными метками совместимости и CTA **«Добавить в список покупок»** (или «Применить SKU» в режиме редактирования).
- В форме создания позиции (контекст выбранного узла) доступны **комплекты обслуживания**: по кнопке «Добавить комплект в список покупок» backend создаёт несколько независимых wishlist-строк; flow установки для них остаётся поштучным.
- Перед добавлением комплекта показывается preview по элементам (будет добавлено / уже есть в активном списке / узел не сопоставлен). Если доступных элементов нет, добавление блокируется.
- Для строк, созданных из комплекта, в списке показывается компактный badge `Из комплекта: <название>` (UI-only маркировка из `comment`, без новых полей в БД/контракте). Основной комментарий позиции остаётся видимым без дублирования строки префикса.

## Expo (`apps/app/app/vehicles/[id]/wishlist/**`)

- Экран списка: `/vehicles/[id]/wishlist` — полный список всех статусов, включая `INSTALLED`; порядок групп такой же, как web (`NEEDED` → `ORDERED` → `BOUGHT` → `INSTALLED`), внутри групп порядок API (`updatedAt desc`). Есть status chips с количеством, поиск по названию/SKU/узлу/стоимости/комментарию, сворачиваемые группы (по умолчанию свернута `Установлено`) и **«Показать ещё»** внутри группы. Карточка показывает **компактный блок каталога при `sku`**, **строку стоимости при наличии**, комплектный badge, комментарий, смену статуса через системный `Alert`, удаление с подтверждением и переход к редактированию по нажатию.
- После выбора статуса **Установлено** с **`nodeId`** — переход на экран добавления сервисного события с предзаполнением; статус применяется только после сохранения события. Без узла — подсказка выбрать узел; после сохранения события — возврат к списку покупок с подсветкой установленной позиции; при фокусе карточки мотоцикла дерево и журнал подтягиваются как раньше.
- Создание: `/vehicles/[id]/wishlist/new` (query `nodeId` — предзаполнение узла).
- Редактирование: `/vehicles/[id]/wishlist/[itemId]`.
- Форма создания/редактирования wishlist в Expo keyboard-aware: контент прокручивается при открытой клавиатуре (`KeyboardAvoidingView` + `ScrollView`), **`keyboardShouldPersistTaps="handled"`** для выбора SKU из списка при открытой клавиатуре; тот же опциональный поиск/выбор SKU и node-based рекомендации, что и на web (`getPartSkus`, `getRecommendedSkusForNode`).
- В режиме создания доступны те же **комплекты обслуживания**, что и на web: один tap добавляет несколько строк в wishlist, каждая строка независима для дальнейшей установки.
- На Expo используется тот же preview и те же правила дедупликации active wishlist-строк перед добавлением kit.
- На карточках wishlist в Expo используется та же UI-only маркировка происхождения из комплекта (`Из комплекта: ...`) с тем же смыслом, что на web.
- Для установленной позиции в Expo показывается **«В журнал»**, если найдено связанное сервисное событие. Журнал Expo принимает `serviceEventId`, скроллит/подсвечивает событие и делает **«Из списка покупок»** кликабельным переходом обратно к конкретной позиции (`wishlistItemId`).
- С карточки мотоцикла — блок-ссылка «Запчасти и расходники»; в дереве узлов — действие добавления/открытия wishlist с передачей `nodeId`; в dashboard-блоке **«Требует внимания»** тап по узлу открывает bottom sheet быстрых действий, где **«Подбор запчасти»** ведёт в `/vehicles/[id]/wishlist/new?nodeId=...`.

## Parity

Один и тот же backend и те же клиентские контракты. Dashboard/preview wishlist остаётся **активным списком**; full parts/wishlist screens на web и Expo показывают все статусы для операционной работы и обратных ссылок из журнала. **История установок** — через **журнал обслуживания**. Доступны: просмотр, создание, обновление статуса, удаление, обязательная привязка к leaf-узлу, **опциональный выбор SKU из каталога (поиск + очистка)**, **консервативные рекомендации SKU по выбранному узлу с одинаковыми метками**, **опциональная стоимость и валюта (по умолчанию `RUB`)**, перенос стоимости и **данных каталога в комментарий** сервисного события при **`INSTALLED`** с **`nodeId`**, и быстрый вход с предвыбранным узлом.
