# MotoTwin — дополненный дизайн-концепт для Cursor

> [!IMPORTANT]
> Актуальная версия дизайн-правил перенесена в единый документ: `docs/mototwin_unified_design_concept.md`.
> Этот файл сохранен как reference по интеграции существующих flow (дерево/журнал).

## Назначение документа

Этот документ дополняет дизайн-концепт MotoTwin с учетом уже реализованного функционала:

1. открытие полного дерева узлов мотоцикла;
2. открытие полного журнала обслуживания;
3. отображение на странице мотоцикла только статусов по top-nodes;
4. визуальный стиль иконок для top-nodes в стиле MotoTwin;
5. сохранение MVP-подхода: без перегруза страницы конкретного мотоцикла.

Главный принцип:

```text
Страница мотоцикла = краткая понятная сводка по top-nodes.
Полное дерево узлов = отдельный экран / существующий функционал.
Полный журнал обслуживания = отдельный экран / существующий функционал.
```

---

# 1. Общая логика интерфейса

MotoTwin должен выглядеть как **цифровой гараж**, но не превращаться в перегруженный инженерный интерфейс.

Страница конкретного мотоцикла должна отвечать на вопросы:

1. что это за мотоцикл;
2. в каком состоянии основные top-nodes;
3. что требует внимания;
4. какие последние работы были выполнены;
5. какие ближайшие действия;
6. куда перейти для подробностей.

Важно:

```text
На странице мотоцикла не нужно показывать полное дерево всех узлов.
На странице мотоцикла показываются только top-nodes.
```

Полное дерево открывается отдельным действием:

```text
Открыть полное дерево узлов
```

Полный журнал открывается отдельным действием:

```text
Открыть полный журнал обслуживания
```

# 2. Информационная архитектура

## 2.1. Основные уровни

В интерфейсе есть три уровня детализации.

### Уровень 1. Гараж

Экран со списком мотоциклов пользователя.

Показывает:

- список мотоциклов;
- краткое состояние каждого;
- Garage Score;
- быстрые действия;
- ближайшие проблемы.

### Уровень 2. Страница мотоцикла

Экран конкретного мотоцикла.

Показывает:

- профиль мотоцикла;
- top-nodes;
- краткие статусы top-nodes;
- блок “Требует внимания”;
- последние события журнала;
- краткий блок расходов;
- preview подбора деталей;
- ссылки на полное дерево и полный журнал.

### Уровень 3. Детальные экраны

Отдельные экраны:

- полное дерево узлов;
- детальная страница узла;
- полный журнал обслуживания;
- полная страница расходов;
- подбор деталей.

# 3. Страница “Мой гараж”

## 3.1. Назначение

Страница “Мой гараж” — главный экран после входа.

Она показывает список мотоциклов пользователя в виде карточек.

## 3.2. Содержание карточки мотоцикла

Карточка мотоцикла содержит:

1. бренд и модель;
2. год;
3. пробег;
4. ride profile;
5. маленький схематичный силуэт класса;
6. Garage Score;
7. краткие счетчики статусов top-nodes;
8. краткий блок “Требует внимания”;
9. быстрые действия.

Пример:

```text
BMW F 850 GS Adventure
2021 · 42 180 км · Mixed Touring

[малый силуэт Adventure]      Garage Score 82 / 100

Top-nodes:
7 OK · 2 Soon · 1 Overdue · 0 Recently

Требует внимания:
Задняя шина · Просрочено
Тормозные колодки · Скоро

[Открыть] [ТО] [Расход]
```

## 3.3. Важно

На карточке мотоцикла не показывать полное дерево узлов.

Карточка показывает только агрегированное состояние top-nodes.

# 4. Страница конкретного мотоцикла

## 4.1. Назначение

Страница конкретного мотоцикла — это главный обзор цифрового двойника.

Она не должна быть полной энциклопедией всех узлов.

Она должна быть компактной и рабочей.

## 4.2. Структура страницы

```text
Header
Hero card / карта мотоцикла
Garage Score
Top-nodes status
Требует внимания
Быстрые действия
Последние события журнала
Расходы
Подбор деталей
Переходы к полным разделам
```

## 4.3. Header

Показывать:

```text
BMW F 850 GS Adventure
2021 · 42 180 км · Mixed Touring
VIN: WB10K...
```

Кнопки:

```text
[Добавить ТО]
[Добавить расход]
[Подобрать деталь]
```

На мобильном:

```text
[ТО] [Расход] [Деталь]
```

# 5. Карта мотоцикла

## 5.1. Принцип

Карта мотоцикла — это не фото, а схематичный силуэт класса.

Подпись:

```text
Схематичный вид · Adventure class
```

или:

```text
Схематичный вид · Cruiser class
```

## 5.2. Что показывать на карте

На карте показывать только top-nodes.

Не выводить все вложенные узлы.

Пример top-nodes:

1. Двигатель
2. Тормоза
3. Колеса / шины
4. Цепь / трансмиссия
5. Электрика / АКБ
6. Подвеска
7. Охлаждение
8. Фильтры / расходники

Для MVP overview используется 6 агрегированных карточек (источник: TOP-12):

1. Двигатель
2. Тормоза
3. Шины
4. Цепь / звезды
5. Электрика
6. Подвеска

## 5.3. Поведение top-node на карте

По клику на top-node:

- открыть detail top-node или существующий экран узла;
- показать связанные события;
- показать статус;
- показать ближайшее обслуживание;
- показать совместимые детали, если есть.

Если уже реализовано открытие дерева узлов, то кнопка или ссылка должна вести в существующий функционал.

# 6. Top-nodes на странице мотоцикла

## 6.1. Главный принцип

На странице мотоцикла отображаются статусы **только по top-nodes**.

Не нужно выводить вложенные элементы дерева на главную страницу мотоцикла.

Правильно:

```text
Двигатель · OK
Тормоза · Soon
Шины · Overdue
Цепь / звезды · OK
Аккумулятор · OK
Подвеска · Recently
```

Неправильно:

```text
Двигатель
  Масло
  Масляный фильтр
  Свечи
  Воздушный фильтр
  Охлаждение
  ...
```

Вложенные элементы должны быть доступны через полное дерево узлов.

## 6.2. Формат блока

```text
Состояние основных узлов

[Двигатель]       OK          Масло через 1 800 км
[Тормоза]         Soon        Проверить колодки через 450 км
[Шины]            Overdue     Задняя шина требует замены
[Цепь / звезды]   OK          Обслужено 220 км назад
[Аккумулятор]     OK          Напряжение в норме
[Подвеска]        Recently    Сальники вилки заменены

[Открыть полное дерево узлов]
```

## 6.3. Desktop layout

На desktop показывать top-nodes карточками:

```text
┌ Двигатель ─────────────┐  ┌ Тормоза ─────────────┐  ┌ Шины ───────────────┐
│ OK                     │  │ Soon                 │  │ Overdue             │
│ Масло через 1 800 км   │  │ Проверить колодки    │  │ Задняя шина         │
└────────────────────────┘  └──────────────────────┘  └────────────────────┘
```

## 6.4. Mobile layout

На mobile показывать top-nodes как сетку 2×3 или вертикальный список.

Вариант 2×3:

```text
[Двигатель OK] [Тормоза Soon]
[Шины Overdue] [Цепь OK]
[АКБ OK]       [Подвеска Recently]
```

Если текста много, использовать вертикальный список.

# 7. Полное дерево узлов

## 7.1. Уже реализованный функционал

В проекте уже есть функционал открытия полного дерева узлов.

Этот функционал нужно встроить в дизайн как отдельный сценарий.

На странице мотоцикла должен быть явный переход:

```text
Открыть полное дерево узлов
```

или:

```text
Все узлы и регламенты →
```

## 7.2. Где разместить ссылку

Ссылку можно разместить:

1. в конце блока “Состояние основных узлов”;
2. в quick actions;
3. в tab navigation;
4. в нижней части карты мотоцикла.

Рекомендуемый вариант:

```text
Состояние основных узлов                         Все узлы →
```

## 7.3. Как должен выглядеть экран полного дерева

Экран полного дерева должен показывать уже существующую структуру.

Дизайн должен быть согласован с общей темой:

- темный или светлый фон в зависимости от темы;
- карточки или tree-list;
- top-nodes как крупные группы;
- вложенные узлы раскрываются;
- статусы видны на уровне top-node и child-node;
- поиск / фильтр можно добавить позже.

## 7.4. Пример структуры

```text
Полное дерево узлов

Двигатель
  Масло двигателя
  Масляный фильтр
  Воздушный фильтр
  Свечи
  Охлаждение

Тормоза
  Передние колодки
  Задние колодки
  Передний диск
  Задний диск
  Тормозная жидкость

Колеса / шины
  Передняя шина
  Задняя шина
  Давление
  Балансировка

Цепь / звезды
  Цепь
  Передняя звезда
  Задняя звезда
  Натяжение
```

## 7.5. Важно для Cursor

Cursor не должен заново придумывать и переписывать функционал дерева, если он уже реализован.

Нужно только:

- встроить ссылку / кнопку;
- привести визуальный переход к новому дизайну;
- сохранить существующую бизнес-логику;
- не ломать текущие маршруты.

# 8. Полный журнал обслуживания

## 8.1. Уже реализованный функционал

В проекте уже есть функционал открытия полного журнала обслуживания.

На странице мотоцикла нужно показывать только preview последних событий.

Полный журнал открывается по ссылке:

```text
Открыть полный журнал обслуживания
```

или:

```text
Все события →
```

## 8.2. Preview журнала на странице мотоцикла

Показывать 3–5 последних событий.

Пример:

```text
Последние события

14.04.2026  Замена масла              4 200 ₽
12.04.2026  Масляный фильтр HF155       900 ₽
02.04.2026  Передняя шина            15 000 ₽

Все события журнала →
```

## 8.3. Что не показывать на странице мотоцикла

На странице мотоцикла не нужно показывать:

- все события журнала;
- длинные фильтры;
- расширенную таблицу;
- историю по всем годам;
- сложную сортировку.

Это должно быть в полном журнале.

## 8.4. Экран полного журнала

Полный журнал должен использовать существующий функционал.

Дизайн должен поддерживать:

- список событий;
- фильтр по узлу;
- фильтр по периоду;
- фильтр “только события с расходами”;
- создание нового события;
- переход к событию.

## 8.5. Пример полного журнала

```text
Журнал обслуживания

[Добавить событие]

Фильтры:
Период: Апрель 2026
Узел: Все узлы
[✓] Только события с расходами

14.04.2026
Двигатель
Замена масла
4 200 ₽

12.04.2026
Двигатель
Масляный фильтр HF155
900 ₽

02.04.2026
Колеса / шины
Передняя шина
15 000 ₽
```

## 8.6. Важно для Cursor

Cursor должен:

- оставить существующий функционал журнала;
- не переписывать бизнес-логику;
- встроить переход из preview;
- привести визуальный стиль к общей системе MotoTwin;
- не тащить полный журнал на overview-страницу мотоцикла.

# 9. Статусы top-nodes

## 9.1. Используемые статусы

Для top-nodes использовать 4 статуса:

```text
OK
Soon
Overdue
Recently
```

## 9.2. Русские пояснения

В компактных бейджах можно использовать английские короткие статусы.

В поясняющем тексте использовать русский:

```text
OK = В норме
Soon = Скоро
Overdue = Просрочено
Recently = Недавно обслужено
```

## 9.3. Цвета статусов

Темная и светлая темы используют одинаковую смысловую логику:

```css
OK: зеленый
Soon: желтый
Overdue: красный
Recently: голубой
```

## 9.4. Важно

Не полагаться только на цвет.

Всегда показывать текст статуса.

# 10. Иконки top-nodes в стиле MotoTwin

## 10.1. Общий стиль

Иконки top-nodes должны выглядеть как часть технической системы MotoTwin.

Стиль:

- line icons;
- 1.75–2 px stroke;
- rounded line caps;
- без мультяшности;
- без лишней детализации;
- монохромная база;
- статусный цвет используется как акцент;
- хорошо читаются в размере 20–32 px.

## 10.2. Цвет иконок

Базовый цвет в темной теме:

```css
--icon-default-dark: #9CA3AF;
```

Базовый цвет в светлой теме:

```css
--icon-default-light: #4B5563;
```

При статусе можно окрашивать:

- только маленький индикатор;
- или stroke иконки;
- или фон бейджа.

Рекомендуемый MVP-вариант:

```text
Иконка остается серой.
Статус показывается отдельной цветной точкой / бейджем.
```

Так интерфейс выглядит спокойнее.

## 10.3. Иконки для top-nodes

### 1. Двигатель

Образ:

- упрощенный блок двигателя;
- два цилиндра или технический корпус;
- маленькая линия снизу как картер.

Описание для генерации / отрисовки:

```text
Minimal outline icon of motorcycle engine block, technical line style, rounded stroke, no brand details.
```

### 2. Тормоза

Образ:

- тормозной диск;
- суппорт;
- несколько отверстий на диске.

Описание:

```text
Minimal outline icon of motorcycle brake disc with caliper, technical line style, circular geometry.
```

### 3. Шины / колеса

Образ:

- колесо / шина сбоку;
- несколько линий протектора;
- можно использовать один круг с внутренним кольцом.

Описание:

```text
Minimal outline icon of motorcycle tire, side view, circular tire with simple tread marks.
```

### 4. Цепь / звезды

Образ:

- короткий фрагмент цепи;
- или цепь вокруг маленькой звезды;
- лучше не перегружать.

Описание:

```text
Minimal outline icon of motorcycle chain link and sprocket, technical line style.
```

### 5. Аккумулятор / электрика

Образ:

- аккумулятор с плюс/минус;
- можно добавить маленький bolt.

Описание:

```text
Minimal outline icon of motorcycle battery with plus and minus marks, technical line style.
```

### 6. Подвеска

Образ:

- амортизатор;
- пружина;
- диагональная стойка.

Описание:

```text
Minimal outline icon of motorcycle shock absorber with spring, technical line style.
```

### 7. Охлаждение

Образ:

- радиатор;
- вертикальные линии;
- маленькая волна температуры.

Описание:

```text
Minimal outline icon of motorcycle radiator with cooling lines, technical line style.
```

### 8. Фильтры / расходники

Образ:

- цилиндрический фильтр;
- масляная капля;
- или маленькая канистра.

Описание:

```text
Minimal outline icon of oil filter cartridge with small oil drop, technical line style.
```

## 10.4. Размеры иконок

Для карточек top-nodes:

```css
width: 28px;
height: 28px;
```

Для маленьких списков:

```css
width: 20px;
height: 20px;
```

Для карты мотоцикла:

```css
width: 32px;
height: 32px;
```

## 10.5. SVG-рекомендации

Если рисовать SVG вручную:

- использовать `viewBox="0 0 24 24"`;
- использовать `stroke="currentColor"`;
- использовать `fill="none"`;
- использовать `strokeWidth="1.75"` или `2`;
- использовать `strokeLinecap="round"`;
- использовать `strokeLinejoin="round"`.

Пример базового шаблона:

```tsx
<svg
  viewBox="0 0 24 24"
  fill="none"
  stroke="currentColor"
  strokeWidth="1.75"
  strokeLinecap="round"
  strokeLinejoin="round"
  aria-hidden="true"
>
  ...
</svg>
```

# 11. Иконки: требования к реализации через Cursor

Cursor должен:

1. создать или обновить компонент иконок top-nodes;
2. использовать единый стиль SVG;
3. не подключать новую библиотеку без необходимости;
4. не использовать emoji;
5. не использовать случайные иконки из разных стилей;
6. не делать иконки слишком детализированными;
7. обеспечить читаемость в dark и light theme;
8. привязать иконки к top-node type.

Пример структуры:

```text
src/components/mototwin/node-icons.tsx
```

Пример API:

```tsx
<NodeIcon nodeType="engine" />
<NodeIcon nodeType="brakes" />
<NodeIcon nodeType="tires" />
<NodeIcon nodeType="chain" />
<NodeIcon nodeType="battery" />
<NodeIcon nodeType="suspension" />
```

Если уже есть компонент иконок, не создавать новый без необходимости, а расширить существующий.

## 12.1. Назначение

Блок показывает top-nodes и их состояние.

Это не полное дерево.

## 12.2. Заголовок

```text
Состояние основных узлов
```

```text
Все узлы →
```

```text
[иконка] Двигатель
OK
Масло через 1 800 км
```

```text
[иконка] Тормоза
Soon
Проверить колодки через 450 км
```

```text
[иконка] Шины
```

При клике:

- если top-node имеет detail page, открыть detail page;

- не ломать существующий функционал.

## 13.2. Заголовок

```text
Последние события
```

# 18. Что не делать

Не делать:

1. не показывать полное дерево узлов на странице мотоцикла;
2. не показывать полный журнал на странице мотоцикла;
3. не создавать новую логику дерева, если она уже есть;
4. не переписывать существующий журнал;
5. не рисовать иконки разных стилей;
6. не использовать emoji вместо иконок;
7. не использовать фото конкретных моделей;
8. не использовать 3D;
9. не перегружать страницу всеми данными;
10. не добавлять новые библиотеки без необходимости.

---

# 19. Acceptance Criteria

Результат считается готовым, если:

1. На странице мотоцикла отображаются только top-nodes.
2. У каждого top-node есть иконка в стиле MotoTwin.
3. Иконки выполнены в едином line-art стиле.
4. У top-nodes отображаются статусы OK / Soon / Overdue / Recently.
5. Есть ссылка “Все узлы →” или “Открыть полное дерево узлов”.
6. Ссылка открывает уже реализованное полное дерево узлов.
7. На странице мотоцикла показываются только последние события журнала.
8. Есть ссылка “Весь журнал →”.
9. Ссылка открывает уже реализованный полный журнал обслуживания.
10. Существующий функционал дерева и журнала не переписан и не сломан.
11. Страница остается компактной.
12. Есть поддержка mobile layout.
13. Есть поддержка dark и light theme.
14. UI-тексты на русском.
15. Код, компоненты и переменные на английском.
16. Не добавлены лишние библиотеки.
17. Все изменения соответствуют MVP scope.
