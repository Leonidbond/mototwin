# Визуальный паритет Web + Expo (MotoTwin)

**Дата:** 2026-04-18  
**Цель:** Один продуктовый визуальный язык (фон, карточки, типографика, ошибки, CTA, статусы) без копирования вёрстки пиксель-в-пиксель.

## Общие токены (`packages/design-tokens`)

Добавлен объект **`productSemanticColors`**: canvas, card, приглушённые поверхности, границы, иерархия текста, основная кнопка, ошибка, акценты журнала (timeline SERVICE / STATE_UPDATE), бейдж сервиса, тень, оверлей модалки (`overlayModal`).

Уже существовали **`statusSemanticTokens`** и русские/английские подписи статусов — для цветов статусов по-прежнему используются они.

**Правило:** новые экраны и правки цвета предпочтительно делают через `productSemanticColors` и `statusSemanticTokens`, а не через произвольный hex.

## Web

- **Гараж** (`src/app/garage/page.tsx`): фон страницы — `productSemanticColors.canvas`; блок ошибки — `errorSurface` / `errorBorder` / `error`; основные CTA — `primaryAction`; пустое состояние: кнопка **«Добавить мотоцикл»** (как в основном CTA), без формулировки «Перейти к onboarding».
- **Onboarding** (`src/app/onboarding/page.tsx`): фон — `canvas` для согласования с гаражом и Expo-оболочкой.
- **Карточка мотоцикла** (`src/app/vehicles/[id]/page.tsx`): фон страницы — `canvas`; блок ошибки загрузки — те же error-токены.
- **Журнал обслуживания (модалка на странице ТС):** оверлей — `overlayModal`; таймлайн (линия, точка, фон/бордер карточки, бейдж «Сервис» / «Обновление состояния») — те же поля `productSemanticColors`, что и в Expo `service-log.tsx` (`timelineService*` / `timelineState*`, `serviceBadge*`, `card` / `cardMuted`, `border`, и т.д.). Вёрстка и Tailwind-классы для layout не вычищались.

## Expo (`apps/app/app/**`)

В экранах гаража, карточки ТС, журнала, добавления мотоцикла/сервиса, состояния и профиля **литералы `#RRGGBB` в `StyleSheet` заменены** на поля `productSemanticColors as c`, плюс `ActivityIndicator` / `placeholderTextColor` завязаны на токены.

**Гараж** (`app/index.tsx`):

- Заголовок и описание приведены к тону web: «MotoTwin | Гараж», «Ваш гараж», тот же смысл описания.
- Пустое состояние: **«В гараже пока нет мотоциклов»** и тот же подзаголовок, что на web.

**Добавление мотоцикла** (`vehicles/new.tsx`): заголовок экрана — **«Добавление мотоцикла»** (как на web onboarding).

**Журнал** (`service-log.tsx`): заливка точки STATE_UPDATE на таймлайне — `timelineStateFill` (не смешивается семантически с `divider`).

## Намеренные отличия

- **Плотность и компоновка:** web — сетки и модалки; Expo — списки и полноэкранные формы.
- **Expo гараж:** строка «Текущий API: …» на экране ошибки показывается **только в development** (`__DEV__`); в production-сборке скрыта, пользовательские сообщения об ошибке сохраняются (см. `docs/frontend-expo.md` §4.1).
- **Tailwind на web:** большинство классов (`text-gray-*`, `border-gray-*`) сохранены; точечно подмешан `style` с токенами для фона страницы, ошибок и primary CTA, чтобы не переписывать весь `tailwind.config`.

## Отображение дат (web + Expo)

- **Смысл даты** везде один: та же моментальная метка из API (`eventDate`, даты в `statusExplanation`), без смены часового пояса на клиенте.
- **Строка записи в журнале обслуживания** (`dateLabel` в VM из `buildServiceLogTimelineProps`):
  - **Web:** `ServiceLogEntryDateStyle` **`"default"`** — `toLocaleDateString("ru-RU")` без опций (полная локальная дата).
  - **Expo:** **`"compact"`** — день (число) + короткий месяц + год (`ru-RU`), плотнее для мобильного списка.
- **Заголовок месяца в журнале** (группировка по месяцам): на обоих клиентах из domain — **длинное название месяца + год** (`ru-RU`), одна реализация в `packages/domain` (`formatMonthYearLabel` внутри группировки).
- **Пояснение статуса узла** (web модалка, Expo модалка): **полная** дата как у журнала `default` — общий хелпер **`formatIsoCalendarDateRu`** в `@mototwin/domain` (без дублирования в страницах).
- **Неразобранная дата:** для `compact`-строки журнала — усечение ISO до `YYYY-MM-DD`; для полной/пояснения — возврат исходной строки.

## Спеки на карточке гаража

- Источник строк: **`buildGarageCardProps`** → четыре highlight: двигатель, охлаждение, колёса, тормоза (как в модели; при отсутствии данных — плейсхолдеры в сыром списке).
- **Отображение на гараже (web и Expo):** после сборки карточки применяется **`filterMeaningfulGarageSpecHighlights`** — в UI попадают только осмысленные значения; плейсхолдеры «Не указан*» не показываются, чтобы не засорять гараж.
- **Карточка ТС:** полный техблок через **`buildVehicleTechnicalInfoViewModel`** (дополнительные поля, только непустые строки).
