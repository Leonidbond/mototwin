# Рабочий процесс паритета Web + Expo (MotoTwin)

Практичный порядок планирования и сдачи фич при двух клиентах (Next.js и Expo). Дополняет раздел **«25. Cross-Platform Parity Rules»** в [coding-rules.md](./coding-rules.md). Шаблон промпта для Cursor: [parity-task-template.md](./parity-task-template.md).

---

## 1. Планирование фичи по слоям

Любая задача проходит оценку по столбцам. Если столбец не затронут — явно напишите **«нет»** или **«не применимо»**.

| Слой | Что уточнить | Типичные артефакты |
|------|----------------|---------------------|
| **Backend** | Меняются ли маршруты, поля ответов, валидация, ошибки? | Контракт ответа/запроса, статусы, сообщения об ошибках |
| **Shared (`packages/`)** | Нужны ли новые типы, доменные хелперы, формы, контракты секций, API-клиент? | `@mototwin/types`, `@mototwin/domain`, при необходимости `api-client`, `design-tokens` |
| **Web (`src/`)** | Какие страницы/состояния затронуты? | Маршруты, UI, вызовы API |
| **Expo (`apps/app/`)** | Какие экраны/навигация? | Screens, вызовы API |
| **Документация** | Нужно ли обновить контракты, QA, гайды? | `docs/*.md` по смыслу задачи |
| **QA** | Что проверить на обоих клиентах и на API? | Чеклист ниже, граничные случаи |

**Backend impact:** фича не считается описанной, пока не ясно: какие эндпоинты, какие поля, обратная совместимость, ошибки для клиента.

**Shared impact:** если логика или смысл данных общие — по возможности типы и хелперы в shared, а не дублирование в web/Expo.

**Web / Expo impact:** для каждого клиента — конкретные точки входа (страница/экран) или явное **отложено** с причиной.

**Documentation impact:** новые контракты, переименования API, исключения паритета — зафиксировать в `docs/`.

**QA impact:** минимум один сценарий «счастливый путь» на web и на Expo; при изменении API — проверка ответа/ошибок.

## 2. Чеклист планирования фичи (Feature planning checklist)

Перед разработкой ответьте (да / нет / N/A + кратко «что именно»):

- [ ] Требуются ли **изменения backend** (новые/изменённые маршруты, логика, ответы)?
- [ ] Требуются ли **изменения модели данных** (Prisma, миграции) — *если да, задача должна явно это разрешать по правилам репозитория*?
- [ ] Нужны ли **общие типы** (`packages/types`)?
- [ ] Нужны ли **общие доменные хелперы** (`packages/domain`)?
- [ ] Нужны ли **общие контракты форм / секций** (формы, component contracts)?
- [ ] Затрагивает ли задача **web**?
- [ ] Затрагивает ли задача **Expo**?
- [ ] **Одинаковы ли ожидаемые UX-исходы** (что пользователь понял и что произошло с данными)?
- [ ] Есть ли **намеренные отличия** платформ (навигация, плотность UI, нативные паттерны)? Если да — перечислить.
- [ ] **Что обязательно протестировать на обоих клиентах** (и при необходимости через API)?

## 3. Definition of done для кроссплатформенных фич

Фича считается **завершённой** только если выполнено **всё** ниже:

1. **Контракт backend ясен:** задокументированы или однозначно выведены из кода эндпоинты, полезная нагрузка, ошибки; нет «догадок» у второго клиента.
2. **Shared-слой обновлён при необходимости:** типы, хелперы, формы, API-клиент, токены — всё, без чего клиенты разъедутся по смыслу данных.
3. **Web:** реализация сделана **или** явно **отложена** (срок/тикет/причина в задаче или `docs/`).
4. **Expo:** реализация сделана **или** явно **отложена** (аналогично).
5. **Разрыв паритета задокументирован:** если один клиент впереди другого — в задаче или в `docs/` указано, что именно не совпадает и что делать дальше.
6. **QA:** есть чеклист, который **покрывает оба клиента** (или явно помечены пункты только web / только mobile с обоснованием).

Исключение: чисто инфраструктурные или одноплатформенные задачи — в промпте явно указать **«паритет не требуется»** и почему.

## 4. Шаблон задачи для Cursor

Полная копируемая форма — в [parity-task-template.md](./parity-task-template.md). Минимум полей в каждом промпте:

- Context, Goal  
- Files allowed / must not change  
- **Web impact** · **Expo impact** · **Shared impact** · **Backend impact**  
- **Parity expectation** (полный паритет / намеренные отличия / отложено)  
- QA requirements · Documentation requirements  
- Expected result  

---

## 5. Быстрая итерация (MVP)

- Не требовать общих UI-компонентов ради паритета: достаточно общих **типов, домена, контрактов, API**.
- Зафиксировать отложенный клиент **одной строкой** в задаче лучше, чем молчаливый долг.
- После изменения API первым делом обновить **типы + api-client**, затем клиентов.
