# Ручной QA: журнал обслуживания Expo ≈ web

**Дата:** 2026-05-10  
**Связанные документы:** [web-expo-service-log-parity-fixes.md](./web-expo-service-log-parity-fixes.md)  
**Эталон web:** `src/app/vehicles/[id]/service-log/page.tsx` — `ServiceLogRow`, `ServiceLogEventDetails`  
**Эталон Expo:** `apps/app/app/vehicles/[id]/service-log.tsx`, `apps/app/app/vehicles/[id]/expenses.tsx`, `apps/app/app/vehicles/[id]/wishlist/picker.tsx`, `apps/app/app/vehicles/[id]/wishlist/index.tsx`, `apps/app/app/vehicles/[id]/parts.tsx`; bundle-форма сервисного события — **`apps/app/components/vehicle-detail/basic-service-event-bundle-form.tsx`** (экран `service-events/new.tsx`).

## Подготовка

- [ ] Один и тот же мотоцикл с данными: минимум одно **сервисное** событие (желательно BASIC и отдельно ADVANCED с несколькими строками bundle), одно **STATE_UPDATE**, у сервисного — связанные **расходы**, по возможности **wishlist** в `installedPartsJson`, заполненные **напоминание** / **исполнитель** / **моточасы**.
- [ ] Открыть ту же машину на **web** и в **Expo** (симулятор или устройство с тем же API).

---

## Сценарий A — строка ленты

Для одной и той же **сервисной** записи:

- [ ] Сравнить **дату** (день + месяц, год), **пробег** в левой колонке с web.
- [ ] Сравнить **цвет rail/точки** с типом работы (REPLACE / INSPECT / …).
- [ ] Сравнить **заголовок** (`mainTitle`) и **вторую строку** (`secondaryTitle`).
- [ ] Сравнить три подписи метрик: **Стоимость**, **Интервал**, **Исполнитель** — значения совпадают по смыслу с web.

Для записи **STATE_UPDATE**:

- [ ] Заголовок и вторая строка приглушены относительно сервисной записи, как на web.
- [ ] Тап открывает sheet с секцией «Изменения состояния»; в футере только **Закрыть** (нет редактирования/удаления/повтора).

---

## Сценарий B — sheet (сервисное событие)

Тап по сервисной строке:

- [ ] Sheet открывается; строка остаётся **selected** (рамка).
- [ ] Проверить по порядку и содержанию: метрики (дата, пробег, интервал, стоимость), **Узлы** (чипы), **Режим записи**, при наличии — **Моточасы**, **Следующее напоминание**, комментарии по bundle, **Стоимость по статьям**.
- [ ] Для **BASIC:** блока **«Установленные запчасти»** нет (как на web).
- [ ] Для **ADVANCED:** блок **«Установленные запчасти»** есть; строки с ценой совпадают с web; тап по строке открывает **picker** или **wishlist** с ожидаемыми параметрами (см. сценарий D).
- [ ] **Исполнитель** и **Источники** / **Расходы** — те же данные, что на web-панели.
- [ ] **Редактировать** / **Удалить** / **Повторить ТО** вызывают те же маршруты Expo, что и раньше (`service-events/new` с query).
- [ ] Закрытие sheet (крестик или свайп системный) — выделение строки снимается.

---

## Сценарий C — deep link и сброс highlight в журнале

- [ ] Сформировать ссылку на журнал с `?serviceEventId=<id>` или `?highlightServiceEventId=<id>` (дополнительно можно добавить `nodeIds`, `paidOnly=1`, `month=`).
- [ ] Открыть в Expo: запись **в зоне видимости** после скролла, подсветка применена.
- [ ] Через короткую задержку в адресе **нет** `serviceEventId` / `highlightServiceEventId`, остальные query **сохранены**.

---

## Сценарий D — расходы из sheet

Из sheet сервисного события с расходами:

- [ ] Тап **«Стоимость»** (когда доступна) или строка в блоке **Расходы** / **Стоимость по статьям** → экран расходов.
- [ ] В query: `year`, `serviceEventId`, при тапе по конкретному расходу — `highlightExpenseId`.
- [ ] Список отфильтрован по событию; нужная карточка **подсвечена**; затем `highlightExpenseId` **исчезает** из URL, событие в фильтре остаётся.
- [ ] **Назад** в шапке расходов при наличии `returnTo` → возврат на журнал с `highlightServiceEventId` в URL (проверить, что снова подсвечивается нужная запись и снова срабатывает сброс query по правилам журнала).

---

## Сценарий E — подбор (picker) и `partsSearch`

Из sheet **ADVANCED**, строка установленной запчасти **без** wishlist id, но с **узлом** и/или названием:

- [ ] Открывается `/vehicles/:id/wishlist/picker?...` с `nodeId` и при необходимости `partsSearch`.
- [ ] Поле поиска в picker содержит переданный **`partsSearch`** (или эквивалент после trim).

---

## Сценарий F — узлы и wishlist из sheet

- [ ] Тап по чипу **Узлы** → дерево `/vehicles/:id/nodes?nodeId=...`, правильный узел.
- [ ] Тап по источнику **wishlist** → `/vehicles/:id/parts?…` → редирект на `/vehicles/:id/wishlist?wishlistItemId=…` (и при необходимости `partsSearch`, `returnTo`); список с фильтром по **статусу** позиции (например «Установленные»), деталь открыта, **без** кратковременного переключения в «Все» и закрытия детали.

---

## Сценарий G — фильтры журнала (регрессия)

- [ ] `paidOnly`, фильтр по узлу (`nodeIds` + `nodeLabel`), сортировка — счётчики месяцев и список записей согласованы с ожиданием (сравнить с web при тех же фильтрах).
- [ ] **Сбросить** в блоке фильтров — полный сброс query журнала (как описано в parity-доке).
