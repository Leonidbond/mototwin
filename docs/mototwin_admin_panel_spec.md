# MotoTwin — спецификация функционала и визуала админ-панели

## 1. Назначение админ-панели

**Админ-панель MotoTwin** — это внутренний инструмент для управления качеством данных, каталогами, совместимостью деталей, пользовательскими отчетами, модерацией и развитием поддержки моделей.

Админка должна быть не просто набором CRUD-таблиц, а **центром управления data quality** для MotoTwin.

Главная задача админки:

> показывать не только то, что уже есть в базе, но и где данные слабые, где есть конфликты, что нужно проверить и какие модели пора переводить на следующий уровень поддержки.

Админ-панель должна поддерживать стратегию MotoTwin:

- цифровой гараж;
- сервисная история;
- база деталей;
- пользовательские fitment-отчеты;
- коллективная валидация совместимости;
- trust / confidence layer;
- управляемый рост поддержки моделей.

## 2. Основные задачи админки

Админ-панель должна позволять управлять:

1. пользователями и их активностью;
2. мотоциклами пользователей;
3. каталогом брендов, моделей, поколений и модификаций;
4. уровнями поддержки моделей;
5. деревом узлов мотоцикла;
6. каталогом запчастей;
7. пользовательскими fitment-отчетами;
8. правилами совместимости;
9. confidence / trust-статусами;
10. правилами регламентного обслуживания;
11. массовыми загрузками;
12. модерацией и верификацией;
13. отчетами по пользователям, мотоциклам, деталям и качеству данных;
14. уведомлениями;
15. подписками;
16. аудитом изменений.

## 3. Роли пользователей админки

На старте достаточно 4 ролей.

| Роль | Описание прав |
|---|---|
| **Super Admin** | Все права, управление ролями, удаление, откаты, системные настройки |
| **Catalog Manager** | Бренды, модели, узлы, детали, правила совместимости, массовые загрузки |
| **Moderator** | Проверка пользовательских деталей, fitment-отчетов, фото, дублей и жалоб |
| **Analyst / Read-only** | Отчеты, выгрузки, просмотр данных без редактирования |

В будущем можно добавить роль **Model Expert** — эксперт по конкретной марке или модели, например BMW R 1250 GS или KTM 790 Adventure.

## 5. Визуальная концепция админки

Админ-панель должна быть в стиле MotoTwin, но более операторской и плотной.

### Общий стиль

- темная тема;
- плотные таблицы;
- много фильтров;
- четкие статусы;
- минимум декоративности;
- акцентный оранжевый для основных действий;
- красный только для ошибок, конфликтов и safety-critical;
- зеленый для verified / approved;
- желтый для pending / low confidence / needs review;
- серый для archived / inactive.


### Правый инспектор

На большинстве рабочих экранов должна быть правая панель-инспектор.

Пример:

- слева таблица fitment-отчетов;
- справа выбранный отчет;
- внутри: деталь, мотоцикл, узел, фото, голоса, confidence, история изменений, действия.

Основные действия в правом инспекторе:

- одобрить;
- отклонить;
- запросить уточнение;
- объединить с дублем;
- создать правило совместимости;
- пересчитать confidence;
- открыть историю изменений.

## 6. Главный дашборд

Дашборд должен отвечать на вопрос:

> что происходит с продуктом, пользователями и качеством базы сегодня?

### 6.1. Верхние KPI-карточки

#### Пользователи

- всего пользователей;
- новые за 7 дней;
- активные за 7 дней;
- пользователи с добавленным мотоциклом;
- пользователи без мотоцикла после регистрации.

#### Мотоциклы

- всего мотоциклов в гаражах;
- новые за 7 дней;
- топ брендов;
- топ моделей;
- модели без fitment-данных.

#### Fitment

- новых fitment-отчетов;
- ожидают проверки;
- конфликтных отчетов;
- verified by MotoTwin;
- low confidence.

#### Каталог

- всего деталей;
- новых деталей от пользователей;
- дублей-кандидатов;
- деталей без категории;
- деталей без привязки к узлу.

#### Регламенты

- моделей с заполненными service rules;
- моделей без регламентов;
- узлов без правил ТО;
- просроченных напоминаний у пользователей.

## 7. Отчеты

Раздел **“Отчеты”** должен быть отдельным крупным модулем.

---

### 7.1. Отчеты по пользователям

Цель: понимать, кто пришел, что сделал и где теряется.

Основные разрезы:

- регистрации по дням / неделям;
- активация: зарегистрировался → добавил мотоцикл → добавил пробег → добавил сервисное событие → посмотрел fitment;
- пользователи с мотоциклом / без мотоцикла;
- пользователи с одним / несколькими мотоциклами;
- активность по сервисному журналу;
- активность по fitment-отчетам;
- активность по голосованиям;
- пользователи, которые давно не обновляли пробег / моточасы;
- пользователи, у которых много просроченных событий;
- пользователи с высоким вкладом в базу;
- потенциальные power users.

KPI-карточки:

```text
Новые пользователи
Активированные пользователи
Добавили мотоцикл
Добавили сервисное событие
Добавили fitment-report
Подтвердили чужой fitment
Давно не обновляли пробег
```

---

Основные разрезы:

- бренд;
- модель;
- поколение;

- регион;

- количество активных владельцев;
- средний пробег;
- наличие моточасов;
- полнота профиля;
- количество сервисных событий;
- количество fitment-отчетов;
- количество verified деталей;
- количество conflicting деталей.

### 7.3. Отчеты по качеству fitment

Показывать:

- сколько деталей имеют статус `Verified by MotoTwin`;
- сколько `Community confirmed`;
- сколько `Fits with modification`;
- сколько `Mixed reports`;
- сколько `Low confidence`;
- сколько `Rejected / likely incompatible`;
- где много конфликтов;
- где мало evidence;
- где много голосов, но нет модерации;
- какие safety-critical категории требуют проверки.

---

### 7.4. Отчеты по каталогу

Показывать:

- количество деталей по категориям;
- количество деталей по брендам;
- детали без SKU;
- детали без фото;
- детали без атрибутов;
- детали без привязки к узлу;
- детали без привязки к моделям;
- дубли SKU;
- похожие названия;
- OEM / aftermarket соотношение;
- покрытие по приоритетным узлам.

---

### 7.5. Отчеты по регламентам ТО

Показывать:

- какие модели имеют service rules;
- какие узлы покрыты правилами ТО;
- какие модели без регламентов;
- какие правила часто приводят к просрочкам;
- где пользователи часто игнорируют напоминания;
- какие узлы чаще всего обслуживаются.

## 8. Модерация и верификация добавленных запчастей

Раздел **“Модерация”** должен быть рабочим местом оператора.

### 8.1. Очереди модерации

Вкладки:

1. **Новые детали**
2. **Новые fitment-отчеты**
3. **Изменения существующих деталей**
4. **Конфликты совместимости**
5. **Дубли деталей**
6. **Safety-critical**
7. **Жалобы пользователей**
8. **Запросы на уточнение**

---

### 8.2. Карточка проверки детали

В карточке детали показывать:

- кто добавил;
- дата добавления;
- SKU / артикул;
- бренд детали;
- название;
- категория;
- узел;
- фото;
- описание пользователя;
- похожие детали в базе;
- возможные дубли;
- где уже используется;
- связанные fitment-отчеты;
- источники / evidence.

Действия:

- **Одобрить как новую деталь**;
- **Связать с существующей**;
- **Объединить с дублем**;
- **Отправить на уточнение**;
- **Отклонить**;
- **Пометить как spam / invalid**;
- **Добавить alias / cross-reference**;
- **Назначить категорию**;
- **Привязать к узлу**.

---

### 8.3. Карточка проверки fitment-отчета

Показывать:

- мотоцикл пользователя;
- бренд / модель / год / модификация;
- узел;
- деталь;
- результат: подходит / не подходит / с доработкой;
- комментарий;
- фото установки;
- пробег после установки;
- связанное сервисное событие;
- голоса других пользователей;
- trust score автора;
- конфликты с другими отчетами;
- safety-critical флаг.

Действия:

- **Approve report**;
- **Reject report**;
- **Mark as needs evidence**;
- **Mark as conflicting**;
- **Escalate to expert**;
- **Convert to verified fitment**;
- **Create / update FitmentRule**;
- **Recalculate confidence**.

---

### 8.4. Safety-critical режим

Для тормозов, подвески, рулевого управления, двигателя и электрики нужна отдельная логика.

Для таких категорий:

- нельзя быстро переводить в `Verified by MotoTwin`;
- нужен повышенный порог подтверждений;
- желательно требовать фото;
- желательно требовать связанное сервисное событие;
- спорные кейсы должны автоматически попадать в отдельную очередь.

## 9. Массовое добавление запчастей и правил совместимости

Раздел **“Массовые загрузки”** должен быть сделан как безопасный мастер, а не как простая загрузка CSV.

### 9.1. Сценарий массовой загрузки

#### Шаг 1. Загрузить файл

Поддерживаемые форматы:

- CSV;
- XLSX;
- JSON позже.

#### Шаг 2. Выбрать тип загрузки

Типы загрузок:

- детали;
- правила совместимости;
- service rules;
- модели;
- конфигурации;
- OEM cross-reference;
- part aliases.

#### Шаг 3. Сопоставить колонки

Примеры колонок:

- SKU;
- part brand;
- title;
- node;
- model;
- year from;
- year to;
- fitment status;
- source;
- confidence;
- notes.

#### Шаг 4. Валидация

Проверять:

- обязательные поля;
- неизвестные бренды;
- неизвестные модели;
- неизвестные узлы;
- дубли SKU;
- конфликты с существующими правилами;
- неполные интервалы годов;
- safety-critical категории.

#### Шаг 5. Preview

Показывать:

- сколько будет создано;
- сколько обновлено;
- сколько пропущено;
- сколько конфликтов;
- сколько требует ручной проверки.

#### Шаг 6. Dry run

Прогон без записи в базу.

#### Шаг 7. Import

- создание `ImportBatch`;
- сохранение каждой строки;
- лог ошибок;
- возможность отката.

#### Шаг 8. Review after import

- отчет по загрузке;
- список созданных записей;
- список конфликтов;
- кнопка rollback.

---

### 9.2. Массовые операции в таблицах

### 24.1. RBAC и доступ к админке

На старте остается заглушка авторизации (`getCurrentUserContext`), но добавляется явная ролевая модель.

Prisma-расширение модели `User`:

```prisma
enum AdminRole {
  SUPER_ADMIN
  CATALOG_MANAGER
  MODERATOR
  ANALYST
}

model User {
  // ...
  isModerator Boolean   @default(false) // оставить для обратной совместимости
  adminRole   AdminRole?                // основной источник прав в админке
}
```

Матрица прав (R = read, W = write, A = approve/publish):

- **Super Admin** — все разделы R/W/A + назначение ролей + аудит-откаты + системные настройки.
- **Catalog Manager** — Каталог деталей, Модели и поддержка, Fitment, Регламенты ТО, Массовые загрузки, Справочники: R/W/A. Пользователи / Мото / Модерация / Аудит: R.
- **Moderator** — Модерация: R/W/A. Каталог деталей и Fitment: R/W (без массового публикейшна), создание FitmentRule из подтвержденных отчетов. Пользователи / Мото: R. Аудит: R по своим действиям.
- **Analyst (Read-only)** — все разделы только R, плюс выгрузки CSV.

Гейтинг:

- `src/middleware.ts` редиректит `/admin/*` без `adminRole` на `/`.
- API: `requireAdminRole(role | role[])` хелпер в `src/lib/admin-auth.ts` оборачивает каждый `/api/admin/*` route.
- Заглушка остается: dev-хедер `x-mototwin-dev-user` переключает контекст; в `prisma/seed.ts` добавляются 4 админа (по одному на роль).

#### Терминология fitment в коде vs в спеке

В коде уже существуют `PartFitment` (правило: part × modelVariant × node × yearRange × status) и `FitmentConfidence` (confidence-агрегат). В спеке они называются `FitmentRule` / `FitmentConfidence`. Договоренность для админки:

- `FitmentRule` в UI и API == строка в Prisma `PartFitment`;
- `FitmentConfidence` остается как есть.

Дополнительной модели `FitmentRule` создавать **не нужно**.

### 24.4. Фильтр периода и кеширование

- варианты периода: `1d / 7d / 14d / 30d / 90d / custom`;
- по умолчанию `7d` (как на референсе);
- применяется к KPI-карточкам, спарклайнам, блокам «Растут быстрее всего», «Активность и сигналы»; не применяется к статическим справочникам и матрицам покрытия;
- кеш: `unstable_cache` (Next.js 16) на 60 с для KPI и 300 с для тяжелых агрегаций; кнопка `Refresh` инвалидрует через `revalidateTag`;
- все timestamp нормализуются к зоне `Europe/Moscow`.

- **Loading**: skeleton-карточки в стиле `GarageSummary`, для таблиц — 8 placeholder-строк;
- **Empty**: иллюстрация + текст + основной CTA (например, «Создать правило» / «Загрузить файл»);
- **Error**: inline баннер + кнопка `Повторить`, плюс toast.

---

### 24.6. Адаптив

- desktop-first, целевое разрешение `≥1280px`;
- `1024px–1279px` — сайдбар автоматически свернут до иконок;
- `<1024px` — заглушка «Откройте на десктопе»;
- мобильные браузеры в админку не пускаем (планшет в landscape — допускаем).

---

### 24.7. Быстрые действия (плитки на дашборде)

Каждая кнопка ведет на конкретное место/действие:

- **Добавить деталь** → `/admin/catalog/new` (форма создания PartMaster);
- **Загрузить CSV** → `/admin/imports/new`;
- **Создать правило совместимости** → `/admin/fitment/new`;
- **Открыть модерацию** → `/admin/moderation`;
- **Пересчитать confidence** → `POST /api/admin/fitment/recalculate` (для MVP — синхронно, с прогрессом в toast).

---

### 24.8. Колокольчик алертов в верхней панели

Источник числа в бейдже — агрегат `dashboard.alerts`:

- pending модерация (детали + fitment reports);
- конфликты совместимости;
- safety-critical в очереди;
- import errors за последние 24 часа;
- правила ТО, требующие проверки.

При клике открывается popover со ссылками на соответствующие очереди.

### 24.9. Локализация

### 24.10. Seed демо-данных

`prisma/seed.ts` (или новый script) должен создать:

- 4 админа: `super@mototwin.local`, `catalog@mototwin.local`, `moderator@mototwin.local`, `analyst@mototwin.local` — с разными `adminRole`;
- `DEV_USER_OPTIONS` обновляется так, чтобы админов можно было переключать в dev-хедере;
- набор данных для непустого дашборда (несколько тысяч моков по моделям, 100+ fitment-отчетов с разными статусами, проблемные зоны, спарклайны).
