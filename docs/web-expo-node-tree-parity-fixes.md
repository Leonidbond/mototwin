# Паритет дерева узлов и статусов (Web + Expo)

**Дата:** 2026-04-18  
**Цель:** Одинаковый смысл статусов, подписей, кратких пояснений и доступа к детальному объяснению; общие view models и токены; отдельная обработка ошибок дерева на Expo.

## Shared (`packages/domain`, `packages/types`, `packages/design-tokens`)

- **Пустое поддерево:** если у корневого (top-level) узла и у всех потомков нет осмысленных `directStatus` / `computedStatus` / агрегированного статуса, API `GET …/node-tree` возвращает для этого корня **`effectiveStatus: null`** и **`status: null`** — это не «OK» и не подтверждённое здоровое состояние. Дочерние узлы без данных ведут себя так же. Бейдж статуса не показывается (`statusLabel` = null).
- **Статусы:** `buildNodeTreeItemViewModel` по-прежнему задаёт `effectiveStatus`, `statusLabel` (RU из `statusTextLabelsRu`), `statusBadgeLabel` (EN из `statusBadgeLabelsEn` — для компактных сценариев).
- **Краткая строка пояснения:** `shortExplanationLabel` берётся из `getNodeTreeItemReasonShortLine` → `node.statusExplanation?.reasonShort` для **любого узла**, если API вернул блок `statusExplanation` (раньше строка показывалась только для листьев через `getLeafStatusReasonShort`).
- **Детальное пояснение:** `canOpenNodeStatusExplanationModal(node)` — `node.statusExplanation != null`. Web и Expo открывают модалку только если есть полный объект объяснения и показывают кликабельную строку только при непустом `shortExplanationLabel` и этой проверке.
- **Лист / не лист:** `canAddServiceEvent` и `actions.addServiceEventAvailable` остаются привязаны к отсутствию дочерних узлов (`!hasChildren`).
- **Цвета бейджей:** web и Expo используют `statusSemanticTokens` из `@mototwin/design-tokens`.

## Web (`src/app/vehicles/[id]/page.tsx`)

- В дереве узлов в бейдже отображается **`statusLabel`** (русский текст), как на Expo, а не `statusBadgeLabel`.
- Кнопка краткого пояснения: условие `shortExplanationLabel && canOpenNodeStatusExplanationModal(node)` (защита от рассинхрона данных).
- **Статус как действие:** бейдж статуса — интерактивная кнопка (`cursor-pointer`, hover-ring, подсказка). По клику открывается модалка журнала с **фильтром по узлу** (лист: только `nodeId`; родитель: этот узел + все листья поддерева — см. `createServiceLogNodeFilter` в `@mototwin/domain`).

## Expo (`apps/app/app/vehicles/[id]/index.tsx`)

- **Статус как действие:** бейдж обёрнут в `Pressable` (лёгкое pressed-состояние). По нажатию выполняется переход на экран журнала с query `nodeIds` и `nodeLabel` (тот же набор id, что и на web через `createServiceLogNodeFilter`).
- Загрузка **разделена:** сначала `getVehicleDetail`, затем `getNodeTree`. Ошибка карточки ТС не скрывает успешный ответ по дереву и наоборот.
- При ошибке дерева: сообщение **«Не удалось загрузить дерево узлов.»**, кнопка **«Повторить»** (повторный полный `load` через `useFocusEffect` уже обновляет экран при фокусе).
- Во время загрузки дерева: строка **«Загрузка дерева узлов...»** под заголовком секции (аналог web).
- Краткое пояснение: `canOpenNodeStatusExplanationModal` вместо прямой проверки `statusExplanation` (семантика как на web).
- Полное дерево Expo теперь повторяет web-логику **`ТОП-узлы`** (до 15 узлов из overview + родители), **«Свернуть»** и auto-expand цепочек с одним дочерним узлом.
- Контекст узла Expo повторяет web action matrix: `Что входит в узел` для parent/top-level, кликабельные recent events с `serviceEventId`, row/quick-add для совместимых деталей и комплектов, блок активных неустановленных позиций.
- **Визуальная структура контекста:** на Expo нижний лист контекста узла разбит на **секции с рамкой и шапкой** (как блоки в web `NodeContextReferencePanel`), плюс в секции последних событий в шапке — действие **«Журнал»** (аналог «Открыть журнал» на web).
- При переходах из дерева в журнал/корзину Expo сохраняет selected node, статус-фильтр, TOP-mode и раскрытые ветки в mobile-safe storage; возврат восстанавливает leaf focus.

## Намеренные отличия UX

- Web: сетка корневых карточек, раскрытие `+`/`−` только на кнопке.
- Expo: вертикальный список в одной карточке, раскрытие по нажатию на строку узла с детьми.
- Web: полное дерево показывает сводку расходов по узлам за выбранный сезон через `GET /api/expenses/node-summary`.
- Expo: в дереве сейчас нет отдельной сводки расходов по узлам; расходы остаются отдельным экраном.

## «Требует внимания»

- **Web:** кнопка у заголовка ТС открывает модалку со списком узлов `OVERDUE` / `SOON`, собранным через `buildAttentionSummaryFromNodeTree` (`packages/domain/src/attention.ts`). Окраска кнопки — **`buildAttentionActionViewModel`** + **`statusSemanticTokens`** (худший статус среди пунктов). **Полное пояснение** — та же модалка, что у дерева (`setSelectedStatusExplanationNode` + `buildNodeTreeItemViewModel` по `nodeId`), без отдельного inline-разворота в модалке внимания. Действие **«Журнал по узлу»** использует тот же `createServiceLogNodeFilter`, что и клик по бейджу статуса в дереве.
- **Expo:** экран `vehicles/[id]/attention.tsx` — тот же shared summary и та же логика окраски пилюли на `index.tsx`; полное пояснение через общий **`StatusExplanationModal`** (как на экране ТС). Переход в журнал с `nodeIds` / `nodeLabel` через `buildVehicleServiceLogHref`.
- Подробнее: [attention-flow-mvp.md](./attention-flow-mvp.md).

## QA (ручная проверка)

1. Корень без данных по всему поддереву: в JSON `effectiveStatus === null`, на web/Expo **нет** цветного бейджа и **нет** клика по статусу; при наличии данных у потомков — агрегация по приоритету (OVERDUE → … → OK) как раньше.
2. Сверить **одинаковые** подписи статусов (RU) на корнях и листьях при одном ответе API, когда статус ненулевой.
3. Нажать или тапнуть статус у листа и у родителя: журнал открывается с фильтром по узлу; сброс фильтра возвращает полный журнал; месячные группы и сводка соответствуют отфильтрованному набору.
4. У узлов с `reasonShort` и `statusExplanation` — открытие модалки с полным текстом на web и Expo.
5. **+** добавления сервиса только у листьев; у узлов с детьми кнопки нет.
6. Expo: симулировать сбой только `getNodeTree` — карточка ТС видна, под секцией деревьев ошибка и «Повторить».
7. После сохранения пробега / сервиса — повторный заход на экран: статусы совпадают по смыслу на обоих клиентах.
8. **Требует внимания:** счётчик на кнопке совпадает с числом карточек в модалке/экране; из карточки журнал фильтруется по тому же узлу, что и из дерева; краткая строка с `statusExplanation` открывает **ту же** модалку пояснения, что и в дереве.
